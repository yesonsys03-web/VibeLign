## 🏁 처음 시작

### 🚀 `vib start`

> 처음 시작할 때 딱 한 번만 실행해요

**💡 이게 뭐야?**

새 프로젝트에서 VibeLign을 처음 쓸 때 필요한 파일을 자동으로 만들어줘요.
AGENTS.md, AI_DEV_SYSTEM_SINGLE_FILE.md 같은 파일이 생기는데,
이게 있어야 AI가 내 프로젝트를 제대로 이해하고 작업해요.

Git을 쓰는 프로젝트라면 커밋 전에 비밀정보를 막아주는 보호도 같이 켜줘요.
그래서 API 키, 토큰, .env 같은 걸 실수로 올리려 하면 커밋 단계에서 한 번 더 잡아줘요.

`vib start --all-tools`를 쓰면 Claude, Antigravity, OpenCode, Cursor, Codex 준비도 한 번에 해줘요.
기본값은 기존 설정을 최대한 보존하면서 없는 것만 추가해요.
정말 다시 깔듯이 새로 만들고 싶을 때만 `--force`를 써요.

**🕐 언제 써?**

- 새 프로젝트 폴더에서 VibeLign을 처음 쓸 때
- AI한테 코딩을 맡기기 전에 한 번만 세팅할 때
- 설치 직후 처음 설정할 때
- 여러 AI 도구를 한 번에 준비하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib start   # 기본 세팅 (가장 많이 씀)
vib start --all-tools   # Claude, Antigravity, OpenCode, Cursor, Codex를 한 번에 준비
vib start --all-tools --force   # 기존 VibeLign 설정도 다시 생성
vib start --quickstart   # 세팅 + 앵커 자동 삽입까지 한 번에
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `--all-tools` | Claude, Antigravity, OpenCode, Cursor, Codex 설정을 한 번에 준비해요. 기존 설정은 최대한 보존하면서 없는 것만 추가해요. |
| `--tools` | 원하는 도구만 골라서 준비해요. 예: claude,opencode,cursor,antigravity,codex |
| `--force` | 기존 VibeLign 설정 파일도 다시 만들거나 덮어써요. 기본값은 안전 모드라서 기존 설정을 최대한 유지해요. |
| `--quickstart` | start 실행 후 anchor --auto 까지 자동으로 실행해요. 한 번에 다 끝내고 싶을 때 써요. |
| `준비 상태 안내` | start가 끝나면 도구별로 '바로 사용 가능' / '거의 끝남'을 알려줘요. 현재는 Claude, Antigravity, OpenCode는 바로 사용 가능으로 보여줘요. |
| `비밀정보 커밋 보호` | Git 저장소라면 커밋할 때마다 자동 검사도 같이 켜줘요. 커밋 전에 API 키, 토큰, 개인키, .env 같은 걸 검사해줘요. |


| 옵션 | 설명 |
|------|------|
| `--force` | 이미 최신 버전이어도 강제로 다시 설치해요. |

## 💾 세이브 & 되돌리기

### 💾 `vib checkpoint`

> 게임 세이브처럼 지금 상태를 저장해요

**💡 이게 뭐야?**

AI한테 코드를 맡기기 전에 현재 상태를 저장해두는 기능이에요.
뭔가 잘못되면 이 시점으로 되돌릴 수 있어요.
게임에서 '세이브' 버튼 누르는 것과 똑같아요.

메시지 없이 실행하면 메모를 입력할 수 있는 화면이 나와요.
git commit 메시지처럼 '이 시점에 뭘 했는지' 적어두면
나중에 vib undo 에서 어떤 걸 골라야 할지 쉽게 알 수 있어요.

체크포인트 저장 시 PROJECT_CONTEXT.md도 자동으로 갱신돼요.
AI 툴을 바꿀 때 별도로 vib transfer를 실행하지 않아도 됩니다.

**🕐 언제 써?**

- AI한테 수정을 시키기 바로 직전
- 기능이 완성됐을 때 기록을 남기고 싶을 때
- 큰 작업을 시작하기 전에 안전망을 만들 때

**✏️ 이렇게 쳐봐**

```
vib checkpoint   # 실행 → 메시지 입력 화면이 나와요
vib checkpoint "로그인 완성"   # 메시지를 바로 지정해서 저장
vib checkpoint "버그 수정 전"   # 작업 전 백업
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `message` | 저장할 때 메모를 남길 수 있어요. 메시지 없이 실행하면 입력 화면이 나와요. 엔터만 누르면 메시지 없이 저장돼요. 예: vib checkpoint "회원가입 버튼 추가" |

---

### ⏪ `vib undo`

> 저장한 곳으로 되돌려요

**💡 이게 뭐야?**

AI가 코드를 이상하게 바꿨을 때 저장한 시점으로 되돌려요.
게임에서 '불러오기' 버튼 누르는 것과 똑같아요.
checkpoint로 저장해뒀어야 쓸 수 있어요.

실행하면 저장된 목록이 번호와 함께 나와요:
  [1] 오늘 16:52:47   로그인 기능 추가 전  ← 가장 최근
  [2] 오늘 14:30:05   시작
  [0] 취소

번호를 입력하면 그 시점으로 되돌아가요.
엔터만 누르면 가장 최근 저장으로 되돌려요.
0을 누르면 아무것도 안 하고 취소돼요.

**🕐 언제 써?**

- AI가 코드를 망가뜨렸을 때
- 수정 결과가 마음에 들지 않을 때
- 실수로 파일을 지웠을 때
- 특정 시점으로 골라서 되돌리고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib undo   # 목록 보고 번호 선택 → 되돌리기
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `번호 입력` | 목록에서 원하는 시점의 번호를 입력해요. 엔터 = 가장 최근 / 0 또는 q = 취소 |

### 📋 `vib history`

> 저장 목록을 봐요

**💡 이게 뭐야?**

지금까지 checkpoint로 저장한 기록을 전부 보여줘요.
언제 저장했는지, 어떤 메시지를 남겼는지 확인할 수 있어요.

시간은 '오늘 16:52:34' / '어제 09:00:11' 처럼 읽기 쉽게 나와요.
되돌리려면 vib undo 를 쓰세요.

**🕐 언제 써?**

- 내가 언제 저장했는지 확인하고 싶을 때
- 어떤 버전으로 되돌릴지 결정하기 전에 목록을 먼저 볼 때
- 저장이 제대로 됐는지 확인하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib history   # 저장 기록 전체 보기
```

## 🔬 점검 & 확인

### 🩺 `vib doctor`

> 프로젝트 건강 상태를 확인해요

**💡 이게 뭐야?**

내 프로젝트가 AI 작업을 받기에 괜찮은 상태인지 점검해요.
점수로 보여주고, 문제가 있으면 뭐가 문제인지 알려줘요.
마치 의사가 건강검진 해주는 것처럼요.

**🕐 언제 써?**

- AI한테 코딩을 시키기 전에 준비됐는지 확인할 때
- 프로젝트 상태가 좋은지 수시로 확인할 때
- 뭔가 이상한 것 같을 때 진단받고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib doctor   # 기본 점검 (점수 + 문제 목록)
vib doctor --strict   # 더 꼼꼼하게 점검
vib doctor --detailed   # 문제마다 자세한 설명 포함
vib doctor --fix   # 앵커 없는 파일에 자동으로 앵커 추가
vib doctor --write-report   # 결과를 파일로 저장
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `--strict` | 기본 점검보다 더 꼼꼼하게 검사해요. 작은 문제도 놓치지 않아요. |
| `--detailed` | 각 문제마다 왜 문제인지 설명을 추가로 보여줘요. |
| `--fix-hints` | 각 문제를 어떻게 고치면 되는지 힌트를 줘요. |
| `--fix` | 앵커가 없는 파일에 자동으로 앵커를 달아줘요. 직접 하기 귀찮을 때 편해요. |
| `--write-report` | 점검 결과를 파일로 저장해요. 나중에 다시 볼 수 있어요. |
| `--json` | 결과를 JSON 형식으로 출력해요. (개발자용) |

### 🛡️ `vib guard`

> AI가 코드를 망가뜨리지 않았는지 검사해요

**💡 이게 뭐야?**

AI가 코드를 수정한 후, 구조가 망가지지 않았는지 종합 검사해요.
doctor (건강 점검) + explain (변경 설명)을 합친 거예요.
AI 작업 후 항상 한 번 실행해보세요.

**🕐 언제 써?**

- AI가 코드를 수정한 직후
- 뭔가 이상한 것 같을 때 종합 검진을 받고 싶을 때
- AI 작업 결과를 확인하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib guard   # 기본 검사
vib guard --strict   # 더 꼼꼼하게 검사
vib guard --write-report   # 결과를 파일로 저장
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `--strict` | 더 꼼꼼하게 검사해요. 작은 문제도 잡아줘요. |
| `--since-minutes 60` | 최근 몇 분 동안의 변경만 확인해요. 기본값은 120분이에요. |
| `--write-report` | 검사 결과를 파일로 저장해요. |
| `--json` | 결과를 JSON 형식으로 출력해요. (개발자용) |

### 📖 `vib explain`

> 뭐가 바뀌었는지 쉽게 알려줘요

**💡 이게 뭐야?**

최근에 코드가 어떻게 바뀌었는지 쉬운 말로 설명해줘요.
AI가 수정한 내용을 이해하기 어려울 때 쓰세요.
개발 경험 없어도 "이 파일에서 이런 게 바뀌었어요" 수준으로 알려줘요.

**🕐 언제 써?**

- AI가 수정한 내용이 뭔지 이해하고 싶을 때
- 어떤 파일이 바뀌었는지 확인하고 싶을 때
- 변경 내역을 누군가한테 설명해야 할 때

**✏️ 이렇게 쳐봐**

```
vib explain   # 전체 변경 설명
vib explain main.py   # 특정 파일만 설명
vib explain --ai   # AI가 더 자세하게 분석
vib explain --since-minutes 30   # 최근 30분 변경만 보기
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `file` | 특정 파일의 변경만 설명해요. 예: vib explain login.py |
| `--ai` | AI가 변경 내용을 더 자세하게 분석해서 설명해줘요. |
| `--since-minutes 숫자` | 최근 몇 분 동안의 변경만 보여줘요. 기본값은 120분이에요. |
| `--write-report` | 설명 결과를 파일로 저장해요. |
| `--json` | 결과를 JSON 형식으로 출력해요. (개발자용) |

## ✏️ AI 수정 요청

### 🛠️ `vib patch`

> 말로 요청하면 안전한 수정 계획을 만들어요

**💡 이게 뭐야?**

"로그인 버튼 추가해줘" 같이 말로 요청하면,
어떤 파일의 어느 부분을 어떻게 수정할지 계획을 세워줘요.
이 계획을 복사해서 AI (Claude, ChatGPT 등)에 붙여넣으면 돼요.

**🕐 언제 써?**

- AI한테 코드 수정을 요청할 때 더 정확하게 전달하고 싶을 때
- 어떤 파일을 수정해야 할지 모를 때 도움받고 싶을 때
- 수정 전에 계획을 먼저 확인하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib patch "로그인 버튼 추가"   # 수정 계획 만들기
vib patch "버그 수정" --ai   # AI가 더 정확하게 분석
vib patch "사이드바 제거" --preview   # 미리 보기
vib patch "기능 추가" --copy   # 결과를 클립보드에 복사
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `request` | 수정 요청을 말로 써요. 예: vib patch "다크모드 추가해줘" |
| `--ai` | AI가 코드를 더 자세히 분석해서 정확한 계획을 세워요. 조금 더 시간이 걸려요. |
| `--preview` | 수정 계획을 미리 보기로 확인해요. |
| `--copy` | AI에 전달할 프롬프트를 클립보드에 복사해요. 바로 붙여넣기할 수 있어요. |
| `--write-report` | 수정 계획을 파일로 저장해요. |
| `--json` | 결과를 JSON 형식으로 출력해요. (개발자용) |

### ⚓ `vib anchor`

> AI가 정확한 위치를 찾을 수 있도록 표식을 달아요

**💡 이게 뭐야?**

코드 파일에 '여기서부터 여기까지가 이 기능이야'라는 표식(앵커)을 달아요.
이 표식이 있으면 AI가 수정할 때 정확한 위치를 찾을 수 있어요.
지도에 핀 꽂는 것처럼요.

**🕐 언제 써?**

- AI한테 처음으로 코딩을 시키기 전에 (한 번만 하면 돼요)
- 새 파일을 추가했을 때
- 앵커가 제대로 있는지 확인하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib anchor --auto   # 모든 파일에 자동으로 앵커 달기 (가장 많이 씀)
vib anchor --suggest   # 어떻게 달면 좋을지 추천만 보기 (실제로 바꾸지 않음)
vib anchor --validate   # 앵커가 제대로 달려있는지 검사
vib anchor --dry-run   # 실제로 바꾸지 않고 어떻게 바뀔지 미리 보기
vib anchor --only-ext .py   # Python 파일만 처리
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `--auto` | 모든 파일에 자동으로 앵커를 달아줘요. 처음 설정할 때 이걸 써요. |
| `--suggest` | 앵커를 어떻게 달면 좋을지 추천만 보여줘요. 실제로 파일을 바꾸지는 않아요. |
| `--validate` | 앵커가 올바르게 달려있는지 검사해요. 짝이 안 맞는 앵커를 찾아줘요. |
| `--dry-run` | 실제로 바꾸지 않고 어떻게 바뀔지만 미리 보여줘요. |
| `--only-ext .py` | 특정 확장자의 파일만 처리해요. 예: --only-ext .py (파이썬만), --only-ext .js (자바스크립트만) |
| `--json` | 결과를 JSON 형식으로 출력해요. (개발자용) |

### 🔍 `vib scan`

앵커 검사, 앵커 인덱스 갱신, 코드맵 재생성을 한 번에 실행해요.

- 앵커가 꼬인 것 같을 때 한 번에 정리하고 싶을 때
- 코드맵을 최신 상태로 갱신하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib scan   # 앵커 스캔 + 코드맵 갱신
vib scan --auto   # 문제 있는 앵커 자동 수정 + 코드맵 갱신
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|

## 🗂️ 파일 & 설정

### 🔒 `vib protect`

> 중요한 파일을 AI가 건드리지 못하게 잠가요

**💡 이게 뭐야?**

절대 바뀌면 안 되는 파일을 보호해요.
보호된 파일은 AI가 수정 대상으로 포함시키지 않아요.
설정 파일, 환경 변수 파일 등에 쓰면 좋아요.

**🕐 언제 써?**

- .env 같은 설정 파일을 AI가 건드리지 않았으면 할 때
- 중요한 파일을 실수로 수정하는 걸 막고 싶을 때
- 보호 목록을 관리하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib protect main.py   # main.py 보호
vib protect .env   # .env 파일 보호
vib protect --list   # 보호된 파일 목록 보기
vib protect main.py --remove   # main.py 보호 해제
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `file` | 보호할 파일 이름을 써요. 예: vib protect settings.py |
| `--list` | 현재 보호되어 있는 파일 목록을 보여줘요. |
| `--remove` | 보호를 해제해요. 예: vib protect main.py --remove |

### 🔄 `vib transfer`

> AI 툴을 바꿀 때 맥락 파일을 만들어요

**💡 이게 뭐야?**

AI한테 코딩을 시키다 보면 대화창이 꽉 차거나
다른 AI 툴로 바꿔야 할 때가 있어요.
그럴 때 새 AI한테 '지금까지 뭘 했는지, 다음에 뭘 해야 하는지'
한 번에 알려주는 파일을 만들어줘요.

만들어지는 파일: PROJECT_CONTEXT.md

--handoff 옵션을 쓰면 맨 위에 'Session Handoff' 블록이 생겨요.
새 AI는 이 블록만 읽어도 바로 이어서 작업할 수 있어요.

**🕐 언제 써?**

- AI 대화창이 꽉 차서 새 채팅을 열어야 할 때
- Claude → Cursor 같이 AI 툴을 바꾸기 직전에
- 다른 사람에게 프로젝트를 넘길 때
- 새 AI 채팅에 프로젝트 설명을 자동으로 전달하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib transfer   # 기본 생성 (PROJECT_CONTEXT.md 만들기)
vib transfer --compact   # 가벼운 버전 (토큰 절약)
vib transfer --full   # 파일 트리를 더 깊이 포함
vib transfer --handoff   # AI 전환용 인수인계 블록 포함
vib transfer --handoff --print   # 인수인계 내용을 화면에도 출력
vib transfer --handoff --no-prompt   # 질문 없이 자동으로 만들기
vib transfer --handoff --dry-run   # 저장 없이 내용 미리 보기
vib transfer --out ctx.md   # 파일 이름 바꾸기
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `--compact` | 파일 크기를 줄여서 토큰을 아껴요. 파일 트리를 얕게만 보여줘요 (깊이 2단계). 무료 플랜이나 토큰이 부족할 때 좋아요. |
| `--full` | 파일 트리를 더 깊이까지 보여줘요 (깊이 4단계). 프로젝트가 복잡하거나 AI한테 더 많은 구조를 알려주고 싶을 때 써요. |
| `--handoff` | AI를 바꾸거나 새 채팅을 열기 직전에 쓰는 옵션이에요. 파일 맨 위에 'Session Handoff' 블록을 넣어줘요. 새 AI한테 '오늘 뭘 했는지, 다음에 뭘 해야 하는지' 한 줄로 알려줄 수 있어요. ⚠️ --compact, --full 과 같이 쓸 수 없어요. |
| `--print` | --handoff 와 함께 써요. 인수인계 내용을 파일에 저장하면서 화면에도 바로 출력해줘요. 새 AI 채팅창에 복붙하기 편해요. |
| `--no-prompt` | --handoff 와 함께 써요. 보통은 '다음 AI가 뭘 해야 해?' 하고 물어보는데 이 옵션을 쓰면 아무것도 안 물어보고 자동으로 만들어줘요. 모르는 내용은 '(not provided)' 라고 표시돼요. |
| `--out 파일명` | 저장할 파일 이름을 바꿀 수 있어요. 기본값은 PROJECT_CONTEXT.md예요. 예: --out handoff.md |
| `--dry-run` | 파일을 실제로 저장하지 않고 '이런 내용이 들어갈 거예요'를 미리 보여줘요. 마치 게임에서 아이템 설명 읽고 살지 말지 결정하는 것처럼, handoff 내용을 확인하고 나서 진짜로 실행할 수 있어요. --handoff 와 함께 쓰는 게 가장 유용해요. |
| `--help` | transfer 명령어의 옵션 목록을 보여줘요. 뭘 써야 할지 모를 때 먼저 확인해보세요. |

### 🚫 `vib secrets`

> API 키 같은 비밀정보가 Git에 올라가기 전에 막아요

**💡 이게 뭐야?**

커밋하기 전에 지금 올리려는 내용을 검사해서
API 키, 토큰, 개인키, .env 같은 비밀정보가 들어갔는지 확인해줘요.
쉽게 말하면 '실수로 중요한 비밀번호를 인터넷에 올리기 전에 막아주는 문지기'예요.

보통은 `vib start`를 하면 자동으로 연결돼서 따로 신경 쓸 일이 거의 없어요.
직접 확인하고 싶을 때만 `vib secrets --staged`를 실행하면 돼요.

**🕐 언제 써?**

- API 키나 .env를 실수로 커밋할 파일 목록에 넣은 것 같을 때
- 커밋 전에 한 번 더 확인하고 싶을 때
- 왜 커밋이 막혔는지 이해하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib secrets --staged   # 지금 커밋하려는 내용에 비밀정보가 있는지 검사
vib secrets --install-hook   # 앞으로 커밋할 때마다 자동 검사되게 켜기
vib secrets --uninstall-hook   # 자동 검사 끄기
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `--staged` | 지금 커밋할 내용만 검사해요. 한 번만 직접 확인하고 싶을 때 쓰면 돼요. |
| `--install-hook` | 커밋 버튼을 누를 때마다 자동 검사되게 켜요. 보통은 vib start가 자동으로 해줘요. |
| `--uninstall-hook` | 커밋할 때마다 돌던 자동 검사를 꺼요. |
| `오탐 예외 처리` | 진짜 비밀정보가 아닌데도 막히면 그 줄 끝에 `vibelign: allow-secret`를 붙여서 예외 처리할 수 있어요. |

### 👁️ `vib watch`

> 파일이 바뀔 때마다 자동으로 코드맵을 갱신해요

**💡 이게 뭐야?**

파일이 저장될 때마다 자동으로 코드맵을 최신 상태로 유지해요.
AI가 작업하는 동안 켜두면 항상 최신 정보로 작업할 수 있어요.
watchdog 패키지가 설치되어 있어야 해요. (vib start 에서 자동 설치 제안)

**🕐 언제 써?**

- AI가 코딩하는 동안 백그라운드로 켜두고 싶을 때
- 파일이 자주 바뀌는 작업 중에
- 코드맵을 항상 최신 상태로 유지하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib watch   # 실시간 감시 시작 (Ctrl+C 로 종료)
vib watch --strict   # 더 꼼꼼한 감시 모드
vib watch --write-log   # 감시 로그를 파일로 저장
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `--strict` | 더 꼼꼼하게 감시해요. 작은 변화도 잡아줘요. |
| `--write-log` | 감시 중 발생한 일을 파일로 기록해요. |
| `--debounce-ms 숫자` | 파일이 바뀐 후 몇 밀리초 후에 처리할지 설정해요. 기본값은 800ms예요. 너무 자주 갱신되면 늘려요. |
| `--json` | 결과를 JSON 형식으로 출력해요. (개발자용) |

## 🤖 MCP (AI 자동 연동)

### 🤖 `MCP (AI 자동 연동)`

> AI가 VibeLign 기능을 자동으로 써요

**💡 이게 뭐야?**

보통은 AI한테 '체크포인트 저장해줘'라고 말하면
AI가 명령어를 알려주고, 여러분이 직접 터미널에서 쳐야 해요.

MCP가 연결되면 달라져요.
AI한테 말하면 AI가 직접 VibeLign을 실행해요.
터미널을 따로 열 필요가 없어요.

설정: vib start 실행 → Claude Code 재시작 (딱 한 번만)

추가로 `vib start --all-tools`를 쓰면 여러 도구를 한 번에 준비할 수 있어요.
현재 안내 기준으로는 Claude, Antigravity, OpenCode는 바로 사용 가능으로 보여주고,
Cursor, Codex는 마지막 연결 확인이 한 번 더 필요할 수 있어요.

쉽게 생각하면:
  Claude        → 거의 바로 됨
  Antigravity   → 사용자 경험상 바로 되는 경우가 많음
  OpenCode      → 프로젝트 설정 파일까지 함께 준비 가능
  Cursor/Codex  → 규칙 파일은 준비되지만 마지막 연결 확인이 필요할 수 있음

━━ 두 가지 수정 방식 ━━

방식 1 — 일반 수정 (기본)
  평소처럼 말하면 AI가 알아서 수정해요.
  예: '로그인 버튼 색 파란색으로 바꿔줘'

방식 2 — 바이브라인 안전 수정 (키워드: '바이브라인으로')
  요청 앞에 '바이브라인으로'를 붙이면 전체 안전 워크플로우 자동 실행:
  patch_get → 정확한 위치 확인 → 수정 → guard_check → checkpoint 저장
  예: '바이브라인으로 로그인 버튼 색 파란색으로 바꿔줘'

━━ 사용 가능한 MCP 도구 ━━
  checkpoint_create   — 체크포인트 저장   (vib checkpoint 와 같아요)
  checkpoint_list     — 저장 목록 보기    (vib history 와 같아요)
  checkpoint_restore  — 특정 시점 복원    (vib undo 와 같아요)

  ⚠️  복원할 때는 '바이브라인 언두해줘'라고 말하면 돼요.
     AI가 목록을 먼저 보여주고 → 번호를 고르면 → 자동으로 복원해요.
     ('vib undo'를 직접 실행하면 MCP 환경에서 멈춰요. AI한테 맡기세요!)

  project_context_get — AI 전환 시 컨텍스트 전달 (vib transfer 와 같아요)
  doctor_run          — 건강 진단         (vib doctor 와 같아요)
  guard_check         — AI 작업 후 확인   (vib guard 와 같아요)
  protect_add         — 파일 보호 등록    (vib protect 와 같아요)
  patch_get           — 자연어→CodeSpeak 번역 + 수정 위치 특정 (vib patch 와 같아요)
  anchor_run          — 앵커 자동 삽입    (vib anchor --auto 와 같아요)
  anchor_list         — 앵커 목록 보기
  explain_get         — 변경 내역 분석    (vib explain 와 같아요)
  config_get          — 현재 설정 확인

**🕐 언제 써?**

- Claude Code와 VibeLign을 함께 쓸 때
- Antigravity나 OpenCode도 같이 준비하고 싶을 때
- 매번 터미널에서 명령어 치기 귀찮을 때
- AI가 작업 전후를 자동으로 관리해주길 원할 때
- 어느 파일을 건드려야 할지 애매한 수정을 안전하게 하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib start --all-tools   # 여러 AI 도구를 한 번에 준비
"저장해줘"   # checkpoint_create 호출
"저장된 목록 보여줘"   # checkpoint_list 호출
"바이브라인 언두해줘"   # 목록 보여줌 → 번호 선택 → checkpoint_restore 자동 호출
"3번으로 복원해줘"   # checkpoint_restore 바로 호출
"바이브라인으로 로그인 버튼 크기 키워줘"   # patch_get → 수정 → guard_check → checkpoint 자동 실행
"바이브라인으로 다크모드 배경색 바꿔줘"   # 안전 수정 전체 워크플로우 자동 실행
"프로젝트 상태 진단해줘"   # doctor_run 호출
"방금 수정한 거 문제없는지 확인해줘"   # guard_check 호출
"앵커 없는 파일에 앵커 삽입해줘"   # anchor_run 호출
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `설정 방법` | vib start 한 번 실행 → Claude Code 재시작 |
| `여러 도구 한 번에 준비` | vib start --all-tools Claude, Antigravity, OpenCode, Cursor, Codex 준비를 한 번에 시도해요. |
| `바로 사용 가능` | 현재 start 안내 기준으로 Claude, Antigravity, OpenCode는 바로 사용 가능으로 보여줘요. |
| `마지막 확인 필요` | Cursor, Codex는 설정 화면이나 설정 파일에서 VibeLign 연결을 한 번 더 확인해야 할 수 있어요. |
| `'바이브라인으로' 키워드` | 요청 앞에 붙이면 안전 수정 모드. 로직·기능 변경 시 권장 |
| `일반 수정` | 단순 텍스트 변경, 오타 수정은 그냥 말해도 돼요 |
| `복원은 시점 지정 필수` | '되돌려줘'만 하면 AI가 목록을 먼저 보여줘요. 번호나 메시지를 함께 말하세요 |
| `한 번에 여러 요청 가능` | '저장하고, 앵커 확인하고, 작업 시작해줘' → AI가 순서대로 자동 처리 |

## 📋 AI 개발 규칙

### 📋 `VibeLign AI 개발 규칙`

> AI가 코드를 작성할 때 반드시 지켜야 할 규칙들이에요

**💡 이게 뭐야?**

vib start를 실행하면 프로젝트에 AI_DEV_SYSTEM_SINGLE_FILE.md 파일이 생겨요.
이 파일이 AI한테 '이렇게 코딩해야 해'를 알려주는 규칙서예요.
Claude Code, OpenCode, Cursor 등 어떤 AI를 써도 이 규칙을 읽고 따라요.

━━ 핵심 원칙 ━━

🔹 패치 우선 원칙
  AI는 파일 전체를 다시 쓰지 않아요.
  꼭 필요한 부분만 최소한으로 수정해요.
  관련 없는 파일은 절대 건드리지 않아요.

━━ 두 가지 수정 방식 ━━

방식 1 — 일반 수정 (기본)
  평소처럼 말하면 AI가 알아서 수정해요.
  예: '로그인 버튼 색 파란색으로 바꿔줘'

방식 2 — 바이브라인 안전 수정 ('바이브라인으로' 또는 'vibelign')
  요청에 '바이브라인으로'를 붙이면 전체 안전 워크플로우 자동 실행:
  patch_get → 정확한 위치 확인 → 수정 → guard_check → checkpoint 저장
  예: '바이브라인으로 로그인 버튼 색 파란색으로 바꿔줘'

  ⚠️  '바이브라인으로' 없으면 → AI가 직접 수정 (VibeLign 툴 사용 안 함)

━━ 파일 구조 규칙 ━━

🔹 진입 파일 보호 (main.py, index.js 등)
  진입 파일은 작게 유지해요. 실제 로직은 별도 파일로 분리해요.

🔹 UI와 비즈니스 로직 분리
  UI 파일: 화면 레이아웃, 버튼, 입력창만
  로직 파일: 실제 처리, 파일 읽기/쓰기, 네트워크
  두 가지를 한 파일에 섞으면 안 돼요.

🔹 파일 크기 관리
  파일이 커지면 역할별로 나눠요.
  utils.py, helpers.py, misc.py 같은 모호한 이름 금지.
  대신: backup_worker.py, translation_pipeline.py처럼 구체적으로 지어요.

━━ 함수 설계 규칙 ━━

🔹 함수 길이
  함수가 40줄 넘으면 → 분리 고려
  함수가 80줄 넘으면 → 반드시 분리
  한 함수는 딱 한 가지 일만 해요.

🔹 함수 이름
  좋은 예: load_config(), parse_excel_row(), validate_input_path()
  나쁜 예: do_stuff(), process(), handle(), run_all()

🔹 파일 간 연결
  순환 import 금지 (A가 B를 부르고, B가 다시 A를 부르는 구조)
  공통 로직은 별도 파일로 분리해서 양쪽에서 가져다 써요.

━━ 유지보수 규칙 (코알못도 관리할 수 있게) ━━

🔹 매직 넘버 금지
  나쁜 예: if retry > 3
  좋은 예: MAX_RETRY = 3 / if retry > MAX_RETRY

🔹 에러 메시지는 사람이 읽을 수 있게
  나쁜 예: raise Exception('NoneType')
  좋은 예: raise Exception('파일을 찾을 수 없어요. 경로를 확인하세요')

🔹 조용한 실패 금지
  except: pass 절대 금지
  에러가 나면 반드시 화면에 표시해요.

🔹 죽은 코드 제거
  주석 처리된 코드 덩어리 남기지 않아요.
  쓰지 않는 import, 변수, 함수 삭제해요.

🔹 의존성 동기화
  새 패키지를 추가하면 pyproject.toml도 함께 업데이트해요.

🔹 주석도 함께 업데이트
  코드를 바꾸면 위에 있는 설명 주석도 같이 바꿔요.
  오래된 주석은 없는 것보다 더 혼란스러워요.

━━ 앵커 규칙 ━━

🔹 앵커(ANCHOR)가 있으면 그 안에서만 수정
  앵커를 무시하고 파일 전체를 바꾸면 안 돼요.
  앵커가 없는 큰 파일은 먼저 앵커를 달고 작업해요.

🔹 앵커를 허락 없이 지우면 안 돼요

**🕐 언제 써?**

- AI가 내 규칙을 제대로 따르고 있는지 확인하고 싶을 때
- '바이브라인으로' 키워드를 언제 써야 하는지 헷갈릴 때
- AI가 이상하게 코딩할 때 어떤 규칙이 있는지 확인할 때
- 내 프로젝트 규칙을 처음부터 다시 보고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib manual rules   # 전체 규칙 보기
vib start   # 프로젝트에 규칙 파일(AI_DEV_SYSTEM_SINGLE_FILE.md) 생성
cat AI_DEV_SYSTEM_SINGLE_FILE.md   # 규칙 파일 직접 보기
```

| `어떤 AI든 적용` | AGENTS.md를 읽는 Claude Code, OpenCode, Cursor 모두 자동 적용 |
