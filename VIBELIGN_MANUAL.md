## 🏁 처음 시작

### 🚀 `vib start`

> 처음 시작할 때 딱 한 번만 실행해요

**💡 이게 뭐야?**

새 프로젝트에서 VibeLign을 처음 쓸 때 필요한 파일을 자동으로 만들어줘요.
AGENTS.md, AI_DEV_SYSTEM.md 같은 파일이 생기는데,
이게 있어야 AI가 내 프로젝트를 제대로 이해하고 작업해요.

**🕐 언제 써?**

- 새 프로젝트 폴더에서 VibeLign을 처음 쓸 때
- AI한테 코딩을 맡기기 전에 한 번만 세팅할 때
- 설치 직후 처음 설정할 때

**✏️ 이렇게 쳐봐**

```
vib start   # 기본 세팅 (가장 많이 씀)
vib start --quickstart   # 세팅 + 앵커 자동 삽입까지 한 번에
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `--quickstart` | start 실행 후 anchor --auto 까지 자동으로 실행해요. 한 번에 다 끝내고 싶을 때 써요. |

### 🔄 `vib init`

> VibeLign을 최신 버전으로 다시 설치해요

**💡 이게 뭐야?**

VibeLign을 최신 버전으로 재설치해요.
업데이트가 있을 때, 또는 뭔가 이상할 때 쓰세요.
uv 또는 pip을 자동으로 감지해서 설치해요.

**🕐 언제 써?**

- VibeLign 업데이트를 하고 싶을 때
- 설치가 꼬인 것 같을 때 다시 설치하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib init   # 최신 버전으로 재설치
vib init --force   # 강제로 다시 설치
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `--force` | 이미 최신 버전이어도 강제로 다시 설치해요. |

## 💾 세이브 & 되돌리기

### 💾 `vib checkpoint`

> 게임 세이브처럼 지금 상태를 저장해요

**💡 이게 뭐야?**

AI한테 코드를 맡기기 전에 현재 상태를 저장해두는 기능이에요.
뭔가 잘못되면 이 시점으로 되돌릴 수 있어요.
게임에서 '세이브' 버튼 누르는 것과 똑같아요.

**🕐 언제 써?**

- AI한테 수정을 시키기 바로 직전
- 기능이 완성됐을 때 기록을 남기고 싶을 때
- 큰 작업을 시작하기 전에 안전망을 만들 때

**✏️ 이렇게 쳐봐**

```
vib checkpoint   # 빠르게 저장
vib checkpoint "로그인 완성"   # 메시지와 함께 저장
vib checkpoint "버그 수정 전"   # 작업 전 백업
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `message` | 저장할 때 메모를 남길 수 있어요. 나중에 어떤 상태였는지 알기 쉬워요. 예: vib checkpoint "회원가입 버튼 추가" |

---

### ⏪ `vib undo`

> 저장한 곳으로 되돌려요

**💡 이게 뭐야?**

AI가 코드를 이상하게 바꿨을 때, 마지막으로 저장한 시점으로 되돌려요.
게임에서 '불러오기' 버튼 누르는 것과 똑같아요.
checkpoint로 저장해뒀어야 쓸 수 있어요.

**🕐 언제 써?**

- AI가 코드를 망가뜨렸을 때
- 수정 결과가 마음에 들지 않을 때
- 실수로 파일을 지웠을 때

**✏️ 이렇게 쳐봐**

```
vib undo   # 마지막 저장으로 되돌리기
vib undo --list   # 저장 목록 보고 선택해서 되돌리기
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `--list` | 저장된 체크포인트 목록을 보여줘요. 원하는 시점을 골라서 되돌릴 수 있어요. |

---

### 📋 `vib history`

> 저장 목록을 봐요

**💡 이게 뭐야?**

지금까지 checkpoint로 저장한 기록을 보여줘요.
언제 저장했는지, 어떤 메시지를 남겼는지 확인할 수 있어요.

**🕐 언제 써?**

- 내가 언제 저장했는지 확인하고 싶을 때
- 어떤 버전으로 되돌릴지 결정하기 전에

**✏️ 이렇게 쳐봐**

```
vib history   # 저장 기록 전체 보기
```

## 🔬 점검 & 확인

### 🩺 `vib doctor`

> 프로젝트 건강 상태를 확인해요

**💡 이게 뭐야?**

내 프로젝트가 AI 작업을 받기에 괜찮은 상태인지 점검해요.
점수로 보여주고, 문제가 있으면 뭐가 문제인지 알려줘요.
마치 의사가 건강검진 해주는 것처럼요.

**🕐 언제 써?**

- AI한테 코딩을 시키기 전에 준비됐는지 확인할 때
- 프로젝트 상태가 좋은지 수시로 확인할 때
- 뭔가 이상한 것 같을 때 진단받고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib doctor   # 기본 점검 (점수 + 문제 목록)
vib doctor --strict   # 더 꼼꼼하게 점검
vib doctor --detailed   # 문제마다 자세한 설명 포함
vib doctor --fix   # 앵커 없는 파일에 자동으로 앵커 추가
vib doctor --write-report   # 결과를 파일로 저장
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `--strict` | 기본 점검보다 더 꼼꼼하게 검사해요. 작은 문제도 놓치지 않아요. |
| `--detailed` | 각 문제마다 왜 문제인지 설명을 추가로 보여줘요. |
| `--fix-hints` | 각 문제를 어떻게 고치면 되는지 힌트를 줘요. |
| `--fix` | 앵커가 없는 파일에 자동으로 앵커를 달아줘요. 직접 하기 귀찮을 때 편해요. |
| `--write-report` | 점검 결과를 파일로 저장해요. 나중에 다시 볼 수 있어요. |
| `--json` | 결과를 JSON 형식으로 출력해요. (개발자용) |

### 🛡️ `vib guard`

> AI가 코드를 망가뜨리지 않았는지 검사해요

**💡 이게 뭐야?**

AI가 코드를 수정한 후, 구조가 망가지지 않았는지 종합 검사해요.
doctor (건강 점검) + explain (변경 설명)을 합친 거예요.
AI 작업 후 항상 한 번 실행해보세요.

**🕐 언제 써?**

- AI가 코드를 수정한 직후
- 뭔가 이상한 것 같을 때 종합 검진을 받고 싶을 때
- AI 작업 결과를 확인하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib guard   # 기본 검사
vib guard --strict   # 더 꼼꼼하게 검사
vib guard --write-report   # 결과를 파일로 저장
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `--strict` | 더 꼼꼼하게 검사해요. 작은 문제도 잡아줘요. |
| `--since-minutes 60` | 최근 몇 분 동안의 변경만 확인해요. 기본값은 120분이에요. |
| `--write-report` | 검사 결과를 파일로 저장해요. |
| `--json` | 결과를 JSON 형식으로 출력해요. (개발자용) |

### 📖 `vib explain`

> 뭐가 바뀌었는지 쉽게 알려줘요

**💡 이게 뭐야?**

최근에 코드가 어떻게 바뀌었는지 쉬운 말로 설명해줘요.
AI가 수정한 내용을 이해하기 어려울 때 쓰세요.
개발 경험 없어도 "이 파일에서 이런 게 바뀌었어요" 수준으로 알려줘요.

**🕐 언제 써?**

- AI가 수정한 내용이 뭔지 이해하고 싶을 때
- 어떤 파일이 바뀌었는지 확인하고 싶을 때
- 변경 내역을 누군가한테 설명해야 할 때

**✏️ 이렇게 쳐봐**

```
vib explain   # 전체 변경 설명
vib explain main.py   # 특정 파일만 설명
vib explain --ai   # AI가 더 자세하게 분석
vib explain --since-minutes 30   # 최근 30분 변경만 보기
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `file` | 특정 파일의 변경만 설명해요. 예: vib explain login.py |
| `--ai` | AI가 변경 내용을 더 자세하게 분석해서 설명해줘요. |
| `--since-minutes 숫자` | 최근 몇 분 동안의 변경만 보여줘요. 기본값은 120분이에요. |
| `--write-report` | 설명 결과를 파일로 저장해요. |
| `--json` | 결과를 JSON 형식으로 출력해요. (개발자용) |

## ✏️ AI 수정 요청

### 🛠️ `vib patch`

> 말로 요청하면 안전한 수정 계획을 만들어요

**💡 이게 뭐야?**

"로그인 버튼 추가해줘" 같이 말로 요청하면,
어떤 파일의 어느 부분을 어떻게 수정할지 계획을 세워줘요.
이 계획을 복사해서 AI (Claude, ChatGPT 등)에 붙여넣으면 돼요.

**🕐 언제 써?**

- AI한테 코드 수정을 요청할 때 더 정확하게 전달하고 싶을 때
- 어떤 파일을 수정해야 할지 모를 때 도움받고 싶을 때
- 수정 전에 계획을 먼저 확인하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib patch "로그인 버튼 추가"   # 수정 계획 만들기
vib patch "버그 수정" --ai   # AI가 더 정확하게 분석
vib patch "사이드바 제거" --preview   # 미리 보기
vib patch "기능 추가" --copy   # 결과를 클립보드에 복사
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `request` | 수정 요청을 말로 써요. 예: vib patch "다크모드 추가해줘" |
| `--ai` | AI가 코드를 더 자세히 분석해서 정확한 계획을 세워요. 조금 더 시간이 걸려요. |
| `--preview` | 수정 계획을 미리 보기로 확인해요. |
| `--copy` | AI에 전달할 프롬프트를 클립보드에 복사해요. 바로 붙여넣기할 수 있어요. |
| `--write-report` | 수정 계획을 파일로 저장해요. |
| `--json` | 결과를 JSON 형식으로 출력해요. (개발자용) |

### ⚓ `vib anchor`

> AI가 정확한 위치를 찾을 수 있도록 표식을 달아요

**💡 이게 뭐야?**

코드 파일에 '여기서부터 여기까지가 이 기능이야'라는 표식(앵커)을 달아요.
이 표식이 있으면 AI가 수정할 때 정확한 위치를 찾을 수 있어요.
지도에 핀 꽂는 것처럼요.

**🕐 언제 써?**

- AI한테 처음으로 코딩을 시키기 전에 (한 번만 하면 돼요)
- 새 파일을 추가했을 때
- 앵커가 제대로 있는지 확인하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib anchor --auto   # 모든 파일에 자동으로 앵커 달기 (가장 많이 씀)
vib anchor --suggest   # 어떻게 달면 좋을지 추천만 보기 (실제로 바꾸지 않음)
vib anchor --validate   # 앵커가 제대로 달려있는지 검사
vib anchor --dry-run   # 실제로 바꾸지 않고 어떻게 바뀔지 미리 보기
vib anchor --only-ext .py   # Python 파일만 처리
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `--auto` | 모든 파일에 자동으로 앵커를 달아줘요. 처음 설정할 때 이걸 써요. |
| `--suggest` | 앵커를 어떻게 달면 좋을지 추천만 보여줘요. 실제로 파일을 바꾸지는 않아요. |
| `--validate` | 앵커가 올바르게 달려있는지 검사해요. 짝이 안 맞는 앵커를 찾아줘요. |
| `--dry-run` | 실제로 바꾸지 않고 어떻게 바뀔지만 미리 보여줘요. |
| `--only-ext .py` | 특정 확장자의 파일만 처리해요. 예: --only-ext .py (파이썬만), --only-ext .js (자바스크립트만) |
| `--json` | 결과를 JSON 형식으로 출력해요. (개발자용) |

---

### 🔍 `vib scan`

> 앵커 스캔 + 코드맵 갱신을 한 번에 해요

**💡 이게 뭐야?**

앵커 검사, 앵커 인덱스 갱신, 코드맵 재생성을 한 번에 실행해요.
anchor 와 start 를 따로 실행하지 않아도 돼요.
뭔가 꼬인 것 같을 때 이걸 실행하면 대부분 해결돼요.

**🕐 언제 써?**

- 파일을 많이 추가하거나 삭제했을 때
- 앵커가 꼬인 것 같을 때 한 번에 정리하고 싶을 때
- 코드맵을 최신 상태로 갱신하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib scan   # 앵커 스캔 + 코드맵 갱신
vib scan --auto   # 문제 있는 앵커 자동 수정 + 코드맵 갱신
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `--auto` | 문제 있는 앵커를 자동으로 고쳐줘요. 앵커를 지웠다가 다시 달아서 깨끗하게 만들어요. |

## 🗂️ 파일 & 설정

### 🔒 `vib protect`

> 중요한 파일을 AI가 건드리지 못하게 잠가요

**💡 이게 뭐야?**

절대 바뀌면 안 되는 파일을 보호해요.
보호된 파일은 AI가 수정 대상으로 포함시키지 않아요.
설정 파일, 환경 변수 파일 등에 쓰면 좋아요.

**🕐 언제 써?**

- .env 같은 설정 파일을 AI가 건드리지 않았으면 할 때
- 중요한 파일을 실수로 수정하는 걸 막고 싶을 때
- 보호 목록을 관리하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib protect main.py   # main.py 보호
vib protect .env   # .env 파일 보호
vib protect --list   # 보호된 파일 목록 보기
vib protect main.py --remove   # main.py 보호 해제
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `file` | 보호할 파일 이름을 써요. 예: vib protect settings.py |
| `--list` | 현재 보호되어 있는 파일 목록을 보여줘요. |
| `--remove` | 보호를 해제해요. 예: vib protect main.py --remove |

---

### 💬 `vib ask`

> 파일이 뭘 하는지 쉬운 말로 설명해줘요

**💡 이게 뭐야?**

코드 파일을 AI가 분석해서 쉬운 말로 설명해줘요.
코드를 모르는 사람도 이 파일이 뭘 하는 건지 알 수 있어요.
특정 질문도 할 수 있어요.

**🕐 언제 써?**

- 이 파일이 뭘 하는 건지 궁금할 때
- AI가 만든 코드를 이해하고 싶을 때
- 특정 파일에 대해 질문하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib ask main.py   # main.py가 뭘 하는지 설명
vib ask login.py "비밀번호는 어디서 확인해?"   # 특정 질문하기
vib ask app.py --write   # 설명을 파일로 저장
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `file` | 설명할 파일 이름이에요. 예: vib ask login.py |
| `question` | 파일에 대해 궁금한 걸 물어볼 수 있어요. 예: vib ask login.py "이 함수는 뭐야?" |
| `--write` | 설명 결과를 VIBELIGN_ASK.md 파일로 저장해요. |

### 🔑 `vib config`

> AI 기능을 쓰기 위한 API 키를 설정해요

**💡 이게 뭐야?**

AI 기능(ask, patch --ai 등)을 사용하려면 API 키가 필요해요.
이 명령어로 Anthropic(Claude) 또는 Gemini API 키를 설정할 수 있어요.

**🕐 언제 써?**

- 처음 설치하고 AI 기능을 쓰려고 할 때
- API 키를 바꾸고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib config   # API 키 설정 시작
```

### 📤 `vib export`

> Claude, Cursor 등 AI 도구에 맞는 설정 파일을 만들어요

**💡 이게 뭐야?**

Claude Code, Cursor, OpenCode 같은 AI 도구에서
VibeLign을 연동해서 쓸 수 있도록 설정 파일을 만들어줘요.

`vib export cursor`는 `vibelign_exports/cursor/` 템플릿과 프로젝트 루트 `.cursorrules`를 준비해줘요.
Cursor MCP 연결 자체는 `vib start`가 `.cursor/mcp.json`에 자동 등록해줘요.

예를 들면 이런 구조가 만들어져요:

```text
your-project/
  .cursor/
    mcp.json
  .cursorrules
  vibelign_exports/
    cursor/
      RULES.md
      SETUP.md
      PROMPT_TEMPLATE.md
```

- `.cursor/mcp.json` → Cursor가 실제로 쓰는 MCP 연결 설정
- `.cursorrules` → Cursor가 읽는 프로젝트 규칙
- `vibelign_exports/cursor/` → 참고용 템플릿과 설명 문서

**🕐 언제 써?**

- Claude Code를 쓰면서 VibeLign도 같이 쓰고 싶을 때
- Cursor에서 VibeLign 설정을 적용하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib export claude   # Claude Code용 설정 파일 생성
vib export cursor   # Cursor용 설정 파일 생성
vib export opencode   # OpenCode용 설정 파일 생성
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `tool` | 어떤 AI 도구용인지 선택해요. 선택지: claude / opencode / cursor / antigravity |

### 👁️ `vib watch`

> 파일이 바뀔 때마다 자동으로 코드맵을 갱신해요

**💡 이게 뭐야?**

파일이 저장될 때마다 자동으로 코드맵을 최신 상태로 유지해요.
AI가 작업하는 동안 켜두면 항상 최신 정보로 작업할 수 있어요.
watchdog 패키지가 설치되어 있어야 해요. (vib start 에서 자동 설치 제안)

**🕐 언제 써?**

- AI가 코딩하는 동안 백그라운드로 켜두고 싶을 때
- 파일이 자주 바뀌는 작업 중에
- 코드맵을 항상 최신 상태로 유지하고 싶을 때

**✏️ 이렇게 쳐봐**

```
vib watch   # 실시간 감시 시작 (Ctrl+C 로 종료)
vib watch --strict   # 더 꼼꼼한 감시 모드
vib watch --write-log   # 감시 로그를 파일로 저장
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `--strict` | 더 꼼꼼하게 감시해요. 작은 변화도 잡아줘요. |
| `--write-log` | 감시 중 발생한 일을 파일로 기록해요. |
| `--debounce-ms 숫자` | 파일이 바뀐 후 몇 밀리초 후에 처리할지 설정해요. 기본값은 800ms예요. 너무 자주 갱신되면 늘려요. |
| `--json` | 결과를 JSON 형식으로 출력해요. (개발자용) |

### ⌨️ `vib completion`

> 탭키로 명령어를 자동완성해요

**💡 이게 뭐야?**

vib + 탭키를 누르면 명령어 목록이 뜨게 해줘요.

**🕐 언제 써?**

- 명령어 이름이 기억 안 날 때

**✏️ 이렇게 쳐봐**

```
vib completion --install   # 자동완성 자동 설정 (추천!)
vib completion   # 설정 방법 안내 보기
```

**⚙️ 옵션 설명**

| 옵션 | 설명 |
|------|------|
| `--install` | 자동완성을 자동으로 설정해줘요. zsh/bash/PowerShell을 자동으로 감지해요. 새 터미널을 열면 바로 사용할 수 있어요. |

## 🤖 MCP (AI 자동 연동)

### 🤖 MCP가 뭐야? {#mcp-intro}

**💡 MCP가 뭔지 쉽게 설명하면?**

여러분이 Claude Code 같은 AI랑 대화할 때,
보통은 AI한테 "체크포인트 저장해줘"라고 말하면 AI가 명령어를 알려주고,
여러분이 직접 터미널에서 `vib checkpoint`를 쳐야 해요.

MCP가 연결되면 달라져요.
AI한테 "작업 전에 저장해줘"라고 하면,
**AI가 직접 VibeLign을 실행해서 체크포인트를 저장해요.**
여러분은 터미널을 따로 열 필요가 없어요.

```
[MCP 없을 때]
나: "저장해줘"
AI: "vib checkpoint 명령어를 실행하세요"
나: (직접 터미널에서 타이핑) vib checkpoint "저장"

[MCP 있을 때]
나: "저장해줘"
AI: (자동으로 VibeLign을 실행해서 체크포인트 저장) 완료!
```

**🕐 MCP는 누가 쓰는 거야?**

- **Claude Code 사용자**: Claude Code와 VibeLign을 함께 쓰는 사람
- **자동화를 원하는 사람**: 매번 명령어 치기 귀찮을 때
- **AI 워크플로우를 쓰는 사람**: AI가 코딩 작업 전체를 관리해주길 원할 때

**⚙️ MCP 연결하는 방법**

`vib start`를 실행하면 자동으로 설정돼요. 딱 한 번만 하면 돼요.

```
vib start   # Claude Code와 Cursor에 MCP가 자동 등록돼요
```

이후 Claude Code나 Cursor를 재시작하면 AI가 자동으로 VibeLign 기능을 쓸 수 있어요.

> 주의: MCP를 쓰려면 `vib start` 이후 Claude Code나 Cursor를 한 번 재시작해야 해요.

---

### ⚡ 두 가지 수정 방식 {#mcp-two-modes}

VibeLign MCP가 연결되면 AI 코드 수정에 **두 가지 방식**을 쓸 수 있어요.

---

#### 방식 1 — 일반 AI 수정 (기본)

평소처럼 AI한테 말하면 AI가 알아서 판단해서 수정해요.

```
나: "로그인 버튼 색 파란색으로 바꿔줘"
AI: (직접 코드 수정)
```

빠르고 간단한 요청에 적합해요.

---

#### 방식 2 — 바이브라인 안전 수정 (키워드 트리거)

요청에 **"바이브라인으로"** 를 붙이면 VibeLign이 전체 안전 워크플로우를 자동으로 실행해요.

```
나: "바이브라인으로 로그인 버튼 색 파란색으로 바꿔줘"

AI가 자동으로:
  1. patch_get 호출 → 어느 파일 어느 위치인지 정확히 확인
  2. 그 위치만 수정 (다른 곳은 절대 건드리지 않음)
  3. guard_check 호출 → 수정 후 이상 여부 자동 확인
  4. checkpoint_create 호출 → 현재 상태 자동 저장
```

---

**두 방식 비교**

| | 방식 1 (일반) | 방식 2 (바이브라인으로) |
|--|--|--|
| 속도 | 빠름 | 조금 느림 |
| 수정 위치 | AI가 추측 | 앵커로 정확히 확정 |
| 엉뚱한 파일 수정 | 가능 | 방지됨 |
| 작업 후 검증 | 없음 | guard_check 자동 실행 |
| 자동 저장 | 없음 | checkpoint 자동 생성 |
| 언제 써? | 단순 수정, 텍스트 변경 | 로직 변경, 기능 추가, 구조 변경 |

---

**💬 "바이브라인으로"를 붙여야 할 때**

```
"바이브라인으로 회원가입 유효성 검사 추가해줘"
"바이브라인으로 다크모드 배경색 바꿔줘"
"바이브라인으로 UI 확대·축소해도 화면이 따라오게 해줘"
"바이브라인으로 로그인 오류 메시지 한국어로 바꿔줘"
```

> 어느 파일을 건드려야 할지 확신이 없거나, 기능이 달라지는 수정이라면 "바이브라인으로"를 붙이세요.
> AI가 틀린 위치를 건드리는 걸 VibeLign이 막아줘요.

---

### 💾 `checkpoint_create` {#mcp-checkpoint-create}

> AI가 작업 전에 자동으로 현재 상태를 저장해요

**💡 이게 뭐야?**

`vib checkpoint "메시지"`와 완전히 같아요.
AI가 코드를 수정하기 전에 이 기능을 자동으로 실행해서 안전망을 만들어요.

**🕐 언제 쓰여?**

- AI가 코드 수정을 시작하기 전에 자동으로
- "저장하고 작업 시작해줘"라고 말했을 때
- 중요한 작업 전에 백업이 필요할 때

**💬 AI한테 이렇게 말해봐**

```
"로그인 기능 추가하기 전에 체크포인트 저장해줘"
"지금 상태 저장하고 작업 시작해줘"
"작업 전에 백업해줘"
```

**📥 AI에게 전달되는 정보**

| 항목 | 설명 |
|------|------|
| `message` | 저장할 때 남길 메모. 예: "로그인 완성" |

---

### 📋 `checkpoint_list` {#mcp-checkpoint-list}

> 지금까지 저장한 체크포인트 목록을 보여줘요

**💡 이게 뭐야?**

`vib history`와 같아요.
언제 어떤 내용을 저장했는지 목록으로 확인해요.
주로 되돌리기 전에 어떤 시점이 있는지 확인할 때 써요.

**💬 AI한테 이렇게 말해봐**

```
"저장된 체크포인트 목록 보여줘"
"언제 저장했는지 알려줘"
"어떤 버전이 있어?"
```

**📤 AI가 알려주는 정보**

```
# 체크포인트 목록

- 20260318T003633...  오늘 00:36  로그인 완성
- 20260318T003056...  오늘 00:30  버튼 추가
- 20260317T224100...  어제 22:41  초기 세팅
```

---

### ⏪ `checkpoint_restore` {#mcp-checkpoint-restore}

> 특정 체크포인트 시점으로 프로젝트를 되돌려요

**💡 이게 뭐야?**

AI가 망쳤을 때 "바이브라인 언두해줘"라고 하면 AI가 직접 복원해줘요.

> ⚠️ **중요**: 터미널에서 `vib undo`를 직접 치면 MCP 환경에서 멈춰버려요!
> AI한테 말로 부탁하면 AI가 알아서 처리해줘요.

**실제 흐름:**

```
나: "바이브라인 언두해줘"

AI: (자동으로)
  1. checkpoint_list() 호출 → 저장 목록을 보여줌

  📋 저장 목록:
  1. 로그인 기능 추가 완료 (2시간 전)
  2. UI 색상 변경 (1시간 전)
  3. 버튼 크기 조정 (30분 전)

  "몇 번으로 복원할까요?"

나: "2번으로 해줘"

AI: checkpoint_restore("2번 ID") → 복원 완료!
```

**🕐 언제 쓰여?**

- AI가 코드를 이상하게 바꿨을 때
- "아까 상태로 돌아가줘"라고 할 때
- 특정 시점의 코드로 돌아가고 싶을 때

**💬 AI한테 이렇게 말해봐**

```
"바이브라인 언두해줘"
"이전 상태로 되돌려줘"
"저장 목록 보고 복원해줘"
"3번으로 복원해줘"   ← 번호를 알면 바로 지정해도 돼요
```

### 📄 `project_context_get` {#mcp-project-context-get}

> 토큰이 부족하거나 AI를 바꿀 때 새 AI가 프로젝트를 즉시 파악할 수 있게 해줘요

**💡 이게 뭐야?**

바이브코딩을 하다 보면 대화가 길어져서 토큰이 부족해지거나,
Claude에서 ChatGPT로, ChatGPT에서 Cursor로 AI를 갈아타야 할 때가 있어요.

그럴 때마다 새 AI한테 "우리 프로젝트가 이런 거고, 지금까지 이걸 하고 있었어..."
처음부터 다시 설명하면 시간도 걸리고 토큰도 낭비돼요.

`project_context_get`을 호출하면 이 프로젝트가 뭔지, 어떤 파일이 있는지,
최근에 어떤 작업을 했는지를 요약한 파일을 만들어줘요.
새 AI한테 이 파일 하나만 던져주면 바로 이어서 작업할 수 있어요.

**🕐 언제 쓰여?**

- 대화가 길어져서 토큰이 부족할 때 새 채팅창을 열기 전에
- Claude에서 ChatGPT로, 또는 다른 AI로 갈아탈 때
- 새 AI 세션을 시작할 때 처음부터 설명하기 귀찮을 때

**💬 AI한테 이렇게 말해봐**

```
"토큰이 부족하니까 컨텍스트 파일 만들어줘"
"ChatGPT로 넘어가려고. 프로젝트 요약해줘"
"새 채팅창 열 건데 지금까지 한 거 요약해줘"
```

**📥 AI에게 전달되는 정보**

| 항목 | 설명 |
|------|------|
| `compact` | 간략 모드. 토큰을 최대한 아끼고 싶을 때. |
| `full` | 전체 모드. 더 자세한 맥락이 필요할 때. |

### 🛡️ `guard_check` {#mcp-guard-check}

> AI 작업 후 코드가 안전한지 자동으로 확인해요

**💡 이게 뭐야?**

`vib guard`와 같아요.
AI가 코드를 수정한 후, 이상한 게 없는지 스스로 점검해요.
AI가 실수했을 때 스스로 발견할 수 있어요.

**🕐 언제 쓰여?**

- AI가 코드 수정을 마친 직후 자동으로
- "방금 수정한 거 괜찮아?" 라고 물었을 때
- 작업 후 이상한 게 없는지 확인할 때

**💬 AI한테 이렇게 말해봐**

```
"방금 수정한 거 문제없는지 확인해줘"
```

| `since_minutes` | 최근 몇 분 이내 변경사항만 확인. 기본값: 30분 |

### 🛠️ `patch_get` {#mcp-patch-get}

> 말로 한 요청을 AI가 이해하는 정밀 언어(CodeSpeak)로 번역하고, 수정할 파일과 위치를 딱 찍어줘요

**💡 이게 뭐야?**

"UI 확대·축소해도 화면이 따라오게 해줘"라고 말했을 때 — AI는 이게 어떤 파일인지, 어느 함수인지 모르면서 코드 전체를 뒤져요. 그러다 엉뚱한 곳을 건드리는 일이 생겨요.

`patch_get`은 두 가지 일을 해요.

1. **번역**: 여러분의 자연어 요청을 **CodeSpeak**라는 구조화된 지시어로 변환해요.
   → AI가 "이 파일의 이 구역에서, 이 동작만 바꿔"라고 명확하게 이해할 수 있는 형태가 돼요.

2. **위치 특정**: 프로젝트 앵커 인덱스를 검색해서 수정할 파일(`target_file`)과 코드 구역(`target_anchor`)을 딱 집어내요.

```
[patch_get 없을 때]
나: "UI 확대·축소해도 화면이 따라오게 해줘"
AI: 파일 전체 탐색 → 비슷한 코드 여러 개 발견 → 추측해서 수정 → 엉뚱한 곳 건드림

[patch_get 있을 때]
나: "UI 확대·축소해도 화면이 따라오게 해줘"
AI → patch_get 자동 호출
VibeLign: target_file  = ui/canvas.py
          target_anchor = CANVAS_RESIZE_HANDLER  ← 딱 이 구역만
          codespeak     = "MODIFY canvas_resize: ADD viewport_follow_scale"
          constraints   = 이 파일만, 구조 유지
AI: 정확한 위치 + 정확한 지시를 받고 수정 → 실수 없음
```

> **CodeSpeak란?** 자연어처럼 읽히지만 AI가 오해할 여지가 없는 구조화된 수정 언어예요.
> "버튼 키워줘" → `MODIFY login_button: SCALE size * 1.5` 이런 식으로요.

**🕐 언제 쓰여?**

- AI가 코드 수정을 시작하기 전에 자동으로 (AGENTS.md 지침에 따라)
- 수정 요청이 어느 파일인지 애매할 때
- AI가 관련 없는 코드까지 건드리는 걸 막고 싶을 때

**💬 AI한테 이렇게 말해봐**

```
"UI 확대·축소해도 화면이 따라오게 해줘"
"로그인 버튼 크기 키워줘"
"다크모드 배경색 바꿔줘"
```

직접 patch_get을 부르지 않아도 돼요. AGENTS.md 규칙이 있으면 AI가 코드 수정 전에 알아서 호출해요.

**📥 AI에게 전달되는 정보**

| 항목 | 설명 |
|------|------|
| `request` | 수정 요청을 말로 써요. 예: "로그인 버튼 크기 키워줘" |

**📤 AI가 받는 정보**

| 항목 | 설명 |
|------|------|
| `target_file` | 수정할 파일 경로 |
| `target_anchor` | 수정할 코드 구역 (앵커 이름) |
| `codespeak` | 요청을 AI가 이해하기 쉬운 언어로 변환한 것 |
| `constraints` | 이 범위 밖은 건드리지 말라는 제한 |
| `confidence` | VibeLign이 이 분석을 얼마나 확신하는지 |

> 참고: `vib patch --ai`는 Gemini 무료 API로 분석을 강화해요. MCP 버전은 연결된 AI(Claude, Codex 등)가 직접 분석하니까 API 키가 필요 없어요.

### 🔄 MCP 실전 활용 예시

**시나리오 1: AI한테 기능 추가 맡기기**

```
나: "로그인 기능 추가해줘. 근데 먼저 저장하고 시작해"

AI: (자동으로)
  1. checkpoint_create("로그인 기능 추가 전") → 저장 완료
  2. 코드 수정 시작...
  3. guard_check() → 문제없음 확인
  4. "완료했어요! 문제 있으면 체크포인트로 되돌릴 수 있어요"
```

**시나리오 2: AI가 코드 망쳤을 때**

```
나: "뭔가 이상해, 되돌려줘"

AI: (자동으로)
  1. checkpoint_list() → 목록 확인
  2. checkpoint_restore("로그인 기능 추가 전") → 복원 완료
  3. "이전 상태로 돌아왔어요"
```

**시나리오 3: AI 도구를 바꿀 때**

```
나: "ChatGPT한테 이어서 작업시키려고. 컨텍스트 요약해줘"

AI: (자동으로)
  1. project_context_get() → 프로젝트 요약 생성
  2. "이 내용을 ChatGPT에 붙여넣으세요: ..."
```
