# 베이비 모드 — 스크립트형 튜토리얼 설계

- 작성일: 2026-06-26
- 상태: 설계 확정 (구현 계획 대기)
- 맥락: "초보자에게 기능이 너무 많아 무겁다"는 지적 → 무거움(UX) 문제와 시장(교육 피보팅) 문제를 분리. 무거움부터 싸게 고치는 **베이비 모드**를 먼저 설계 (피보팅 가설은 별도 검증).

## 1. 목표와 비목표

### 목표 (두 축은 동급이며 서로를 떠받친다)
1. **진짜 작동하는 앱이 완성되는 것을 직접 경험한다.** "내가 만든 앱이 *실제로 돌아간다*"는 손맛 — 감정·동기·**심리적 허들 낮춤**. 원래 피보팅의 출발점이자 완주율·입소문의 보상. 끝에 사용자 폴더에 **진짜 작동하는 앱 하나**가 남고, 사용자가 그걸 *손으로 만져보며* "된다!"를 느낀다.
2. **그 과정에서 VibeLign을 제대로 쓰는 법(안전 루프: 기획 → 체크포인트 → AI작업 → 검사 → 저장/되돌리기)을 체득한다.** 다음에 혼자 반복할 수 있는 전이 능력.

→ ①이 없으면 ②는 공허하고(손에 남는 게 없음), ②가 없으면 ①은 일회성(또 못 만듦)이다. **둘 다 필수.** 만드는 앱은 ②의 수단이자 동시에 ①의 결과물이다.

- "누구라도 따라하면 된다"는 **완주율**, "끝에 작동 앱이 남는다"는 **완성 경험**, "졸업 후 혼자 같은 루프를 반복한다"는 **전이**가 함께 최우선 지표.
- 주의: "한 단계=한 행동, 판단 없음" 단순화가 과해 *기계적 클릭만 반복*하면 앱은 만들어져도 사용법 학습(②)이 빠진다 → 아래 §3의 "왜" 한 줄과 §5의 안전 절반 포함으로 방지.

### 비목표
- 기능을 삭제하지 않는다. (숨김/안내일 뿐, 제거 아님)
- 완성된 앱 소스코드를 번들하지 않는다. 사용자가 *자기 손으로* 만들게 한다.
- 이번 범위에서 K-12 교육용 리포지셔닝(차시·교사 화면·연령 콘텐츠)은 다루지 않는다.

## 2. 핵심 개념: 두 가이드의 공존

| | 기존 가이드 레이어 (`guide.ts`) | 신규 베이비 모드 튜토리얼 |
|---|---|---|
| 성격 | 반응형 (GPS) | 스크립트형 (요리 교실) |
| 동작 | 현재 상태를 추론해 다음 한 걸음 넌지시 안내 | 정해진 완성품을 향해 손잡고 끌고 감 |
| 끝 | 없음 (계속 안내) | 작동하는 앱 1개 완성 후 **졸업** |
| 대상 | 어느 정도 흐름을 아는 사용자 | 완전 초보 / K-12 |

→ 충돌하지 않고 **순차**: 튜토리얼로 1개 완성 → 졸업 시 반응형 가이드로 인계.

## 3. 한 단계의 원자 단위 (아주 단순의 정의)

모든 단계는 **딱 하나의 작은 행동**만 시킨다. 두 가지를 합치지 않고, 선택/판단을 요구하지 않는다.

| kind | 화면에 뜨는 것 | 완료 판정 (자동) | 막히면 (탈출구) |
|---|---|---|---|
| `copy` | "이 문장을 복사하세요" + [📋 복사] | 복사 클릭 즉시 | — (안 막힘) |
| `pasteSend` | 기획방 입력창만 밝게, 나머지 흐림 | `planResponded` | "안 됐어요" → 다시 복사 |
| `click` | 그 버튼 하나만 밝게 강조 | 해당 행동 신호(`changedFiles` 등) | "이 버튼이 안 보여요" → 건너뛰기 |
| `confirm` | "이렇게 나왔나요?" + [네, 됐어요] | 사용자 [네] | [잘 모르겠어요] → 그대로 진행 |

화면엔 **항상 한 번에 한 문장, 한 버튼.**

추가로, 각 단계 말풍선엔 시킬 행동 한 문장(`say`) 아래에 **(선택) "왜 이걸 하나요?" 한 줄(`why`)** 을 둔다. 행동·판단을 늘리지 않으면서 *VibeLign을 왜 그렇게 쓰는지*를 심는 장치다. 예:

> ③ **[체크포인트 저장]을 누르세요.**
> 〔왜?〕 *AI에게 시키기 전에 저장해두면, 결과가 마음에 안 들 때 한 번에 되돌릴 수 있어요.*

## 4. 끌고 가는 UI: 스포트라이트 오버레이

- 화면 전체를 살짝 어둡게 → **지금 눌러야 할 요소 하나만** 환하게(구멍 뚫린 마스크).
- 그 옆 🧭 마스코트 + 크림색 말풍선(`brutalism.css` 재활용): **한 문장 + 버튼 하나.**
- 자동 감지로 행동 완료 시 → 다음 단계로 "뽁" 이동.
- 말풍선 구석에 항상 작은 **"안 돼요 / 건너뛰기"** (탈출구).
- 좌상단 **"3 / 9 단계"** 진척 표시(끝이 보이게 → 완주 동기).

근거: 글로 "이 버튼을 누르세요"라 해도 초보자는 화면에서 못 찾는다. 나머지를 어둡게 해 **물리적으로 하나만 누를 수 있게** 하는 것이 "누구라도 따라하게"의 핵심.

## 5. 대본 데이터 포맷 (그릇 1개 + 데이터 3개)

```ts
type TutorialStep = {
  id: string;
  kind: 'copy' | 'pasteSend' | 'click' | 'confirm';
  say: string;            // 말풍선 한 문장 (시킬 행동, 항상 1개)
  why?: string;           // (선택) "왜 이걸 하나요?" 한 줄 — VibeLign 사용법을 심음. 행동/판단은 안 늘림.
  target?: string;        // 스포트라이트할 요소 (data-tour="...")
  copyText?: string;      // kind='copy'일 때 복사시킬 지시문
  done: 'changedFiles' | 'planResponded' | 'checkpoint' | 'runVerified' | 'manual';
  goPage?: StagePage;     // 이 단계 전에 데려다 놓을 화면 (stages.ts 재활용)
};

type Tutorial = {
  id: 'todo' | 'guestbook' | 'quiz';
  title: string;          // "나만의 할 일 목록 앱 만들기"
  emoji: string;
  goal: string;           // 완성하면 뭐가 되는지 한 줄
  steps: TutorialStep[];
};
```

- v1: `lib/tutorial/scripts.ts`에 3개를 코드로 정의(`guide.ts`와 같은 패턴). 나중에 JSON으로 빼면 엔진으로 진화.
- `done` 값은 전부 **기존 `guide.ts` 신호 재활용** → 자동 감지가 공짜.

### 제공 튜토리얼 3종 (서로 다른 앱 유형)
- **할 일 목록** — 데이터 추가/삭제/저장 (목록형)
- **방명록** — 폼 + 저장, 남이 내 페이지에 글 (웹페이지형)
- **퀴즈 게임** — 점수/정답 판정 로직 (게임형, K-12 반응 최상)

### 대본 필수 조건: VibeLign 안전 루프를 한 바퀴 돌게 한다
대본은 "기획→작업→실행"에만 쏠리면 안 된다. VibeLign의 차별점은 *안전*이므로, 모든 대본은 아래를 **의도적으로 포함**한다:
- **체크포인트 먼저 저장** — AI에게 시키기 *전에* 저장하는 습관을 심는다.
- **검사(guard) 확인** — AI 작업 후 안전 검사를 한 번 거친다.
- **되돌리기 한 번 체험** — 일부러 살짝 바꿨다가 체크포인트로 되돌려본다. "되돌릴 수 있다"는 안심을 *직접* 경험시킨다.
- **저장 또는 되돌리기로 마무리** — 루프의 닫힘을 보여준다.

→ 졸업 시 사용자는 단순히 앱을 만든 게 아니라 **VibeLign 안전 루프를 손으로 한 바퀴 돈** 상태가 된다.

### 출시 순서
포맷·렌더러는 3개를 지원하게 만들되, **대본은 "할 일 목록" 1개를 먼저 실제 AI로 끝까지 돌려 실증**한 뒤 나머지 둘을 같은 틀로 찍어낸다. 진짜 비용은 UI가 아니라 "대본을 실제 AI로 검증"하는 데 있다(AI 출력 비결정성).

## 6. 시작 지점

- **첫 실행**: `Onboarding.tsx` 마지막 → *"처음이신가요? 따라하면서 앱 하나를 같이 만들어볼까요?"* → 3장 카드 선택 → 시작.
- **언제든 다시**: 홈(`StageHubCards` 위)에 **"🧭 따라하며 만들기"** 버튼 → 메뉴 재진입.
- 진행률 기억: `useTutorial.ts`(= `useGuide.ts` localStorage 패턴 복제).

## 7. 완료 경험 (졸업)

졸업은 **두 박자**다 — 먼저 "완성 경험"을 충분히 누리게 한 뒤, 그제서야 "배운 것"을 되짚는다.

1. **완성 경험 (먼저, 충분히):** 마지막 단계는 반드시 사용자가 **방금 만든 앱을 실제로 실행해 손으로 만져보는** 것이다. 할 일 목록이면 직접 항목을 추가해 보고, 퀴즈면 직접 풀어 점수를 보고… **"내가 만든 게 진짜 돌아간다"** 를 몸으로 느끼게 한다. 🎉 **"축하합니다 — 당신이 첫 앱을 완성했어요!"**
2. **배운 것 되짚기 (그다음):** *"방금 이 순서로 만든 거예요 — 기획 → 저장 → AI 작업 → 검사 → 저장/되돌리기."* → **"이제 혼자 같은 순서로 무엇이든 만들 수 있어요."**
3. 튜토리얼 끄고 **반응형 가이드(`guide.ts`)로 졸업 인계.**
- (선택) "다음엔 방명록도 만들어볼래요?" → 다른 튜토리얼 권유.

## 8. 기존 코드 재활용 지도

| 튜토리얼이 쓰는 것 | 기존 자산 | 처리 |
|---|---|---|
| 단계 완료 자동 감지 | `guide.ts` 신호(changedFiles·checkpoint·runVerified·planResponded) | 그대로 재활용 |
| 지시문 복사 | `GuideStrip` "지시문 복사" 로직 | 추출/재활용 |
| 진행률 영속 | `useGuide.ts` 패턴 | 복제 → `useTutorial.ts` |
| 말풍선/마스코트 | `brutalism.css` 크림 말풍선 + 🧭 | 재활용 |
| 화면 이동(goPage) | `stages.ts` 페이지 매핑 | 재활용 |
| 시작 지점 | `Onboarding.tsx` | 끝에 블록 추가 |

### 새로 만드는 파일
```
lib/tutorial/scripts.ts                ← 대본 3개 (데이터)
lib/tutorial/useTutorial.ts            ← 진행 상태 훅
components/tutorial/SpotlightTour.tsx  ← 스포트라이트 오버레이
components/tutorial/TutorialPicker.tsx ← 3장 선택 카드
```

### 유일하게 침습적인 변경
- 스포트라이트가 가리킬 버튼 ~8개에 `data-tour="..."` 속성 추가. (앵커 경계 안, 최소 패치)

## 9. 위험과 완화

| 위험 | 완화 |
|---|---|
| AI 출력 비결정성으로 "다음 버튼이 안 맞음" | 대본은 *정확한 코드*가 아니라 "앱이 만들어졌다" 수준 신호에만 의존(`changedFiles` 등). 단계마다 탈출구. |
| `data-tour` 대상 버튼이 리팩터링으로 사라짐 | 속성 누락 시 스포트라이트가 못 찾으면 자동으로 "건너뛰기"로 폴백(절대 못 막히게). |
| 첫 완성 경험을 증명 못 함(빈 그릇) | 출시 순서대로 "할 일 목록" 1개를 실제 AI로 완주 검증 후 확장. |
| 튜토리얼·반응형 가이드 동시 표시로 혼란 | 튜토리얼 활성 중엔 `GuideStrip` 숨김. 졸업 시 스위치. |

## 10. 검증 (구현 후)

- "할 일 목록" 대본을 **처음 보는 사람이 막힘 없이 완주**해 작동 앱이 폴더에 남는지 (육안 + 스크린 녹화).
- 각 단계가 원자 단위(행동 1개) 원칙을 지키는지 리뷰.
- 모든 단계에서 탈출구가 동작해 **절대 못 막히는지** 확인.
