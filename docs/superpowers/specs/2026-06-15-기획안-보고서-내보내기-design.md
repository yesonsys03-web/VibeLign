# 기획안 → 보고서 내보내기 (Report Export) — 설계 문서

- 작성일: 2026-06-15
- 상태: 설계 승인됨 (구현 계획 대기)
- 대상 영역: 기획방(Planning Room) / 기획안 탭(PlanDocView)

## 1. 배경 · 목적

현재 기획방은 "앱·프로그램 기획"용으로 설계되어 있고, 최종 산출물은
구조화된 마크다운(`plans/{slug}.md`)이다. 이 마크다운은 AI/개발자가 읽을
**구현 명세**에 가깝다(페르소나 대화 재현, 작업 계약 등).

직장인은 매일 보고서를 쓴다. 기획방에서 정리한 내용을 **업무에서 바로
활용 가능한 보고서**(PDF·Word·PPT)로 내보낼 수 있다면, VibeLign의 사용층이
"앱 만드는 사람" → "기획·문서 쓰는 모든 직장인"으로 넓어진다.

핵심 통찰: 기획안을 그대로 PDF로 떨구는 것은 "보고서"가 아니라 "PDF로 바뀐
기획안"일 뿐이다. **진짜 가치는 변환 지능** — 기획 내용을 보고서 독자
(상사·임원·고객)에 맞는 구조·어조로 재구성하는 데 있다.

이 변환 지능의 **1차 담당은 AI가 아니라 템플릿**이다. 종류별 섹션 재매핑·어조
골격이 결정적으로 구조를 만들고(§5·§6), AI 다듬기는 그 위에 얹는 **가산
레이어**일 뿐이다(기본 OFF). 즉 기본 동작만으로도 단순 "PDF로 바뀐 기획안"을
넘어선다 — AI는 품질을 더할 뿐, "보고서다움"의 전제 조건이 아니다.

## 2. 목표 / 비목표

### 목표
- 저장된 기획안 하나를 선택해 **보고서 종류 × 출력 포맷**으로 내보낸다.
- 보고서 종류: **업무보고 / 제안서 / 결과보고** (템플릿 기반, 추가 용이).
- 출력 포맷: **PDF · Word(.docx) · PPT(.pptx)** 동등 지원.
- 변환은 **하이브리드**: 템플릿이 구조를 결정(무료·결정적), AI는 요약·어조만
  보조(opt-in, 무료 provider, 캐시, 실패 시 fallback).
- 비용 중립: 기본 동작에서 유료 AI(Claude 등) 자동 호출 없음.

### 비목표 (YAGNI)
- 보고서 WYSIWYG 편집기 (생성 후 편집은 Word 파일을 외부 앱에서).
- 사내 양식 100% 픽셀 재현 / 브랜드 테마 커스터마이즈 (후속).
- satgat 코드/런타임 통합 (디자인 참고만).
- 다국어 보고서 (한국어 우선).

## 3. 아키텍처 — 3단 파이프라인

```
저장된 기획안(plans/{slug}.md) + 메타(.vibelign/planning/{id}/)
        │
   [1 추출 Extract]  .md 섹션 파싱 → 정규화된 PlanningData
        │
   [2 변환 Transform]  보고서 종류 템플릿 + (opt-in) AI 다듬기 → ReportModel(IR)
        │
   [3 렌더 Render]  ReportModel → PDF · Word · PPT  (렌더러만 교체)
```

포맷에 독립적인 **IR(ReportModel)** 이 설계의 중심이다. 변환 지능은 IR까지만
책임지고, 포맷 3종은 IR을 받아 그리기만 한다. 이 분리 덕에 종류/포맷 확장이
서로 간섭하지 않는다.

### 3.1 단위 책임

| 단위 | 책임 | 입력 | 출력 | 의존 |
|---|---|---|---|---|
| Extractor | 기획 .md + 메타 파싱 | 파일 경로 | `PlanningData` | 파일시스템 |
| Templates | 종류별 섹션 매핑·어조 규칙 | `PlanningData`, 종류 | 섹션 골격 | (순수) |
| AI Polish | 요약·어조 다듬기 (opt-in) | 섹션 텍스트 | 다듬은 텍스트 | 무료 provider, 캐시 |
| Assembler | 골격 + 다듬은 텍스트 → IR | 위 결과 | `ReportModel` | (순수) |
| Renderers | IR → 파일 | `ReportModel` | PDF/docx/pptx | sidecar / 웹뷰 |

각 단위는 잘 정의된 인터페이스로만 통신하고 독립 테스트 가능해야 한다.

## 4. 데이터 모델

### 4.1 PlanningData (추출 결과, 포맷·종류 무관)
기획 .md의 섹션을 정규화한다. 예시 필드:

- `title`: 제목
- `idea`: 한 줄 목표
- `target_users`: 대상 사용자
- `problem`: 핵심 문제
- `features[]`: 핵심 기능
- `flows[]`: 사용자 흐름
- `decisions[]`: 확정된 결정 (소스는 `work_memory.json`의 decisions — cards.json 아님, 구현 시 소스 재확인)
- `exclusions[]`: 제외할 것
- `open_questions[]`: 아직 결정이 필요한 질문
- `readiness`: 준비 상태 지표 (있으면)
- `context_notes`: 구현 전 맥락

비어 있는 섹션("아직 결정이 필요합니다")은 누락으로 처리한다.

> **추출 범위와 증분**: 위 필드는 **최종 목표**다. CLI 계획 1/4의 reader는 우선
> `.md` 헤딩 subset(title·idea·problem·target_users·features·flows·exclusions·
> open_questions)만 파싱한다. `decisions`·`readiness`·`context_notes` 등 **메타
> 추출은 후속 증분**이며, 마크다운 재파싱은 본질적으로 취약하므로 그때
> `.vibelign/planning/{id}/` 구조 소스를 **우선** 읽고 `.md`는 fallback으로 둔다.

### 4.2 ReportModel (IR, 포맷 독립)
```
ReportModel {
  meta: { title, report_type, author?, date, source_plan_path }
  sections: Section[]
}
Section {
  heading: string
  blocks: Block[]
}
Block =
  | { kind: "paragraph", text }
  | { kind: "bullets", items: string[] }
  | { kind: "table", headers: string[], rows: string[][] }
  | { kind: "keyvalue", pairs: {k, v}[] }
  | { kind: "slide_break" }          // PPT 슬라이드 경계 힌트
  | { kind: "summary", text }        // 임원용 요약 (강조 렌더)
```

`slide_break`/`summary` 같은 의미 블록을 두어, 렌더러가 포맷별로 다르게
표현할 수 있게 한다 (PPT는 slide_break마다 슬라이드 분할, PDF/Word는 무시 또는
구분선).

`source_plan_path`는 CLI 증분 1부터 유지한다. 당장 HTML 렌더에 노출하지 않더라도
후속 GUI 재생성, 캐시 키, "어떤 기획안에서 생성됐는지" 추적에 필요하므로 IR에서
잃지 않는다.

## 5. 보고서 종류 = 템플릿

각 템플릿은 "기획 섹션 → 보고서 섹션 매핑 규칙 + 어조 지시" 한 장으로 정의한다.
종류 추가는 템플릿 파일 하나 추가로 끝난다.

초기 3종(예시 골격, 구현 시 확정):

- **업무보고**: 개요 → 진행 현황 → 주요 결정 → 다음 단계 → 이슈/리스크.
  어조: 간결·사실 중심, 상사 대상.
- **제안서**: 배경/문제 → 제안 개요 → 핵심 기능/가치 → 기대 효과 → 실행 계획.
  어조: 설득적, 의사결정자 대상.
- **결과보고**: 목표 → 수행 내용 → 결과/산출물 → 회고 → 향후 과제.
  어조: 정리·공유 중심.

## 6. 하이브리드 변환

- **구조**: 템플릿이 결정 — 무료, 결정적, 즉시. AI 없이도 완전한 보고서가 나온다.
- **AI 패스 (opt-in)**: 무료 provider 체인(codex → opencode → agy; claude는
  opt-in)으로 ① 임원용 한 줄/문단 요약 생성, ② "개발 용어 → 비즈니스 어조"
  다듬기만 담당.
- **캐시**: 결과를 report 캐시에 저장(기획방 `enrich.json` 패턴 재사용 —
  messages/내용 해시 기반). export 반복해도 재호출하지 않는다.
- **Fallback**: AI 실패·타임아웃 시 템플릿 원문으로 자동 대체(꺼진 페르소나
  안내 패턴과 동일한 graceful degradation).

비용 원칙: AI 다듬기 기본 OFF. 켜도 무료 provider만 쓰므로 Claude 과금 노출 없음.
(참조: v2.4.3 "Claude 자동 호출 최소화" 방침과 정합)

## 7. 렌더러

| 포맷 | 구현 위치 | 방식 | 비고 |
|---|---|---|---|
| **PDF** | HTML=Python core, 인쇄=프론트 웹뷰 | IR → HTML(디자인 CSS, `html_renderer`) → 웹뷰 print-to-PDF | 디자인 충실도 높음. **생성 방식(a/b/c)은 §13.1에서 결정** |
| **Word(.docx)** | Python sidecar | IR → python-docx | 편집 가능 |
| **PPT(.pptx)** | Python sidecar | IR → python-pptx, slide_break 단위 분할 | 편집 가능 |

- **HTML 단일 소스**: IR → HTML 렌더는 Python core(`html_renderer`)가 **단일
  소스**다. 프론트는 그 HTML을 미리보기(iframe/웹뷰)와 인쇄(print-to-PDF)에만
  쓴다 — 디자인 CSS가 프론트/백 두 곳으로 갈라지지 않게 한다.
- **HTML 디자인**: satgat을 **디자인 레퍼런스**로 삼아 한글 폰트(OFL, 재배포 OK)·
  CSS 토큰·미감을 참고한다. satgat 코드/런타임은 통합하지 않는다(Next.js 서버·
  Google AI 의존 회피).
- **폰트 임베드(증분 연기)**: 결정적 Win/Mac 출력을 위한 OFL 한글 폰트 `@font-face`
  **번들 임베드**는 **PDF/GUI 증분으로 연기**한다. CLI 증분 1의 HTML은 시스템
  폰트(Mac=Apple SD Gothic·Win=시스템 serif) 폴백을 그대로 쓴다 — 이 단계 산출물은
  브라우저 미리보기·내부 확인용이고, OS별 줄바꿈·페이지네이션 차이는 임베드 도입
  시 해소한다. (Word/PPT 폰트 한계는 §13.2.)
- **sidecar 의존성**: python-docx, python-pptx를 vib-runtime 번들에 추가
  (PDF용 무거운 네이티브 PDF 라이브러리는 도입하지 않음 — 웹뷰 인쇄로 대체).

## 8. UI / UX

위치: **기획안 탭(`PlanDocView`)**. 이미 저장된 기획안을 나열하고 `.md`를
읽으므로 가장 자연스러운 자리다.

흐름:
```
기획안 선택 → [📄 보고서로 내보내기] 버튼 → 모달
   ├ 보고서 종류 선택 (업무 / 제안 / 결과)
   ├ 출력 포맷 선택 (PDF / Word / PPT)
   ├ AI 다듬기 토글 (기본 OFF, "무료 provider" 명시)
   ├ 미리보기 (Python `html_renderer`가 만든 HTML을 그대로 표시)
   └ 내보내기 → .vibelign/reports/ (또는 사용자 선택 경로)에 저장 → 파일 열기
```

- `doc_stale`(저장 후 대화가 더 진행된) 기획안은 "최신 아님" 경고 후 진행 허용.
- 생성 결과 파일 경로 표시 + "파일 열기"(opener 플러그인 `openPath`; 실패 시
  `revealItemInDir`로 폴더 표시 폴백 — Win/Mac에 연결 앱이 없을 때 대비).
- **PDF가 (b) `window.print()` 경로면** 사용자가 OS 저장 다이얼로그에서 경로를
  직접 고르므로 앱이 결과 경로를 모른다 → 그 경우 "경로 표시/파일 열기"를 생략하고
  안내만 한다. (a) 무인 생성 경로면 `.vibelign/reports/` 고정 경로를 쓴다(§13.1).

## 9. 에러 처리

- **AI 실패** → 템플릿 원문 fallback + 안내 메시지(조용한 실패 금지).
- **렌더 실패 격리** → 포맷별 독립. PDF 실패해도 Word 생성은 진행 가능.
- **빈 기획안**(섹션 대부분 미정) → 경고하고, 채워진 섹션만으로 생성.
- **sidecar 미가용** → `html_renderer`도 Python core(sidecar)에 있으므로 **PDF
  포함 모든 내보내기 비활성**("런타임 필요" 안내). PDF만 따로 살리려면 HTML 렌더를
  프론트에 fallback으로 두어야 하나, 단일 소스 원칙(§7)과 트레이드오프 → 기본은
  전체 비활성으로 둔다.
- **연결 앱 없음**(Win에 Word/PowerPoint 미설치 등) → "파일 열기" 실패를 삼키고
  `revealItemInDir`로 폴더만 표시.
- **인코딩** → 파일 쓰기는 항상 `encoding="utf-8"` 명시. Windows 기본 cp949로
  쓰면 한글이 깨진다(조용한 실패 금지의 일부, §13.5).

## 10. 테스트 전략

- **추출**: 기획 .md 샘플 → `PlanningData` 단위 테스트(섹션 누락·비정상 입력 포함).
- **변환**: `PlanningData` + 종류 → `ReportModel` 스냅샷 테스트(템플릿별).
- **AI 패스**: provider mock — 성공/타임아웃/실패 → fallback 경로 검증.
- **렌더러**: 포맷별 골든/스모크 — 파일이 생성되고 열리는지(docx/pptx 유효성,
  PDF HTML 구조). 픽셀 비교는 비목표.
- **캐시**: 동일 입력 재요청 시 provider 재호출 없음 검증.

## 11. 점진적 구현 순서 (제안)

1. **IR + Extractor + Assembler** (포맷·AI 없이 기획 .md → ReportModel).
   추출은 `.md` subset부터; 메타(decisions/readiness/context_notes) 추출은 이후 증분(§4.1).
2. **PDF 레그**: IR → HTML(디자인 CSS, satgat 참고) → 웹뷰 인쇄. UI 모달 최소판.
   ⚠ **선행 스파이크 + 정책 결정**: Tauri v2는 무인 print-to-PDF 코어 API가
   없다(§13.1). 진입 전 a/b/c 중 어느 PDF 생성 경로를 쓸지 확정하고, 고른 방식이
   Win/Mac에서 실제로 되는지 스파이크로 검증한다.
3. **변환 템플릿 3종** + 미리보기.
4. **Word 렌더러** (python-docx, sidecar).
5. **PPT 렌더러** (python-pptx, slide_break).
6. **AI 다듬기 패스** (무료 provider + 캐시 + fallback), 토글.

각 단계가 독립적으로 사용자에게 가치를 준다(1+2만으로도 "PDF 보고서" 동작).

## 12. 미해결 / 후속

- 종류별 템플릿의 정확한 섹션 구성·어조 문구는 구현 시 사용자 검토로 확정.
- 사내 표준 양식 매핑(있다면)은 후속 반복.
- PPT 자동 슬라이드 분할 품질은 휴리스틱부터 시작, 사용 피드백으로 개선.

## 13. 크로스플랫폼 / 엣지케이스 (Win · Mac)

스택 전제: **Tauri v2**(plugins: opener, dialog — print 플러그인 없음) + Python
`vib` sidecar(`std::process` spawn) + `vib-runtime` 번들.

### 13.1 PDF 생성 경로 — 먼저 결정 (★)
Tauri v2 코어엔 무인 "save as PDF" API가 없다. OS 웹뷰 계층엔 다 있으나 Tauri가
노출하지 않는다. 셋 중 택1:

| 경로 | macOS(WKWebView) | Windows(WebView2) | "의존성 0"? | 비고 |
|---|---|---|---|---|
| (a) 무인 print-to-PDF | `createPDF`(objc2) | `PrintToPdfAsync`(webview2-com) | ❌ 플랫폼별 Rust | 경로 자동 = `.vibelign/reports/` 고정 |
| (b) `window.print()` | 네이티브 인쇄 패널 | 인쇄 다이얼로그 | ✅ | 무인 아님, 결과 경로 모름(§8) |
| (c) Python PDF 라이브러리 | weasyprint/reportlab | 동일 | ❌ 무거운 의존성 | 설계상 비선호 |

권장: MVP는 (b)로 "의존성 0"을 유지하되 §8의 경로/열기 한계를 수용하고,
무인·고정경로가 필요해지면 (a)로 승급. **구현 1차 진입 전 확정**(§11-2 스파이크).

### 13.2 폰트
- **PDF**: §7대로 OFL 한글 폰트 `@font-face` 번들 임베드 → Win/Mac 동일 출력.
  단 **임베드는 PDF/GUI 증분으로 연기**하고, CLI 증분 1 HTML은 시스템 폰트 폴백을
  쓴다(증분 1 산출물은 내부 확인용).
- **Word/PPT**: python-docx/pptx는 **폰트를 임베드하지 않고 이름만 참조**한다.
  "맑은 고딕"은 Mac에 없고 "Apple SD Gothic"은 Win에 없다 → 양쪽에서 깨지지 않는
  폰트명 + graceful 대체를 택한다(근본 임베드 불가, "덜 깨지는 선택").

### 13.3 sidecar 새 의존성 — 네이티브 휠 · 서명
- python-docx → **lxml**(C 확장), python-pptx → **lxml + Pillow**(C 확장). 즉
  vib-runtime에 **플랫폼별 바이너리 휠**(win_amd64 / macOS arm64+x86_64)이 들어간다.
  Mac 듀얼아치 빌드에서 arm64 휠 누락 시 Rosetta 폴백 주의.
- macOS codesign/ad-hoc 서명이 `.dylib/.so`를 사후 변경 → `.sha256` manifest
  어긋남(RUST_ENGINE_INTEGRITY_FAILED). **Word/PPT 증분에서 integrity manifest
  재생성**을 단계에 포함한다.

### 13.4 파일명 · 경로 (Windows 기준 sanitize)
- 금지문자 `\ / : * ? " < > |` 제거(제목→파일명). Mac은 `:`만, **Win은 전부**.
- 예약어 CON/PRN/AUX/NUL/COM1… 회피.
- **MAX_PATH 260**: 프로젝트경로 + `.vibelign/reports/` + 긴 한글 slug + 확장자가
  초과 가능 → 파일명 길이 제한(또는 `\\?\` 프리픽스).
- 대소문자: Mac(APFS 기본)·Win은 case-insensitive → `Report`/`report` 충돌 주의.

- 모든 파일 쓰기에 `encoding="utf-8"` 명시(Windows 기본 cp949 회피). sidecar
  stdout 캡처도 PYTHONUTF8 정합 확인.
