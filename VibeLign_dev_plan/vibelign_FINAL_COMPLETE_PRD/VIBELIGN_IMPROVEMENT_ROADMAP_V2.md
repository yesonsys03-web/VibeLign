# VibeLign 2차 개선 로드맵: 사용자 경험 강화

> 1차 로드맵(엔진 내부 강화)이 완료된 후의 다음 단계입니다.
> 1차가 "도구를 잘 작동하게 만들기"였다면, 2차는 "코알못이 실제로 잘 쓸 수 있게 만들기"입니다.

---

## 🔴 1순위: 코알못 온보딩 가이드

**현재 문제:**
`vib_start_cmd.py`는 초기 설정(doctor + hook 설정)을 자동으로 해주지만, 그 다음 뭘 해야 하는지 코알못이 모른다. `다음 할 일: vib anchor --suggest`라고 출력되지만, 코알못은 이게 무슨 뜻인지, 왜 해야 하는지, 그 다음엔 뭘 해야 하는지 모른다.

**왜 중요한가:**
도구가 아무리 좋아도 처음 3분 안에 "아, 이렇게 쓰면 되는구나"를 이해 못 하면 코알못은 그냥 닫아버린다. start 이후의 자연스러운 흐름이 코알못에게 손잡고 안내되어야 한다.

**해결 방안:**

- [ ] `start` 완료 후 단계별 온보딩 메시지 추가
  - 현재: 점수와 다음 할 일만 출력
  - 개선: 코알못이 이해할 수 있는 전체 워크플로우를 단계별로 안내
  ```
  ✅ VibeLign 시작 설정 완료!
  
  📋 이제 이렇게 진행하면 돼요:
  
  1단계: 안전 구역 만들기 (1번만 하면 돼요)
     → vib anchor
     "코드에 표식을 달아서 AI가 정확한 위치를 찾을 수 있게 해요"
  
  2단계: AI에게 코드 수정 시키기
     → vib patch "원하는 변경사항"
     "AI가 이해하기 쉬운 형태로 바꿔줘요. 복사해서 AI에게 붙여넣으세요"
  
  3단계: 수정 결과 확인하기
     → vib guard
     "AI가 이상한 곳을 건드리지 않았는지 자동으로 검사해요"
  
  💡 언제든 vib doctor 로 프로젝트 상태를 확인할 수 있어요
  💡 vib checkpoint "설명" 으로 현재 상태를 저장할 수 있어요 (게임 세이브 같은 거예요)
  ```

- [ ] `--quickstart` 옵션 추가 검토
  - `vib start --quickstart` → start + anchor를 한 번에 실행하여 코알못이 바로 patch를 사용할 수 있는 상태로 만들기
  - 온보딩 단계를 최소화

- [ ] 첫 실행 시 `VIBELIGN_QUICKSTART.md` 가이드 파일 자동 생성
  - 프로젝트 루트에 코알못이 읽을 수 있는 한국어 가이드 파일 생성
  - 자주 쓰는 명령어 5개와 사용 시나리오를 포함
  - 예:
    ```markdown
    # VibeLign 빠른 시작 가이드
    
    ## AI에게 코드 수정 시키고 싶을 때
    vib patch "로그인 버튼 크기 키워줘"
    → 나온 결과를 복사해서 AI(ChatGPT, Claude 등)에 붙여넣기
    
    ## 뭔가 잘못됐을 때
    vib undo
    → 마지막 세이브 포인트로 되돌리기
    
    ## 지금 상태가 궁금할 때
    vib doctor
    → 프로젝트 건강 상태 확인
    ```

- [ ] `start` 실행 시 프로젝트 언어/프레임워크에 맞는 맞춤 안내 제공
  - Python 프로젝트면 Python 예시, JS 프로젝트면 JS 예시로 안내
  - `project_scan.py`의 파일 분석 결과를 활용

## 🔴 2순위: AI에게 코드맵 활용법을 알려주는 프롬프트 자동생성

**현재 문제:**
코드맵과 앵커가 잘 만들어져도, 코알못이 AI에게 수정을 시킬 때 "먼저 코드맵을 읽어"라고 지시하지 않으면 AI가 코드맵을 무시한다. 코알못은 이런 프롬프트 기법을 모른다.

**왜 중요한가:**
앵커와 코드맵의 가치는 AI가 실제로 읽고 활용할 때만 발생한다. 코알못이 복잡한 프롬프트를 작성할 필요 없이, 도구가 자동으로 AI에게 "코드맵 먼저 읽고 → 앵커 확인하고 → 수정해"라는 지시를 만들어줘야 한다.

**해결 방안:**

- [ ] `patch` 커맨드의 handoff 프롬프트에 코드맵 활용 지시 자동 포함
  - 현재 `_build_ready_handoff()`가 생성하는 프롬프트에 코드맵 참조 단계 추가
  - 예:
    ```
    Before making any changes:
    1. Read .vibelign/project_map.json to understand the project structure
    2. Check the anchor list for the target file
    3. Only modify code within the specified anchor boundaries
    4. Check @CONNECTS to avoid breaking related features
    ```

- [ ] 코드맵 기반 "AI 브리핑 파일" 자동 생성 커맨드 추가
  - `vib brief` 또는 `vib context` 커맨드
  - 코드맵 + 앵커 요약 + 프로젝트 현황을 하나의 마크다운으로 출력
  - 코알못은 이 파일을 통째로 AI에게 붙여넣기만 하면 됨
  - 출력 예:
    ```markdown
    # 프로젝트 브리핑 (AI용)
    
    ## 프로젝트 구조
    이 프로젝트는 Python 기반이며, 총 12개 소스 파일로 구성되어 있습니다.
    
    ## 파일별 역할과 앵커
    - pages/login.py [UI] 
      앵커: LOGIN_FORM (이메일/비밀번호 입력 폼), LOGIN_HANDLER (로그인 처리)
    - api/auth.py [Service]
      앵커: AUTH_LOGIN (로그인 API), AUTH_TOKEN (토큰 관리)
      ⚠️ 수정 시 LOGIN_FORM에 영향
    
    ## 규칙
    - 허용된 파일만 수정하세요
    - ANCHOR 마커를 삭제하지 마세요
    - 파일 전체를 다시 쓰지 마세요
    ```

- [ ] `export` 커맨드의 AI 룰 파일에 코드맵 활용 지시 추가
  - `AI_DEV_SYSTEM_SINGLE_FILE.md`에 코드맵 읽기 필수 규칙 포함
  - `AGENTS.md`에 코드맵 참조 워크플로우 추가
  - `.cursorrules`에 코드맵 우선 참조 규칙 포함

- [ ] `watch` 모드에서 코드맵 갱신 시 AI 브리핑 파일도 함께 갱신
  - 코드맵이 바뀔 때마다 브리핑 파일도 최신 상태 유지
  - AI가 항상 최신 정보를 참조할 수 있도록 보장

## 🟡 3순위: AI 간 코드 이동 시 앵커 효과 검증 테스트

**현재 문제:**
앵커와 코드맵이 실제로 AI의 수정 정확도를 높이는지 정량적 증거가 없다. "Codex에서 도움이 된다고 했다"는 정성적 피드백은 있지만, 체계적인 비교 테스트가 없다.

**왜 중요한가:**
이 도구의 핵심 가치 제안이 "앵커가 있으면 AI가 더 정확하게 수정한다"인데, 이 주장을 뒷받침할 데이터가 있어야 한다. 또한 테스트를 통해 앵커 형식이나 코드맵 구조의 실제 약점을 발견할 수 있다.

**해결 방안:**

- [ ] 표준 테스트 시나리오 세트 설계
  - 소규모 테스트 프로젝트(파일 5~10개) 준비
  - 동일한 수정 요청 5가지 정의
    - 예: "로그인 에러 메시지 변경", "버튼 크기 키우기", "새 필드 추가" 등
  - 각 요청에 대한 "정답" 정의 (어떤 파일, 어떤 위치가 수정되어야 하는지)

- [ ] 비교 테스트 실행 절차 문서화
  - A조건: 앵커/코드맵 없이 AI에게 수정 요청
  - B조건: 앵커/코드맵 + 브리핑 파일 포함하여 AI에게 수정 요청
  - 측정 기준:
    - 정확한 파일을 수정했는가? (파일 정확도)
    - 정확한 위치를 수정했는가? (위치 정확도)
    - 다른 기능이 망가지지 않았는가? (안전도)
    - 사용한 토큰 수 (효율성)

- [ ] 최소 3개 AI(GPT, Claude, Gemini)에서 동일 테스트 실행
  - AI 중립성 검증: 앵커가 특정 AI에서만 효과 있는 게 아닌지 확인
  - 각 AI별 강점/약점 파악

- [ ] 테스트 결과를 기록하는 `vib bench` 커맨드 또는 스크립트 검토
  - 수정 전후 diff를 자동 비교
  - 앵커 경계 이탈 여부 자동 판단
  - 결과를 JSON/마크다운으로 출력

- [ ] 테스트 결과를 README/문서에 반영
  - "앵커 있을 때 vs 없을 때" 비교 표
  - 이 데이터가 도구의 가장 강력한 마케팅 자료가 됨

## 🟡 4순위: GUI 준비 (코알못 접근성)

**현재 문제:**
CLI는 코알못에게 공포이다. 까만 화면에 글자만 나오고, 명령어를 외워야 하고, 오타나면 에러가 나온다. PyInstaller로 패키징은 되어 있지만, 실행 자체가 터미널 기반이라 진입장벽이 높다.

**왜 중요한가:**
아무리 좋은 기능도 코알못이 실행을 못 하면 의미가 없다. 버튼 2~3개짜리 간단한 GUI만 있어도 코알못의 접근성이 극적으로 올라간다.

**해결 방안:**

- [ ] GUI 설계 먼저 (코드 전에 화면 구성 결정)
  - 최소 GUI 화면 구성:
    ```
    ┌─────────────────────────────────────┐
    │  VibeLign                     [×]   │
    ├─────────────────────────────────────┤
    │                                     │
    │  📁 프로젝트 폴더: [선택...]        │
    │                                     │
    │  ┌───────────┐  ┌───────────┐       │
    │  │ 🔍 진단   │  │ 🔖 앵커   │       │
    │  └───────────┘  └───────────┘       │
    │  ┌───────────┐  ┌───────────┐       │
    │  │ 💾 세이브  │  │ ↩️ 되돌리기│       │
    │  └───────────┘  └───────────┘       │
    │  ┌───────────────────────────┐      │
    │  │ 👁 자동 감시 시작/중지     │      │
    │  └───────────────────────────┘      │
    │                                     │
    │  상태: ✅ 프로젝트 정상 (85/100)    │
    │  마지막 갱신: 14:32:05              │
    │                                     │
    │  ┌─────────────────────────────┐    │
    │  │ AI에게 시킬 내용 입력:       │    │
    │  │ [                           ]│    │
    │  │ [▶ 패치 준비]               │    │
    │  └─────────────────────────────┘    │
    │                                     │
    │  📋 결과 (복사해서 AI에 붙여넣기):  │
    │  ┌─────────────────────────────┐    │
    │  │ (패치 프롬프트 출력 영역)    │    │
    │  │                             │    │
    │  │ [📋 복사하기]               │    │
    │  └─────────────────────────────┘    │
    └─────────────────────────────────────┘
    ```

- [ ] GUI 기술 스택 결정
  - 옵션 1: `tkinter` (Python 내장, 추가 설치 없음, 외형 제한적)
  - 옵션 2: `customtkinter` (tkinter 기반 + 모던 UI, pip 설치 필요)
  - 옵션 3: 웹 기반 (`Flask` + 브라우저, 가장 예쁘지만 구조 복잡)
  - 권장: `customtkinter` → PyInstaller 호환 + 모던 외형 + 단일 파일 가능

- [ ] CLI 코어와 GUI의 분리 구조 확인
  - 현재 `core/` 모듈이 CLI와 분리되어 있으므로 GUI에서도 재사용 가능
  - GUI는 `core/`의 함수를 직접 호출하는 래퍼(wrapper) 역할만 해야 함
  - `terminal_render.py`의 출력 로직은 GUI에서 사용하지 않도록 분리 필요

- [ ] GUI 전용 진입점 추가
  - `vibelign-gui` 또는 `vib --gui` 옵션으로 GUI 실행
  - PyInstaller 빌드 시 GUI 버전 별도 패키징

- [ ] GUI 최소 기능 목록 (MVP)
  - 프로젝트 폴더 선택 (파일 브라우저)
  - 진단 버튼 (doctor 실행 → 결과 표시)
  - 앵커 버튼 (anchor 실행 → 결과 표시)
  - 세이브/되돌리기 버튼 (checkpoint/undo)
  - 패치 입력 → 결과 복사 기능
  - 상태 표시 (코드맵 최신 여부, 점수)

## 🟢 5순위: patch 출력의 AI 전달 최적화

**현재 문제:**
`vib_patch_cmd.py`의 출력이 상세하지만, 코알못이 "이 중에 뭘 AI에게 붙여넣어야 하는지" 헷갈릴 수 있다. `handoff.prompt` 블록이 있지만, 주변에 다른 정보가 많아 어디를 복사해야 하는지 불분명하다.

**왜 중요한가:**
patch의 최종 목적은 "AI에게 붙여넣을 텍스트를 만드는 것"이다. 코알못에게는 이 최종 결과물이 명확하게 구분되어야 한다.

**해결 방안:**

- [ ] 패치 출력을 "코알못용 안내"와 "AI에게 전달할 블록"으로 명확히 분리
  - 코알못용: "이 요청을 이렇게 이해했어요", "이 부분을 복사해서 AI에 붙여넣으세요"
  - AI용: 코드맵 참조 + 앵커 위치 + 수정 규칙이 포함된 구조화된 프롬프트

- [ ] `--copy` 옵션 추가 검토
  - `vib patch "수정 내용" --copy` → AI 전달용 프롬프트를 클립보드에 자동 복사
  - 코알못은 바로 AI 채팅창에 붙여넣기만 하면 됨
  - `pyperclip` 라이브러리 사용 (크로스 플랫폼)

- [ ] handoff 프롬프트에 코드맵 요약 자동 포함
  - 현재: 파일명과 앵커명만 포함
  - 개선: 해당 파일의 앵커 목록, 연결 관계, 의도 정보까지 포함
  - AI가 이 프롬프트 하나만 읽어도 수정에 필요한 모든 맥락을 가지도록

## 🟢 6순위: doctor 진단 메시지 코알못 최적화

**현재 문제:**
`doctor_v2.py`의 진단 메시지가 개발자 관점이다. "앵커가 없습니다", "파일이 너무 큽니다" 같은 메시지를 코알못은 왜 중요한지, 뭘 해야 하는지 바로 이해하기 어렵다.

**왜 중요한가:**
doctor는 코알못이 "내 프로젝트가 괜찮은지"를 확인하는 첫 번째 수단이다. 이 메시지가 이해 안 되면 doctor 자체를 안 쓰게 된다.

**해결 방안:**

- [ ] 진단 메시지에 "비유" 또는 "비개발자 설명" 추가
  - 현재: "login.py에 앵커가 없습니다"
  - 개선: "login.py에 표식이 없어요. 표식이 없으면 AI가 이 파일의 어디를 고쳐야 하는지 헤매게 돼요. `vib anchor`를 실행하면 자동으로 표식을 달아줘요."

- [ ] 점수에 대한 직관적 설명 추가
  - 현재: "점수: 72 / 100, 상태: Good"
  - 개선: "점수: 72 / 100 (B학점 정도예요) — 크게 걱정할 건 없지만, 작은 정리를 하면 AI가 더 안전하게 수정할 수 있어요"

- [ ] `doctor` 실행 후 자동 해결 제안 강화
  - 문제마다 "이걸 해결하려면 이 명령어 하나만 실행하세요" 형태로 구체적 행동 제시
  - 가능하면 `--fix` 옵션으로 자동 해결 (예: 앵커 없는 파일에 자동으로 앵커 추가)

## 🟢 7순위: explain의 AI 연동 안정성 강화

**현재 문제:**
`ai_explain.py`의 5개 AI provider 폴백 체인이 잘 설계되어 있지만, API 키가 없는 코알못은 explain의 AI 기능을 사용할 수 없다. Gemini 무료 API가 있어도 설정 방법을 모를 수 있다.

**왜 중요한가:**
explain은 코알못이 자기 프로젝트에서 무슨 일이 일어나는지 이해하는 핵심 기능이다. AI 없이 rule 기반만으로는 "중학생 레벨 설명"이 불가능하다.

**해결 방안:**

- [ ] `config` 커맨드에서 Gemini 무료 API 키 설정을 더 쉽게 안내
  - "Google AI Studio에서 무료 API 키를 받을 수 있어요" + URL 직접 제공
  - 키 입력 후 즉시 테스트 호출로 작동 확인

- [ ] API 키 없을 때의 폴백 동작 개선
  - AI 없이도 기본적인 설명이 가능하도록 rule 기반 설명 텍스트 강화
  - "더 자세한 설명을 원하시면 `vib config`에서 AI API를 설정하세요" 안내

- [ ] AI 호출 실패 시 에러 메시지 코알못 최적화
  - 현재: "Gemini API 호출 실패: HTTPError 429"
  - 개선: "AI가 지금 바빠서 응답하지 못했어요. 잠시 후 다시 시도하세요. (기술 코드: 429)"
