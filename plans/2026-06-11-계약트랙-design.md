# 계약 트랙 — 설계 (design)

> 작성 2026-06-11 · 브레인스토밍 세션에서 사용자와 단계별 승인 완료
> 근거: "기획 = 실행 계약" 제품 명제 리뷰 — VibeLign의 기획은 문서 작성이 아니라 **AI 코딩 작업의 목표·범위·지시문·검증 기준·복구 지점을 한 번에 묶는 실행 계약**이어야 한다. 가이드 레이어 v6은 이 명제를 카피 수준까지만 반영(기 구현 사실의 표현)하고 파이프라인은 본 트랙으로 분리했다(가이드 design §6 비목표).
> 선행: 가이드 레이어 v6(e0e9893·ef56347 — 5️⃣ 지문 신호·여정 카피), 기획방 카드 추출기(`planning_chat_cards.rs`)·readiness 판정(`planning_chat_readiness.rs`)·기획안 Markdown 렌더(`planning_chat_markdown.rs`)

## 0. 출발점 — 현재 기획안은 이미 계약에 가깝다 (탐색 결과)

| 계약 요소 | 현재 상태 |
|---|---|
| 목표 | ✅ 기획안 "## 한 줄 목표" 섹션 |
| 금지 범위 | ✅ "## 제외할 것" 섹션 |
| 완료 기준 | 🟡 "## 요구사항별 명세" + 확정 시 readiness 판정(🟢/🔴, 수용 기준 축) |
| 구조화 추출 파이프라인 | ✅ 결정 카드 추출기(`extract_and_apply` — CLI 루브릭 → JSON ops → `cards.json`) |
| **손댈 파일/영역 후보** | ❌ 없음 — 기획이 코드 컨텍스트(project_map)와 미연결 |
| 지시문 주입 | ❌ `buildPlanningWorkInstruction`(planningInstruction.ts)은 기획안 경로만 꽂는 보일러플레이트 |
| 검증(범위 비교) 연동 | ❌ 없음 — guard는 앵커·watch 규칙 검사, 기획과 단절 |

즉 본 트랙은 스키마 신설이 아니라 **배선 3개**: ① 계약 블록 추출(scope 신규 + 기존 섹션 구조화) ② 지시문 주입 ③ 5️⃣ 범위 비교 리포트.

## 1. 확정 결정 (브레인스토밍 승인 이력)

| 결정 | 선택 | 기각 |
|---|---|---|
| scope(손댈 파일/영역 후보)의 출처 | **A: 기획 페르소나가 추출** — 확정 시 대화+project_map 요약으로 추정. 전자동·초보자 무부담·기존 파이프라인 재사용 | B: 외부 AI 도구가 작업 시작 시 선언(도구 협조 필요, 흐름 깨짐) / C: 사후 관찰만(약속이 아니라 관찰 — 명제 약화) |
| v1 출하 범위 | **B: ①추출 + ②지시문 주입 + ③범위 비교 리포트** — 계약 명제 한 바퀴 완성, ③은 v6 지문 인프라 재활용이라 증분 비용 작음 | A: ①+②만(검증 축 공백) / C: +guard 판정 연동(추정 scope를 판정에 넣으면 오탐이 초보자 신뢰 타격 — 후속) |
| UI 노출 수준 | **B: 읽기 전용 노출** — 기획안 안 "## 작업 계약" 섹션 렌더. 계약이 보여야 제품 명제가 사용자 눈에 존재 + 추출 오류의 안전판(읽고 이상하면 무시) | A: 보이지 않는 배관(명제 비가시) / C: 편집 가능(수요 미확인, v1 비대) |
| 저장·추출 방식 | **접근 1: `contract.json` 별도 저장 + 확정 시 1회 추출** — JSON 소스·Markdown 파생 뷰(cards.json 관례), 실패 폴백이 readiness 패턴으로 수렴 | 접근 2: Markdown이 소스(산문 파싱 취약) / 접근 3: 카드 kind 확장(대화 중 scope는 이르고 카드 루브릭 복잡화) |

## 2. 아키텍처 · 데이터 흐름

```
[확정 시점 — save_planning_chat_as_markdown (planning_chat.rs:227)]
  대화(messages) + project_map 요약
      → CLI 페르소나 1회 호출 (judge_readiness 직후 — 같은 시점·같은 패턴)
      → contract.json  (세션 디렉터리, cards.json 옆)
      → planning_chat_markdown.rs가 "## 작업 계약" 섹션으로 렌더 (읽기 전용 뷰)
      → PlanningChatSessionResponse에 contract 필드 (cards가 실리는 방식 그대로)

[4️⃣ — 지시문]
  App의 planningResult.contract
      → buildPlanningWorkInstruction이 목표·scope·제외·완료 기준을 지시문에 명시

[5️⃣ — 범위 비교 리포트]
  contract.scope  vs  guideRelevantEntries(현재 changed-set)   ← 가이드 v6 지문 신호 재활용
      → prefix 매칭 → "약속 범위 안 N건 · 범위 밖 M건" 리포트 (홈 '상태 확인' 결과 영역)
      → guard 판정·가이드 단계 추론에는 불개입 (리포트만)
```

- 신규 Tauri 커맨드 0 (기존 save 흐름 확장 + 응답 필드 추가)
- 신규 영속 파일 1 (`contract.json`, 세션 디렉터리)
- "활성 계약" = 최신 저장 세션의 계약 — 가이드 `hasPlanDoc`(planningResult.outputPath)과 동일 규칙
- 신규 모듈: `planning_chat_contract.rs`(추출·검증·타입), 프론트 `scopeReport` 순수 함수

## 3. 계약 스키마 (`contract.json`, version 1)

```json
{
  "version": 1,
  "extractedAt": "1781234567890",
  "goal": "한 줄 목표",
  "scope": [
    { "path": "src/pages/Home.tsx", "kind": "file", "reason": "상태 확인 버튼이 있는 곳" },
    { "path": "src/components/nav/", "kind": "dir", "reason": "가이드 스트립 추가 위치" }
  ],
  "exclusions": ["결제 흐름은 건드리지 않음"],
  "doneCriteria": ["체크포인트 저장 후 스트립이 4단계로 바뀐다"]
}
```

- `extractedAt`은 epoch ms 문자열 — 메시지 `createdAt` 등 코드베이스 기존 타임스탬프 관례(`timestamp_ms()`)와 동일 (구현계획에서 정정)
- `goal`·`exclusions`·`doneCriteria` = 기획안 기존 섹션(한 줄 목표·제외할 것·요구사항별 수용 기준)을 계약 어휘로 **구조화** — 새 정보 아님
- `scope`만 진짜 신규(페르소나 추정). **추출 직후 실존 검증**: project_map/디스크에 없는 경로는 버림. 전부 탈락하면 `scope: []` — ③ 리포트 미노출, ② 지시문엔 scope 줄 생략
- 소비 규칙: ③은 `scope`만, ②는 전 필드

## 4. 추출 — 프롬프트 입력·파싱·실패 처리

`save_planning_chat_as_markdown` 안, `judge_readiness` 직후. 신규 모듈 `planning_chat_contract.rs` (CARD_RUBRIC 패턴):

- **입력**: ① 전체 대화(readiness와 동일 방식 — 결정 카드는 대화의 파생이라 중복, 미포함) ② project_map 요약 — 경로+카테고리만, `category=core` 우선 상한(200줄). 상한 초과로 잘린 경우 프롬프트에 잘렸음을 명시(구현계획에서 구체화)
- **루브릭 핵심 규칙**: "scope는 제공된 project_map 목록에 있는 경로만 / 확신 없으면 디렉터리 단위 / 모르면 빈 배열" — 환각 경로를 루브릭 + 사후 실존 검증 **이중 차단**
- **출력**: §3 스키마의 strict JSON. 파싱 실패 = 추출 실패
- **실패 모드** (전부 readiness "확인 못 함" 패턴으로 수렴, **계약 없음 = 현행 동작 그대로, 퇴행 0**):
  - AI 미설정·CLI 실패·파싱 실패 → `contract.json` 미생성, Markdown에 "## 작업 계약: 확인 못 함(활성 AI 없음)" 한 줄, 지시문 보일러플레이트 폴백, 리포트 미노출
  - 재확정(재저장) → 재추출·덮어쓰기

## 5. 지시문 주입 (`buildPlanningWorkInstruction`)

contract 존재 시 기존 "작업 기준:" 블록 위에 삽입:

```
이번 작업의 계약(기획안에서 추출):
- 목표: {goal}
- 손댈 범위(이 밖은 건드리기 전에 사용자에게 확인): src/pages/Home.tsx, src/components/nav/ …
- 건드리지 말 것: 결제 흐름은 건드리지 않음
- 완료 기준: 체크포인트 저장 후 스트립이 4단계로 바뀐다 / …
```

- 시그니처에 `contract?: PlanningContract` 옵션 추가 — 미존재 시 현행 출력과 byte 동일(기존 테스트 무변경)
- 이 시점부터 가이드 4️⃣ 카피("VibeLign이 만들어준 안전 지시문")가 내용적으로도 완전한 사실이 됨

## 6. 5️⃣ 범위 비교 리포트

- **순수 함수** (vitest TDD): `scopeReport(scope, changedEntries)` → `{ inScope: number, outOfScope: ChangedEntry[] }`. 매칭: `kind:"dir"`=prefix, `kind:"file"`=완전일치. `.vibelign/`·`.omc/`·`.git/`은 v6 `guideRelevantEntries`가 이미 제외
- **표시**: 홈 '상태 확인' 결과 영역 한 줄 — "약속 범위 안 변경 N건 · **범위 밖 M건**" + 범위 밖 경로 목록(접기). 어조는 판정 금지: "범위 밖 변경이 있어요 — 의도한 작업인지 확인해보세요"
- **불개입 원칙**: guard 판정·가이드 단계 추론에 영향 없음 (Q2 결정 — scope가 추정인 동안 판정 금지)
- **changed-set 조회는 Home이 guard 실행 시점에 직접**(`listChangedFiles`, 구현계획에서 구체화) — 가이드 v6의 App state 재사용 대신 직접 조회로 정정. 근거 ③: ① v6의 변경 effect는 체크포인트(latestBackupId)가 있어야 돌지만 계약 리포트는 체크포인트 없이 기획→작업한 사용자에게도 떠야 함 ② guard 결과와 같은 순간의 changed-set이라 리포트-판정 시점 일관 ③ v6 구현과 의존성 분리(병행 가능). 메타 경로(`.vibelign/`·`.omc/`·`.git/`) 제외는 `scopeReport` 안에서 자체 수행(v6 `guideRelevantEntries`와 동일 목록 — 미구현 모듈 의존 금지)

## 7. 엣지 케이스 · 규칙

1. **비-git 프로젝트**: changed-set 항상 빈 목록(v6 동일) → 리포트 무의미, 미노출 조건에 포함
2. **scope 빈 배열**(추출이 비웠거나 실존 검증 전탈락): 리포트 미노출, 지시문 scope 줄 생략
3. **기획 없이 작업**: 계약 부재 = 현행 동작 — 계약은 가산점이지 게이트가 아님 (B안 "선형 강제 금지" 유지)
4. **stale 계약**(저장 후 코드 대폭 변경): v1 무대응 — 리포트가 "범위 밖 다수"로 자연 신호. 계약 나이 표시는 후속
5. **추출 어조**: scope의 reason은 사용자 노출 텍스트 — 영문 용어 금지(가이드 v5 북극성 #3 동일 기준)

## 8. 검증 방침 (프로젝트 관례)

- 추출 JSON 파싱·실존 검증·Markdown "작업 계약" 렌더 — **Rust 단위테스트** (`planning_chat_synthesis_tests.rs` 패턴)
- `scopeReport`·지시문 주입(contract 유/무 분기) — **vitest TDD** (planningInstruction 기존 테스트 옆)
- UI(홈 리포트 표시) — `tsc --noEmit` + tauri dev 육안 (UI 단위테스트 비요구 관례)
- 회귀: contract 부재 시 지시문 출력 byte 동일, 기존 기획안 저장·readiness 흐름 무변경

## 9. 비목표 (명시)

- guard 판정 연동(범위 밖 변경 → warn 승격) — scope 추정 정확도 검증 후 후속
- anchor intent/protect 자동 설정 — 후속
- 계약 편집 UI(Q3의 C) — 수요 확인 후
- 계약 나이·stale 표시 — 후속
- 외부 AI 도구의 작업 시작 선언(Q1의 B) — MCP 통합 트랙과 함께 검토

## 10. 구현 순서 제안 (계획서에서 상세화)

1. `planning_chat_contract.rs` — 타입·루브릭·파싱·실존 검증 (Rust TDD)
2. save 흐름 통합 + `contract.json` 저장 + 응답 필드 + Markdown "작업 계약" 렌더
3. `buildPlanningWorkInstruction` contract 주입 (vitest TDD)
4. `scopeReport` 순수 함수 (vitest TDD) + App changed entries state 확장
5. 홈 '상태 확인' 결과 영역 리포트 표시
6. 전체 검증 + 가이드 howto 카피 미세 동기화(4️⃣ — 계약 내용 언급 가능해짐)

각 단계 독립 커밋·출하 가능, 위험 오름차순.
