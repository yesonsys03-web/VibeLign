> ## 먼저, 잘 짜인 부분
> "기획 → 구현 → 추적 → 회고" 한 사이클로 묶은 그림(개선 1·3·4의 시너지)은 방향이 맞아. 특히 **개선 4-1 "AI로 만들기" 버튼**은 사용자가 가장 크게 체감할 지점이야. 복사·붙여넣기 한 단계 없애는 게 코알못한테는 다른 어떤 기능보다 커.

> ## 모순 3가지
> 1. **작업 순서가 두 군데서 다름.** 중간 "작업 순서(권장)"은 *개선 1 → 2 → 3* 인데, 끝의 "전체 로드맵"은 *Phase 1 = 개선 1+3-1+4-1 묶음*. 어느 걸 따라야 할지 읽는 사람이 헷갈려. 하나로 통일해야 해.
> 2. **자동 페르소나 라우팅이 고정인지 동적인지 안 정해짐.** "첫 메시지→지오, 2번째→클로이, 3번째→미나"라고 못 박아놓고, 바로 옆에 "또는 AI가 맥락에 따라 결정"이라 적혀 있어. 둘은 완전히 다른 설계야.
> 3. **진행률 "구현됨" 판정의 신뢰 문제.** 키워드로 코드와 대조해서 "✅ 구현됨"을 띄우는데, 껍데기만 있어도 매칭될 수 있어. 사용자가 "됐다는데 안 되잖아"를 한 번 겪으면 대시보드 전체를 안 믿게 돼.
>
> ## 빠진 부분 (제품 관점)
> - **사용자가 판정을 못 고침.** 진행률이 틀렸을 때 "이건 사실 아직 안 됐어"라고 직접 수정하는 길이 없어. 자동 추정은 반드시 *사람이 덮어쓸 수 있어야* 신뢰가 생겨.
> - **코알못은 자기가 코알못인 줄 모름.** "초보 모드 체크박스"를 스스로 켤 거라 기대하면 안 돼. 진입 *기본값*을 초보 모드로 두고, 익숙한 사람이 "직접 선택할게"로 빠지는 구조가 맞아.
> - **지식 저장은 입구만 있고 출구가 없음.** 개선 3은 저장 버튼·자동 감지까지 촘촘한데, 정작 *저장한 걸 어떻게 다시 찾아 쓰는지*가 비어 있어. 저장은 수단이고, "왜 이렇게 만들었지?"에 답하는 게 목적이잖아.
> - **자동 기획안 생성 기준이 너무 기계적.** "턴 수 ≥ 6, 키워드 3개"는 간단한 아이디어는 너무 늦게, 복잡한 건 너무 일찍 끊을 수 있어. 턴 수보다 "핵심 기능이 윤곽을 잡았는가"가 진짜 신호야.

> ### 1. 진행률 신뢰: 다른 도구들은 어떻게 할까?
> *   **Linear / GitHub Projects**: 이 도구들은 절대 AI로 '자동 추정'해서 진행률을 바꾸지 않아. 대신 브랜치 이름에 작업 번호를 넣거나, 커밋 메시지에 `Closes #10` 같은 명시적인 꼬리표를 달았을 때만 상태를 완료로 바꿔줘.
> *   **미나의 질문 (엣지케이스)**:
>     *   **판정 충돌**: AI가 멋대로 코드를 분석해서 "✅ 구현 완료"라고 판단했는데, 사용자가 "아직 버그가 많아서 안 됐어" 하고 진행률을 수동으로 되돌려놓는 상황이 있을 거야. 이때 다음 코드 검사에서 AI가 또 자동으로 "어? 코드에 기능이 있으니 구현 완료네요!" 하고 다시 덮어써 버리면 사용자는 짜증이 나겠지? 사용자의 수동 판정을 우선해서 잠금(Lock)하는 장치가 필요하지 않을까?
>     *   **코알못의 피드백**: 초보 사용자는 코드에 꼬리표를 다는 규칙을 모르잖아. 그렇다면 사용자가 "이 기능 테스트해봤는데 잘 안 돼"라고 기획방에 한마디 하거나 대시보드에서 ⬜(미완료)를 누르는 것만으로도 상태가 확실히 반영되어야 해.
>
> ### 2. 지식 재활용: 나중에 어떻게 다시 꺼내 쓸까?
> *   **Notion / Obsidian / ADR**: 개발자들은 코드 저장소 안에 결정 문서(ADR)를 모아두고 필요할 때 검색해서 읽어. 하지만 초보 사용자가 기획방에서 "💡 지식 저장" 버튼을 눌러서 만든 문서를 나중에 직접 폴더를 뒤져서 찾아 읽을 가능성은 거의 없어.
> *   **미나의 질문 (엣지케이스)**:
>     *   **지식의 자동 소환**: 사용자가 직접 문서를 찾아보는 게 아니라, AI가 코드를 구현하기 직전에 **"이전에 로그인 방식을 OAuth로 결정해 두셨네요. 이 결정대로 개발을 시작할까요?"**라고 사용자에게 확인해 주거나, 코드 작성 시 자동으로 그 지식 문서를 참고해서 적용하는 흐름이 자연스럽지 않을까?
>     *   **결정의 번복**: 기획 도중에 마음이 바뀌어서 "아까는 구글 로그인으로 하려고 했는데, 그냥 이메일 로그인으로 바꿀래"라고 대화를 틀었을 때, 기존에 저장되었던 지식 문서는 어떻게 수정되어야 할까? AI가 "이전 결정을 덮어쓸까요?"라고 물어보고 알아서 지식 문서를 업데이트해 줘야 사용자가 편할 것 같아.
