## 2. 표준 여정 (v3 — 확정)

가이드가 가르치는 단일 소스. 6단계:

| # | 단계 | 위치(IA) | 성격 | 안내 내용 |
|---|---|---|---|---|
| 1️⃣ | 프로젝트 열기 | 온보딩 | 자동 | 폴더 선택 → 코드맵·앵커 준비. **감시 자동 시작**(변경을 지켜보다 위반을 알려줌 — ⚠️ 자동 백업 아님). 가이드는 안심 메시지만 |
| 2️⃣ | 기획 | 기획 단계 · 기획방 | 행동 | "무엇을 만들고 싶은지" 적기 → AI 페르소나 대화 → 기획안 확정(확정안 = 다음 단계 AI 작업의 약속·기준점) |
| 3️⃣ | 체크포인트 저장 | 유지보수 · 백업 | **수동 행동** | AI에게 시키기 **전에** 현재 상태 저장 — "망쳐도 여기로 되돌아올 수 있게" |
| 4️⃣ | AI에게 작업 시키기 | 개발 · 코드탐색 | 행동(외부) | **VibeLign이 기획안 기반 안전 지시문 생성**(기 구현: `buildPlanningWorkInstruction`) → 복사 → 외부 AI 도구(Claude Code 등)에 붙여넣기 |
| 5️⃣ | 결과 검증 | 홈 · 상태 확인(guard) | 행동 | guard 검사 — AI가 약속한 범위만 고쳤는지 확인. 문제 발견 시 진단으로 연결 |
| 6️⃣ | 저장 또는 되돌리기 | 유지보수 · 백업 | 행동 | 잘 됐으면 체크포인트, 망쳤으면 undo(3️⃣ 시점 복귀) → 4️⃣로 반복 |

여정 사실 정정 이력(중요): 감시는 `Onboarding.tsx:122`에서 프로젝트 열기 시 자동 시작. 감시는 **백업이 아니라 변경 감시·위반 감지**(`watch_cmd.py` → watch_engine). 자동 백업 기능은 현재 없음 → 체크포인트는 수동 행동 단계로 유지.

추가 정정(2026-06-10 critic 리뷰 반영): **guard 실행 버튼은 진단(Doctor) 탭에 없다.** guard는 홈 '상태 확인'(`Home.tsx` `handleRunGuard`)과 고급 카드(`GuardCard`)에서만 실행되고, 진단은 guard가 문제를 발견했을 때 `onOpenDoctor`로 연결되는 후속이다. → 5️⃣의 목적지는 **홈**으로 정정 (진단으로 보내면 초보자가 guard 버튼을 못 찾고 길을 잃음).

여정 루프 한계 명시(critic M1): 3️⃣(체크포인트)은 **첫 사이클 전용 안내**다. 두 번째 사이클부터의 "작업 전 저장"은 6️⃣("잘 됐으면 체크포인트")이 담당하며, 6️⃣에서 저장하면 변경 기준점(baseline)이 리셋되어 4️⃣로 돌아간다 — 루프가 닫힌다. 이것은 의도된 설계이지 누락이 아니다.

신호 모델 정정(v6, 2026-06-11 외부 리뷰 4건 반영): 변경 신호의 단위를 "경로 집합"에서 **변경 지문(path+status+mtime+size)**으로 교체한다. 근거 H1 — 체크포인트는 git 상태를 바꾸지 않으므로, 커밋 안 하는 초보자는 첫 사이클 이후 dirty 집합이 누적된 채 baseline에 박히고, 2사이클차부터 AI는 대부분 **이미 baseline에 있는 파일을 재수정**한다. 경로 대칭차로는 이게 0이라 4️⃣→5️⃣ 자동 전환이 사실상 죽는다(엣지가 아니라 정상 상태). 근거 H2 — guard 리셋이 "변경 수 증가" 기준이면 같은 파일 재수정·동수 집합 교체에서 stale pass(거짓 "검증 완료" 6️⃣)가 남는다 — 초보자 신뢰에 직격. 둘 다 뿌리가 같아(집합/수 ≠ 내용) 지문 하나로 동시에 해결한다. 이를 위해 기존 `list_changed_files` 응답에 `mtime_ms`·`size`를 추가 — **"백엔드 변경 0" 원칙의 유일한 최소 예외**(§4-5). 함께 반영: 가이드 OFF 중 축하 발화 금지(M1), AI 도구 재탐지(M2).

카피 보강(v6, "기획 = 실행 계약" 제품 명제 리뷰 반영): 2️⃣ 확정 카피에 "이 문서가 다음 단계 AI 작업의 약속(목표·범위)이 돼요" 프레임, 4️⃣ 카피를 "VibeLign이 기획안 기반 안전 지시문을 만들어준다"로 — 이미 구현된 `buildPlanningWorkInstruction`(planningInstruction.ts)의 사실화이지 과장이 아님. **5️⃣ 카피는 의도적으로 불변**: guard의 실제 검사 대상은 앵커 위반·watch 규칙이지 기획안의 범위·완료 기준이 아니므로, "기획안 기준으로 검사"라고 쓰는 순간 허위가 된다(현재 "약속한 범위" = 앵커 intent라 사실 범위 내). 기획안→계약 블록→지시문 주입→guard 연동 파이프라인은 **별도 계약 트랙**으로 분리(이 설계 비목표).

### 3.1 `guide.ts` (순수)

- `JOURNEY_STEPS: JourneyStep[]` — 6단계 정의: `{ step: 1..6, label, shortAction, detailHowto, targetPage: Page, stage: Stage|null }`
- `inferStep(signals): 2|3|4|5|6` — 신호 → 현재 단계. **1️⃣은 반환하지 않음**: 프로젝트가 열려 있어야 가이드가 렌더되므로 1️⃣은 항상 완료 상태 (1️⃣은 `JOURNEY_STEPS` 정의에는 포함 — 사용법 따라하기 아코디언용). 입력:
  - `hasPlanDoc: boolean` (확정 기획안 존재 = `planningResult?.outputPath` 기준)
  - `hasCheckpoint: boolean`
  - `changedFileCount: number | null` — **마지막 체크포인트 이후** 달라진 파일 수. git changed-set(`listChangedFiles`)을 최신 체크포인트 시점에 캡처한 baseline과 비교해 계산. git HEAD 기준 절대 수가 아님 — 커밋 안 하는 초보자는 git 기준으론 한번 dirty 되면 영원히 >0이라 루프 신호로 못 씀(critic M1·M2 반영). **비교 단위는 경로가 아니라 변경 지문(path+status+mtime_ms+size, v6 외부 리뷰 H1)**: 경로 대칭차에 더해, **같은 경로라도 지문이 다르면 변경으로 센다** — 체크포인트 시점에 이미 dirty였던 파일을 AI가 재수정하는 케이스(커밋 안 하는 초보자에겐 2사이클차부터의 일상)를 잡기 위함. null = **조회가 예외로 실패한 경우만**. ⚠️ **비-git 프로젝트는 null이 아니라 0** — `git_status.rs`가 비-git·git 미설치에 빈 목록을 반환(에러 아님)하므로 변경이 항상 0으로 읽혀 5️⃣·6️⃣ 자동 전환이 영영 없다. 그래서 4️⃣ affordance가 비-git 사용자의 **유일한** 검증 진입로(critic 재리뷰 C1)
  - `guardStatus: "ok" | "issue" | null` — 마지막 변경 이후의 guard 실행 결과(홈 '상태 확인' 콜백). 엔진 `status` 필드의 판정 어휘는 **"pass"/"warn"/"fail"**("ok" 값은 없음, `vib_guard_cmd.py` `_guard_status`): **pass → "ok"**, warn/fail → "issue"(warn=약속 범위 밖 변경 감지라 5️⃣ 유지가 여정 의도). issues 배열이 비어도 warn/fail일 수 있어 `issues.length` 기준은 부족. **리셋 기준은 "guard가 검사한 시점의 changed-set 지문 ≠ 현재 지문"(v6 외부 리뷰 H2)** — App이 guard 실행 순간의 `changedSetFingerprint`를 기억해 두고, 이후 조회된 지문이 달라지면 null로 리셋한다. 이전 기준(변경 수 증가 시에만 리셋)은 같은 파일 재수정·동수 집합 교체(A/B→C/D)에서 stale pass(거짓 "검증 완료" 6️⃣)를 남겼다. 지문 비교는 무변경 재조회(탭 전환)에선 동일하므로 critic 재리뷰 M1이 막으려던 오리셋도 그대로 없다. 부분 undo도 지문이 달라져 리셋되는데(이전 설계는 유지), 변경된 상태가 달라졌으면 재검증 안내가 옳으므로 **의도된 강화**다. null = 아직 안 돌림
  - `planningPending: boolean`
- 보조 순수 함수 (전부 vitest TDD):
  - `latestCheckpointId(backups)` — `backupList` 결과에서 최신 체크포인트 id (createdAt 내림차순, 정렬 순서 가정 안 함)
  - `entryFingerprint(entry)` / `changedSetFingerprint(entries)` — 변경 entry 하나/집합의 지문(v6). entry 지문 = `path+status+mtime_ms+size`, 집합 지문 = entry 지문 정렬 후 결합(순서 무관). guard 리셋 비교와 baseline 비교의 **단일 기준**
  - `countChangesSinceBaseline(baseline, current)` — entry 목록 비교: 경로 대칭차 + **동일 경로 지문 변화**(재수정)도 변경으로 센다(v6)
  - `guideRelevantEntries(entries)` — `.vibelign/`·`.omc/`·`.git/` 메타데이터 경로 제외 (guard 리포트·체크포인트 메타 churn이 변경으로 오인되는 것 방지)
- 영속 상태 (localStorage):
  - `guideEnabled` — 전역 1개 (숙련도는 프로젝트와 무관)
  - `guideStepOverride:<projectDir>` — **프로젝트 경로별** (진행 위치가 프로젝트 간 새면 안 됨)
  - `guideBaseline:<projectDir>` — `{ checkpointId, entries[] }`(entry = path·status·mtime_ms·size, v6). 최신 체크포인트 id가 바뀐 걸 감지한 순간 현재 changed-set을 캡처. "체크포인트 이후 변경" 계산의 기준점

### 3.2 `GuideStrip.tsx` (표시만)

기존 App.tsx 상태 스트립(337~355행: 백업 유무 + 기획 진행)을 대체·통합:

```
🧭 지금: 3️⃣ 체크포인트 저장 — AI에게 시키기 전에 안전 저장  [백업으로 이동 →]  ‹ › ×   (우측: 기존 상태 신호 유지)
```

- `[이동 →]` 버튼 = 기존 `navigate(targetPage)` 재사용 (진단은 launch intent 경유 그대로)
- `‹ ›` = 수동 보정(override 기록), `×` = `guideEnabled=false` (설정에서 재활성)
- 가이드 off 시: 스트립은 기존 상태 표시(백업/기획 진행)만으로 축소 — 스트립 자체는 사라지지 않음
- **시인성(critic M3 반영)**: 가이드 활성 시 가이드 구간은 기존 상태 신호(11px·#888)보다 한 단계 도드라지게 — 13px, 단계 아이콘·라벨은 허브 카드와 같은 `#FBBF24` 강조, 세로 패딩 6px. 가이드 off 시엔 기존 얇은 줄(3px·11px) 그대로. 초보자가 작업 탭 안에 있을 때 유일하게 보이는 안내 표면이므로 두 탭바 아래에서 묻히면 안 됨
- **4️⃣ 수동 전진 affordance(critic M2 반영)**: step 4일 때 상시 노출되는 "AI 작업이 끝났거나 멈췄나요? 확인하러 가기 →" 버튼 — 클릭 시 5️⃣로 override + 5️⃣ 목적지로 이동. 근거 둘: ① 외부 AI 작업 종료는 어떤 신호로도 100% 감지 불가 ② **비-git 프로젝트는 변경이 항상 0으로 읽혀 자동 전환이 아예 없으므로 이 버튼이 유일한 검증 진입로**. step 4에서 changedFileCount는 정의상 0 또는 null이라(>0이면 이미 5️⃣로 추론됨) 별도 게이트 불필요. 문구가 "끝났거나 **멈췄거나**"인 이유: 완주를 전제하면 중도 중단자가 자기 안내가 아니라고 느낌 — 하다 만 작업일수록 "뭐가 바뀌었는지 확인"이 옳은 다음 행동
- **포기 비상구(2026-06-10 사용자 지적 반영)**: 중도 포기는 7번째 단계가 아니라 **4️⃣·5️⃣ 어디서든 빠질 수 있는 비상구**로 다룬다. 4️⃣·5️⃣일 때 스트립에 보조 출구 "잘 안 됐나요? 되돌리기 →"(백업 탭 이동, 회색톤 — 주행동을 가리지 않게) 상시 노출. 근거: undo 안내가 6️⃣ 하우투 안에만 있으면 guard pass를 통과한 완주자만 되돌리기를 발견한다 — 정작 가장 되돌리기가 필요한 "guard가 빨간 결과를 뱉은 5️⃣의 포기 직전 사용자"에게 출구가 안 보임. "망쳐도 되돌아올 수 있다"는 핵심 가치를 가이드 동선에도 반영
- **4️⃣ AI 도구 미보유 분기(북극성 점검 Critical #1)**: 4️⃣는 가이드가 사용자를 앱 밖(외부 AI 도구)으로 내보내는 유일한 지점 — 도구가 없는 입문자에겐 첫 사이클 최대 이탈 절벽. 앱이 이미 가진 `detectInstalledTools()` 신호를 재사용해, **탐지된 도구가 0개로 확정**되면 4️⃣ 스트립에 "AI 도구가 아직 없나요? 설치 도움받기 →"(설정의 'AI 도구 설정' 이동) 버튼을 추가 노출. 탐지 실패·미완은 분기 미노출(보수적 — 추측 안내 금지). **탐지 시점은 마운트 1회 + 설정에서 빠져나올 때 재탐지(v6 외부 리뷰 M2)** — 이 버튼이 유도하는 동선이 "스트립 → 설정 → 설치 → 복귀"인데 마운트 1회만 탐지하면 설치 직후 복귀 지점에서 "도구가 아직 없나요?"가 stale로 남는다(앱 재시작 전까지). 사용법 4️⃣ 하우투에도 정적 한 줄 추가. 전체 설치 마법사·도구별 맞춤 안내는 후속
- **첫 사이클 완주 축하(북극성 점검 #2)**: 6️⃣→4️⃣ 전환(= 저장으로 루프가 닫힌 순간)을 감지해 **프로젝트당 1회** 스트립에 "🎉 첫 사이클 완주!" 한 줄 노출(× 닫기, localStorage 플래그로 재발 방지). 입문자의 첫 성공을 명시적으로 인정해 두 번째 사이클로 끌어들이는 동기 마디 — 지금 루프는 기능적으로는 닫히지만 정서적으로는 닫히지 않는다는 점검 지적 반영. 앱 시작 직후(prev=null)에는 발화 금지(오발 방지). **가이드 OFF 중에는 발화·1회성 플래그 소모 모두 금지(v6 외부 리뷰 M1)** — `useGuide`는 enabled와 무관하게 step을 계산하므로 OFF 상태에서도 6️⃣→4️⃣ 전환이 감지될 수 있다. 게이트는 **App의 발화 effect에**(스트립 prop이 아니라) — prop에서 막으면 OFF 중에 플래그만 소모돼 나중에 가이드를 켠 사용자가 축하를 영영 못 받는다. ref 전진도 게이트 뒤(OFF 중 전환 비소모 — 재활성 시 첫 관찰 전환에서 발화)

### 3.3 `StageHubCards.tsx` 연동

- 현재 단계가 속한 카드: 강조 테두리 + "지금 할 차례" 배지 + 구체 행동 1줄("→ 3️⃣ 체크포인트를 저장하세요")
- 완료 단계 카드: "✓ 완료" 톤다운 / 다음 카드: "다음" 표시
- 매핑: 기획 카드=2️⃣, 개발 카드=4️⃣, 유지보수 카드=3️⃣·6️⃣ (1:1 아님 — 카드는 "현재 단계 포함 여부"로 강조). **5️⃣은 홈의 '상태 확인' 행동이라 카드 밖** — 5️⃣일 땐 어떤 카드도 "지금 할 차례"가 아니고 스트립이 안내를 담당

### 3.4 사용법 탭(따라하기 우선)

`Home`의 `manual_list` 뷰 상단에 여정 6단계 아코디언 추가, 기존 명령어 사전은 그 아래 유지:

- 신규 라우팅 없음 (한 페이지)

## 4. 엣지 케이스 · 규칙

1. **추론 충돌 시 뒤 단계 우선** (결정적 규칙). 예: 기획안 있음 + 코드 변경 감지 → 5️⃣.
2. **override 수명**: 추론 결과가 바뀌는 순간 자동 해제 — 실제 상황 변화가 사용자 고정보다 우선. 영구 어긋남 방지.
3. **기획 건너뛰는 사용자**: 2️⃣ 안내 강요 금지 — ‹ › 보정/× 끄기로 해소 (B안의 "선형 강제 금지" 원칙 유지).
4. **신호 로딩 전 배너 미렌더** — 잘못된 안내 깜빡임 방지.
5. **신호 배선은 v1 범위(critic M1·M2 반영, "후속"에서 승격)**: `changedFileCount`(체크포인트 baseline 대비)·`guardStatus`(홈 guard 콜백) 둘 다 v1에서 배선한다. 기존 프론트 API(`listChangedFiles`·`backupList`)와 localStorage 사용. **"신규 백엔드 0"의 유일한 최소 예외(v6)**: 기존 `list_changed_files` 응답의 `ChangedEntry`에 `mtime_ms`·`size` 두 필드를 추가한다(신규 커맨드·신규 파일·Python 변경 없음) — 경로 집합만으로는 이미 dirty인 파일의 재수정(H1)과 guard stale pass(H2)를 막을 수 없고, 프론트에 내용 지문 재료가 전무함을 확인(ChangedEntry=path·status뿐, checkpoint-diff류 커맨드 부재). 조회가 예외로 실패하면 null → 추론 규칙 보수적 생략(추측 안내 금지). **비-git 프로젝트는 실패가 아니라 빈 changed-set(=0)으로 읽힌다는 점에 주의**(`git_status.rs`는 비-git에 `Ok(Vec::new())` 반환) — null 경로가 아니라 "영원히 0" 경로이며, 이 사용자들에겐 4️⃣ affordance가 유일한 검증 진입로다(critic 재리뷰 C1).
6. **신호 갱신 주기**: `backupList`·`listChangedFiles`는 프로젝트 변경 + **탭(page) 전환 시** 재조회. 기존 App의 backupList는 projectDir 변경 시에만 조회해서, 사용자가 3️⃣을 따라 체크포인트를 저장해도 스트립이 4️⃣로 안 넘어가는 갭이 있었음(critic 리뷰에서 발견) — page 의존성 추가로 함께 해소.
7. **baseline 캡처 타이밍 한계(명시)**: baseline은 체크포인트 생성 순간이 아니라 "App이 새 최신 체크포인트를 알아챈 순간"(다음 재조회)의 changed-set이다. 저장 직후 재조회 전에 AI가 파일을 바꾸는 좁은 경쟁 구간은 허용(휴리스틱 가이드이지 정합성 보장 장치가 아님).
8. **중도 포기·방치는 5️⃣로 수렴(의도)**: AI가 일을 하다 말았어도 파일은 이미 바뀌어 추론이 5️⃣을 가리킨다 — "뭐가 바뀌었는지 확인하라"는 하다 만 작업에도 옳은 안내. 신호가 전부 디스크 기반이라 며칠 뒤 돌아와도 올바른 단계를 재계산한다. 단 `guardStatus`는 메모리 상태라 앱 재시작 시 소실 → 6️⃣에 있던 사용자가 5️⃣로 후퇴하는데, "시간이 지났으면 재검증"이라 안전한 방향이므로 **의도된 동작**이다. v6 외부 리뷰 Open Question(guard 결과 localStorage 영속) 검토 결과 **v1은 메모리 유지 재확인** — 지문이 생겨 영속화가 *안전해지긴* 했지만, 휴리스틱 가이드에서 재시작 후 재검증 안내는 그 자체로 옳은 방향이고 영속화는 스키마만 늘린다. 검사 지문과 함께 저장하는 영속화는 후속. 포기 출구는 §3.2 비상구가 담당.
9. **지문 한계(v6, 명시)**: 내용 지문은 mtime+size **근사**다. 같은 초(저해상도 mtime 파일시스템, 예: FAT 2초)에 같은 크기로 재작성되면 미감지 — 휴리스틱 가이드로 허용. 반대로 내용이 같아도 mtime이 바뀌면(touch·재저장) 변경으로 오인하는데, 이는 5️⃣(재검증) 방향 오탐이라 **안전한 쪽**이다. 4️⃣ 수동 affordance가 미감지의 최종 안전망(§3.2).
