# 강 PD UX 재편 — B안 3단계 IA 설계 (홈 허브 + 단계 전환 바)

> 작성 2026-06-10 · 브레인스토밍(비주얼 컴패니언) 거쳐 확정
> 근거: 강 PD 피드백 ①(프로그램/인터페이스 UI/UX 수정) 중 *상위 IA(탭 구조)* 부분.
> 관계: A안([2026-06-10-강PD-UX-A안-빠른개선-구현계획.md](./2026-06-10-강PD-UX-A안-빠른개선-구현계획.md), 커밋 9836462)은 표면(라벨·유틸·상태)만 손댐. **B안은 그 한 층 위 IA를 재편**한다. A안 산출물(상태 스트립·우측 유틸 클러스터)은 B안 홈 허브에 흡수된다.
> 이 문서는 **설계(spec)** 다. 구현 단계 분해는 후속 implementation plan(writing-plans)에서 다룬다.

---

## 1. 문제 / 목표

- 강 PD 지적: 초보자가 "어디서 뭘 할지" 모름. 상단 탭이 평평하고, **기획방이 탭에 없어 홈 안에 숨어 있음**.
- B안 목표(최대 범위 선택): **3단계 IA(기획·개발·유지보수) + 가이드 레이어**("지금 어디 / 다음 뭐")로 재편하고 **기획방을 1급 단계로 승격**한다.
- 비목표(YAGNI): CS 단계는 만들지 않는다(내용물이 에러로그뿐). 선형 강제(단계 잠금) 안 함.

## 2. IA 구조 (확정: "홈 허브 + 단계 전환 바" — 브레인스토밍 C안)

두 가지 화면 모드:

**(a) 홈 허브 — 첫 화면 / 단계 바깥**
- 상단: 얇은 바 `홈` + 우측 유틸(`사용법 · 폴더열기 · 설정 · 프로젝트명↩`)
- 상태 스트립(A안 재활용): `✓ 백업 데이터 있음 · ● 기획 진행 중`
- 시작 입력: `무엇을 만들고 싶으세요? [____] [시작]` (기존 홈 기능)
- **큰 카드 3장** = 3단계 진입 + 가이드 뱃지(§4)

**(b) 단계 내부 — 집중 작업**
- 상단: **단계 전환 바** `← 홈 │ 기획 · 개발 · [현재단계]` + 우측 유틸
  - 핵심: 홈을 거치지 않고 기획↔개발↔유지보수 즉시 이동 → C안의 유일한 단점(2-depth 클릭) 제거
- 그 아래: **단계별 서브탭** (해당 단계에 속한 페이지들)
- 본문: 선택된 서브탭의 기존 페이지 컴포넌트

비선형 허용: 어느 단계든 언제나 클릭 가능. 가이드는 *권유*지 *강제*가 아니다.

## 3. 페이지 매핑 (기존 페이지 전부 재활용)

| 단계 | 서브탭 = 기존 페이지(컴포넌트) | 기존 `page` 값 |
|---|---|---|
| 📋 **기획** | 기획방(PlanningRoom) ← **승격** | `planning` |
| ⚙️ **개발** | 코드탐색(CodeExplorer) · 문서(DocsViewer) | `code` · `docs` |
| 🛡️ **유지보수** | 진단(Doctor) · 백업(BackupDashboard) · 에러로그(ErrorLogs) | `doctor` · `backups` · `logs` |
| (상단 우측 유틸) | 사용법(manual) · 폴더열기(action) · 설정(settings) | `manual` · — · `settings` |
| (허브) | 홈 | `home` |

→ **새 페이지 컴포넌트를 만들지 않는다.** 기존 `page` 상태값을 그대로 두고, 각 `page`가 "어느 단계에 속하는지"의 매핑만 새로 정의한다(전환 바·서브탭 강조용).

```
stageOf(page):
  planning            -> "기획"
  code, docs          -> "개발"
  doctor, backups, logs -> "유지보수"
  home, manual, settings -> (단계 없음: 허브/유틸)
```

## 4. 가이드 레이어 (기존 신호만 — 신규 백엔드 없음)

홈 허브의 카드 3장에 상태 뱃지:

| 카드 | 뱃지 | 출처(기존 신호) |
|---|---|---|
| 📋 기획 | 없음 / ● 진행 중 / 완료 | `planningResult` (pending 메시지 유무·존재) |
| ⚙️ 개발 | (뱃지 없음, "열기"만) | 기존 신호 없음 → 신규 계산 안 함 |
| 🛡️ 유지보수 | 백업 N개 | `backupList(projectDir).backups.length` (= A안 `hasCheckpoint` 출처) |

단계 내부 상단에도 상태 스트립을 유지할지는 구현 시 결정(허브와 중복 피함). 기본: 허브에만 두고 단계 내부는 전환 바만.

## 5. 진입 흐름 재배선 (기획방 승격)

현재(App.tsx):
- 기획방은 홈에서만 진입: `onOpenPlanning`(기존 세션) / `onStartPlanning`(새 시작 → `openPlanningRoom` → `setPage("planning")`).
- `PlanningRoom`: `onBack → setPage("home")`, `onStartWork → setPage("code")`.

변경:
- 홈 허브 **기획 카드** → 기획 단계 진입(`setPage("planning")`; 세션 없으면 시작 안내/입력).
- 홈 **시작 입력** → 기존 `onStartPlanning` 그대로 → 기획 단계.
- `PlanningRoom.onStartWork → setPage("code")`(개발) = 단계 전환과 자연 일치. `onBack → "home"`(허브)도 유지.
- **네비게이션 셸만 교체**: 평평한 `nav-tabs`(App.tsx:291~333) → ⓐ 홈 허브 카드 + ⓑ 단계 전환 바 + ⓒ 단계별 서브탭. 라우팅 상태(`page`)·페이지 컴포넌트·핸들러는 보존.

## 6. 컴포넌트 영향 (경계와 책임)

| 단위 | 변경 | 책임 |
|---|---|---|
| `App.tsx` 네비 셸 | **교체** | `page`↔`stage` 매핑, 허브/전환바/서브탭 렌더 분기. 라우팅 상태는 그대로. |
| `Home.tsx` | **허브화** | 상태 스트립 + 시작 입력 + 카드 3장(뱃지). 기존 홈 내용 일부 재배치. |
| 신규: 단계 전환 바 | **추가(소형)** | `← 홈 / 기획·개발·유지보수` 버튼 줄. props: 현재 stage, onNavigate. |
| 신규: 단계별 서브탭 | **추가(소형)** | 단계에 속한 page 목록을 탭으로. 데이터는 §3 매핑. |
| 페이지 컴포넌트들 | **무변경** | PlanningRoom·CodeExplorer·DocsViewer·Doctor·BackupDashboard·ErrorLogs 그대로 재활용. |

큰 파일 경고: `App.tsx`는 단일 앵커 `APP:1-375`. 네비 셸 교체로 커지면, 전환 바·서브탭·허브 카드를 **별도 컴포넌트 파일로 분리**(App.tsx는 라우팅 분기만 얇게 유지).

## 7. 검증

- UI 구조 변경이라 단위테스트 대상 아님 → **빌드(tsc) + 시각 렌더 확인**(tauri dev).
- 회귀 기준: 모든 기존 `page` 가 새 IA에서 도달 가능(기획/코드/문서/진단/백업/에러로그/사용법/설정), 기획방 진입·`onStartWork`·`onBack` 동작 보존.

## 8. 미결정 / 후속(implementation plan에서)

- 구현 단계 분해(예: 셸 라우팅 → 전환 바 → 허브 카드 → 가이드 뱃지) 및 점진 출하 순서.
- 단계 내부 상태 스트립 중복 여부 최종 결정.
- 단계 전환 바·서브탭의 정확한 시각 스타일(A안 톤 계승).
- 빈 상태 처리: 기획 세션 없을 때 기획 카드/단계의 안내 문구.
