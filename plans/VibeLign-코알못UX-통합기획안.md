# VibeLign 코알못 UX 통합 기획안

작성일: 2026-05-31
목적: 기존 초보자 단순화 계획과 코알못 UX 개선안을 통합해, VibeLign을 "명령어 도구"가 아니라 "자동 안전 환경"으로 재정의한다.

---

## 1. 결론

VibeLign의 코알못 UX 방향은 다음으로 고정한다.

```text
사용자 → GUI 첫 화면에서 만들고 싶은 것 말하기
      → 기획방에서 LLM CLI들과 채팅으로 md 기획안 자동 생성
      → 심플한 Home에서 프로젝트 안전 상태 확인
      → 기존 patch/CodeSpeak/plan-structure 흐름은 초보 표면에서 제거
```

초보자는 `md`, `patch`, `anchor`, `watch`, `guard`, `CodeSpeak`, `MCP`, `rules` 같은 개념을 배우지 않아야 한다. 초보자가 알아야 할 것은 네 가지뿐이다.

1. **프로젝트 열기**
2. **만들고 싶은 것 말하기**
3. **문제 확인**
4. **되돌리기**

기존의 복잡한 CLI 중심 흐름은 고급 모드와 문서 reference로 남긴다. 단, `vib plan`은 예외다. GUI 기획방과 같은 엔진을 터미널에서도 호출할 수 있는 보조 진입점으로 다룬다.

---

## 2. 제품 판단

### 2-1. `patch`와 CodeSpeak는 실패한 사용자-facing 흐름으로 본다

`vib patch`는 초보자 핵심 경로에서 제거한다. 여기서 `patch`는 단일 CLI 명령만 뜻하지 않는다. **CodeSpeak 기반 patch 흐름 전체**를 포함한다.

퇴출 범위:

- `vib patch`
- GUI `PatchCard`
- CodeSpeak 생성/노출 UX
- 자연어 요청 → CodeSpeak → target file/anchor 제안 흐름
- patch suggester / patch targeting을 초보자 경로에 노출하는 문서와 도움말
- "이 패치 지시문을 AI에 복사해서 붙여넣으세요" 방식의 안내

이유:

- 자연어 요청을 안전한 수정 위치로 매핑하는 정확도가 충분하지 않다.
- CodeSpeak는 코알못에게 설명할수록 더 어려워지는 내부 형식이다.
- GUI Patch card는 이미 기본 노출에서 제거된 이력이 있다.
- 코알못은 `patch`/CodeSpeak 결과를 판단할 수 없기 때문에 잘못된 제안을 그대로 따라갈 위험이 크다.
- 현재 제품 방향은 "VibeLign이 수정안을 만드는 도구"가 아니라 "AI 작업을 안전하게 감싸는 환경"에 더 적합하다.

정책:

| 영역 | 결정 |
|---|---|
| GUI 초보 모드 | 완전 제거 |
| GUI 고급 모드 | 기본 숨김. 필요하면 나중에 "실험실"로 이동 |
| README / 온보딩 | patch/CodeSpeak 초보 경로에서 제거 |
| CLI | 즉시 삭제보다 deprecate 후 제거 |
| MCP 기반 host AI 수정 | 유지. Claude/Cursor/Codex가 CodeSpeak 없이 project map/anchor 도구를 직접 쓰는 방향 |

CLI 제거 단계:

1. `vib patch` 실행 시 deprecated 안내 출력
2. CodeSpeak 출력/복사 안내를 문서에서 제거하고, 고급/legacy로 격하
3. 한두 릴리즈 뒤 삭제 또는 hidden command 처리

## 3. 타겟 사용자

### 코알못 사용자의 전제

- 터미널 명령어를 외우지 못한다.
- `anchor`, `guard`, `MCP`, `hook`, `project map` 같은 말을 이해하지 못한다.
- "AI에게 시켰는데 망가졌을 때" 가장 큰 불안을 느낀다.
- 설정 화면, 카드, 로그가 많으면 바로 이탈한다.

### UX 목표

코알못에게 VibeLign은 다음처럼 느껴져야 한다.

```text
"만들고 싶은 것을 말하고 폴더를 고르면, AI 코딩 준비와 안전장치가 알아서 켜진다."
```

## 4. 핵심 UX 구조

### 4-0. 바이브코딩은 기획부터 어렵다

코알못에게 첫 번째 장벽은 코드를 고치는 순간이 아니라, AI에게 맡길 수 있는 형태로 생각을 정리하는 순간이다.

문제:

- 바이브코딩으로 무언가를 만들려면 먼저 기획이 필요하다.
- AI는 Markdown 기획안이 있으면 훨씬 잘 이해한다.
- 코알못은 Markdown이 무엇인지, `.md` 파일을 만들어야 하는지도 모른다.
- 기획안을 Claude, ChatGPT, Gemini 사이에서 왕복 검토시키는 방식은 효과적이지만, 초보자가 직접 반복하기 어렵다.

따라서 VibeLign에는 GUI 안전장치보다 앞선 "기획 시작" 흐름이 필요하다. 이 흐름은 CLI 중심으로 설계한다.

대표 명령:

```bash
vib plan "동네 카페 예약 앱을 만들고 싶어"
```

사용자 경험:

```text
무엇을 만들고 싶나요?
→ VibeLign이 쉬운 질문 3~5개를 한다.
→ 설계자 클로이, 검토자 지오, 탐색자 미나가 기획안을 서로 검토한다.
→ VibeLign이 하나의 Markdown 기획안으로 합친다.
→ plans/{이름}.md 에 저장한다.
```

중요한 원칙:

- 사용자는 Markdown을 몰라도 된다.
- VibeLign이 `.md` 파일을 만들었다고만 쉽게 알려준다.
- 각 AI는 모델명이 아니라 페르소나 이름으로 불러온다.
- 중간 검토는 채팅창 형식으로 보여주되, 최종 저장 파일은 VibeLign이 정리한다.
- 이 흐름은 CodeSpeak나 patch 지시문을 만들지 않는다.
- `vib plan-structure`는 새 기획방/초보 UI의 전제나 다음 단계로 두지 않는다. UI에서는 완전히 숨겨도 되며, 기존 내부 구현은 이번 범위에서 건드리지 않는다.

### 4-1. 온보딩 첫 화면은 ChatGPT 메인 입력 바 스타일이다

온보딩은 설명서가 아니라 실행 화면이어야 한다. 첫 화면의 시각적 중심은 ChatGPT의 메인 채팅 입력 바와 같은 **단일 가로형 입력 영역**이다. 사용자가 처음 보는 것은 카드 그리드도, 가이드 아코디언도, 터미널 시뮬레이션도 아니다. 빈 화면에 떠 있는 큰 채팅 입력 바 하나다.

#### 화면 레이아웃

```text
                내 프로젝트를 안전하게 만들기
                       (작게, 보조)


  ┌──────────────────────────────────────────────────────────────┐
  │  +    무엇을 만들고 싶나요?                    Instant ⌄   ● │
  └──────────────────────────────────────────────────────────────┘

                       고급 설정 보기
                          (아주 작게)
```

#### 입력 바 구성 요소

| 위치 | 역할 |
|---|---|
| `+` (왼쪽) | 프로젝트 폴더 선택 / 파일 첨부 |
| placeholder | "무엇을 만들고 싶나요?" |
| `Instant ⌄` (오른쪽) | 페르소나/모드 선택 (예: 클로이만, @모두, Codex만) |
| `●` (검은 원, 우측 끝) | 전송 / 시작 |

#### 기존 Onboarding.tsx에서 제거할 것

다음 카드는 첫 화면에서 **모두 제거한다**. "고급 설정 보기" 뒤에도 두지 않는다. 필요하면 별도 도움말/설정 페이지로 이동시킨다.

- **기능 카드 4개**: 코드맵 생성 · AI 폭주 방지 · 원클릭 복구 · AI 이동 자유 (`FEATURE_CARDS` 상수 및 렌더링)
- **터미널 시뮬레이터**: `TERMINAL_LINES_DEFAULT` 상수 및 애니메이션 블록
- **도움말 질문하기 카드**: 질문 입력 + 답하기 버튼 + 답변 블록
- **바이브라인 깃허브 카드**: GitHub URL · Threads 프로필 링크 · 선택 버튼
- **바이브라인 첫걸음 아코디언**: `ONBOARDING_GUIDE_STEPS` 7단계 가이드 및 페이지 도트
- **최근 프로젝트 자동 노출**: secondary 영역으로 강등 또는 "고급 설정 보기" 뒤로

#### 첫 화면에 유지할 것

- 입력 바 정중앙 (화면의 80% 가로 폭, 단일 라인 높이)
- 작은 제목 "내 프로젝트를 안전하게 만들기" (입력 바 위쪽, 보조 사이즈)
- "고급 설정 보기" 토글 (입력 바 아래쪽, 아주 작게)
- Claude Code 자동 설치 체크박스 (§4-1-1) — 입력 바 바로 아래 작은 라인 한 줄

#### 폴더 선택 동작

폴더는 입력 바의 `+` 버튼으로 선택한다. 사용자가 입력 바에 자연어를 먼저 치고 그 후에 폴더를 골라도 되고, 폴더 먼저 골라도 된다. 두 경우 모두 `●` 전송 시 비대화형 `vib start`가 자동으로 실행되고, 동시에 사용자의 자연어 입력이 기획방의 첫 메시지로 전달된다.

폴더 없이 텍스트만 전송하면 VibeLign이 후속으로 "어느 폴더에 만들까요?"를 채팅 응답으로 되묻는다. 입력 바 자체는 추가 모달 없이 작동한다.

#### 진행 메시지

폴더 선택 후 입력 바 **아래쪽 동일 위치**에 ChatGPT 응답처럼 한 줄씩 상태가 올라온다. 입력 바는 사라지지 않고 자리만 유지한다.

```text
프로젝트 확인 중...
안전 규칙 준비 중...
되돌리기 지점 저장 중...
파일 변경 감시 준비 중...
준비 완료
```

준비 완료 후 입력 바는 그대로 기획방의 첫 입력 위치가 된다 (§9-0-1 채팅 UI로 자연 전환).

#### 원칙 요약

- 첫 화면에서 기능 카드, 명령어 목록, 긴 설명, 가이드 아코디언, 터미널 시뮬레이션은 **모두 보여주지 않는다**.
- 첫 화면의 정보 밀도는 ChatGPT 첫 화면 수준 이하로 유지한다.
- 입력 바 외의 모든 요소(제목, 체크박스, 고급 토글)는 입력 바보다 작고 보조적이어야 한다.

### 4-1-1. Claude Code 자동 설치는 첫 화면에서 선택 가능해야 한다

코알못은 AI 코딩 도구 설치 자체도 막히기 쉽다. 따라서 Claude Code 자동 설치는 고급 설정 뒤에 숨기지 말고, 첫 화면에서 선택 가능한 옵션으로 둔다.

표시 방식:

```text
[ ] Claude Code도 자동으로 준비하기
```

보조 설명:

```text
처음 쓰는 분께 추천해요. 설치되어 있지 않으면 VibeLign이 안내에 따라 준비합니다.
```

동작 원칙:

- 기본값은 ON을 권장한다. 단, 이미 설치되어 있으면 "이미 준비됨"으로 표시한다.
- 설치가 필요하면 프로젝트 안전장치 준비 흐름 안에서 함께 실행한다.
- 설치 실패는 전체 온보딩 실패로 처리하지 않는다.
- 실패 시 "Claude Code 준비만 실패했어요. 프로젝트 안전장치는 켜져 있어요."처럼 분리해서 보여준다.
- 사용자가 선택 해제하면 VibeLign 안전장치만 준비한다.

첫 화면 정보 구조:

```text
내 프로젝트를 안전하게 만들기

[프로젝트 폴더 선택]

[x] Claude Code도 자동으로 준비하기

고급 설정 보기
```

### 4-1-2. 사용자는 "쓰는 AI"를 선택만 하고, 연결은 VibeLign이 한다

코알못에게 `claude login` / `codex login` 같은 터미널 로그인을 요구하면 안 된다. 대부분의 사용자는 이미 Claude, ChatGPT, Gemini 같은 LLM을 한두 개 쓰고 있다. 그래서 온보딩에서 사용자는 **"어떤 AI를 쓰세요?"에 선택만** 하면 되고, VibeLign이 그 선택을 해당 공식 CLI로 매핑해 설치/연결을 자동 처리한다.

원칙:

- 사용자 행동은 "쓰는 AI 선택" 하나로 끝난다. 사용자는 CLI 이름(`claude`/`codex`/`agy`)이나 로그인 명령을 몰라도 된다.
- 선택 → VibeLign이 공식 CLI 감지 → 미설치면 설치 안내/실행 → 미인증이면 사용자가 공식 CLI에서 직접 로그인하도록 안내. VibeLign은 인증/토큰을 직접 다루지 않고, 인증된 공식 CLI를 relay만 한다. (§4-1-1 Claude Code 자동 설치 backend 패턴을 일반화해 재사용)
- VibeLign은 토큰/세션 파일을 읽지 않는다(§9-0 경계 유지). "연결"은 공식 CLI의 로그인 흐름을 실행하는 것이지 자격증명을 가로채는 것이 아니다.
- 선택한 AI = 등장할 페르소나. 예: Claude → 설계자 클로이, ChatGPT(Codex) → 검토자 지오, Gemini(Antigravity) → 탐색자 미나.
- §4-1-1의 "Claude Code 자동 준비"는 이 선택의 **Claude 케이스**다. 별도 기능이 아니라 같은 선택→연결 흐름의 한 항목으로 본다.
- 연결은 막지 않는다(best-effort). 선택했지만 아직 연결 안 된 AI가 있어도 template 또는 연결된 다른 AI로 진행한다(§9-0 fallback).
- 로그인 자체는 브라우저 OAuth 등 공식 CLI의 대화형 단계를 거칠 수 있다. "사용자는 선택만"이 UX 목표지만, VibeLign은 그 인증을 수행·열람하지 않고 사용자가 공식 CLI에서 끝내도록 안내만 한다. 이 자동화 안정성은 §15의 최대 리스크에 포함된다.

온보딩 배치 (§4-1 입력 바와 통합):

```text
내 프로젝트를 안전하게 만들기

[ + ]  무엇을 만들고 싶나요?              [ 쓰는 AI 선택 ⌄ ]  [ ● ]

쓰는 AI: Claude · ChatGPT   (선택하면 자동 연결)
고급 설정 보기
```

§4-1의 오른쪽 `Instant ⌄` 드롭다운이 이 "쓰는 AI 선택"을 겸한다. 사용자는 쓰는 AI를 고르고, 바로 입력 바에 만들고 싶은 것을 적어 전송하면 별도 설정 화면 없이 기획방 채팅으로 이어진다.

### 4-2. 초보 Home은 기능 카드 그리드가 아니라 상태 대시보드다

첫 설치와 첫 실행은 초보 모드가 기본값이다. 초보 Home은 기능 목록을 보여주는 런처가 아니라, 사용자가 지금 안전한지와 다음에 무엇을 해야 하는지만 알려주는 상태 화면이어야 한다.

현재 Home의 문제:

- 기능 카드가 너무 많다.
- 사용자가 무엇을 눌러야 할지 판단해야 한다.
- `Anchor`, `Guard`, `Watch`, `Memory`, `Recovery`, `Export`, `Config` 같은 내부 기능명이 한 번에 노출된다.
- 온보딩을 단순하게 만들어도 Home에서 다시 복잡해진다.

초보 Home에는 3개 블록만 둔다.

| 카드 | 역할 | 사용자 문구 |
|---|---|---|
| 프로젝트 안전 상태 | start/watch/guard/anchor/Claude Code 상태 통합 | "안전장치가 켜져 있어요" |
| 지금 할 일 | doctor/guard 결과 기반 최우선 안내 1개 | "지금 고칠 일 1개" 또는 "바로 AI 코딩해도 괜찮아요" |
| 되돌리기 | checkpoint/history/undo 통합 | "이전 상태로 돌아가기" |

권장 화면 구조:

```text
프로젝트 안전 상태

안전장치가 켜져 있어요
마지막 확인: 방금 전

[문제 확인하기]

최근 저장 지점
5분 전 · 자동 저장됨

[되돌리기]

고급 기능 보기
```

원칙:

- 기능별 카드는 초보 Home에 직접 노출하지 않는다.
- 초보 Home은 "다음 행동 1개"만 추천한다.
- 상세 기능은 펼치기 또는 고급 모드 뒤에 둔다.
- `PatchCard`는 초보 Home에서 완전히 제거한다.
- 고급 모드에서도 patch는 legacy/experimental로 표시한다.

하단에만 작게 표시한다.

```text
고급 기능 보기
```

### 4-3. 고급 모드

고급 모드는 기존 기능을 보존한다.

포함:

- doctor 상세
- guard 상세
- anchor 보기/수동 갱신
- checkpoint/history 상세
- recovery/memory
- MCP/export/config
- docs/code explorer
- legacy/experimental patch

고급 모드는 설정에서 켠다. 초보자에게 먼저 노출하지 않는다.

## 5. `vib start`의 의미 변경

현재 `vib start`는 "초기 설정 명령"에 가깝다.

앞으로는 다음 의미로 바꾼다.

```text
vib start = 이 프로젝트의 자동 안전장치를 켠다
```

### GUI 흐름

```text
GUI 실행
→ 프로젝트 폴더 선택
→ GUI가 비대화형 start 실행
→ 자동 안전장치 준비
→ 첫 체크포인트 저장
→ 초보 Home 표시
```

### 자동 준비 항목

| 항목 | 자동화 |
|---|---|
| 규칙 파일 | 자동 생성/갱신 |
| `.vibelign/` 상태 | 자동 생성 |
| project map | 자동 생성 |
| anchor 상태 | 자동 생성 또는 자동 점검 |
| watch | 자동 시작 |
| guard | 자동 연결 |
| checkpoint | 첫 시작 시 자동 생성 |
| pre-commit safety | Git 프로젝트면 자동 연결 |

### 중요한 원칙

자동화는 조용히 실패하면 안 된다.

실패 시 GUI는 반드시 이렇게 보여준다.

```text
자동 안전장치 일부가 꺼져 있어요.
[다시 켜기] [자세히 보기]
```

## 6. 자동 watch / anchor / guard 설계

### 6-1. Watch

watch는 초보자가 직접 켜는 기능이 아니라 프로젝트를 열면 자동으로 켜지는 상태 서비스가 된다.

동작:

- GUI 프로젝트 열기 후 자동 시작
- 파일 변경 감지
- project map 갱신
- anchor stale 여부 확인
- guard 필요 여부 표시

UI:

```text
안전 감시 중
마지막 확인: 방금 전
```

실패 UI:

```text
파일 변경 감시가 꺼졌어요.
[다시 켜기]
```

### 6-2. Anchor

사용자에게 `anchor`라는 단어를 먼저 보여주지 않는다.

초보 모드 용어:

```text
코드 북마크
```

자동화 정책:

- project map 갱신은 항상 자동
- anchor 상태 점검은 항상 자동
- anchor 삽입/수정은 체크포인트 이후에만 자동
- 위험하거나 애매한 경우 자동 변경하지 말고 "코드 북마크를 다시 정리할까요?"로 표시

이유:

- anchor 자동 삽입은 실제 소스 파일을 바꾼다.
- 코알못 UX에서는 자동이 맞지만, 자동 변경은 반드시 되돌릴 수 있어야 한다.

### 6-3. Guard

guard는 사용자가 직접 실행하는 명령이 아니라 안전 상태의 일부가 된다.

실행 시점:

- AI 작업 이후 hook에서 자동 실행
- commit 전 자동 실행
- GUI에서 "문제 확인" 카드 열 때 실행

초보 UI:

```text
안전 검사 통과
```

또는

```text
위험할 수 있는 변경이 있어요.
[무엇이 문제인지 보기] [되돌리기]
```

## 7. 에러 메시지 인간화

에러는 로그가 아니라 가이드여야 한다.

우선순위 1차 대상:

| 현재 | 초보 문구 |
|---|---|
| `ERR_STRICT_SEARCH_NOT_UNIQUE` | 같은 코드가 여러 곳에 있어서 정확한 위치를 못 찾았어요. |
| `ERR_ANCHOR_NOT_FOUND` | 코드 북마크를 찾을 수 없어요. 파일이 많이 바뀌었을 수 있어요. |
| `ERR_FILE_TOO_LARGE` | 이 파일이 너무 커졌어요. 나눠서 관리하는 게 좋아요. |
| Guard blocked | 이 변경은 위험할 수 있어요. 먼저 확인해 주세요. |
| AI CLI not logged in | AI CLI에 아직 로그인되지 않았어요. 기본 템플릿으로 기획안은 만들 수 있어요. |
| watch start failed | 파일 변경 감시를 켜지 못했어요. 다시 시도할 수 있어요. |

초보 UI에서는 원문 에러 코드를 기본적으로 숨긴다.

```text
문제 설명
원인
지금 할 수 있는 일
고급 정보 펼치기
```

## 9. 코알못이 바이브코딩을 쉽게 하기 위한 추가 아이디어

이 섹션은 VibeLign을 "안전장치 설정 앱"에서 한 단계 더 나아가, 코알못이 실제로 바이브코딩을 시작하고 반복할 수 있게 만드는 제품 아이디어다.

### 9-0. 멀티 AI 기획 도우미

AI 작업실보다 먼저 필요한 것은 "무엇을 만들지 정리하는 방"이다.

CLI 흐름:

```bash
vib plan "내 가게 예약을 받는 웹앱을 만들고 싶어"
```

기본 페르소나:

| 페르소나 | 연결 CLI | 역할 |
|---|---|
| 설계자 클로이 | Claude Code CLI | 사용자의 흐릿한 아이디어를 차분한 제품 기획안 초안으로 구조화 |
| 검토자 지오 | Codex CLI | 빠진 요구사항, 모순, 구현 가능성, 테스트 기준 검토 |
| 탐색자 미나 | Antigravity CLI | 대안 흐름, MVP 축소안, 범위 조절, 리스크 검토 |
| VibeLign | 내부 정리기 | 세 응답을 합쳐 최종 Markdown 기획안 생성 |

채팅창 호출 방식:

```text
@클로이 초안 만들어줘
@지오 빠진 점 검토해줘
@미나 더 쉬운 MVP로 줄여줘
@모두 이 기획안 검토해줘
```

중요한 정책:

- VibeLign은 **인증 무관 relay**다 — 공식 CLI 바이너리를 subprocess로 실행해 I/O만 중계하고, OAuth/토큰을 수행·보관·열람하지 않는다(토큰 추출이 밴 사례의 원인 — 멀티AI 설계안 §13-7). CLI 인증(OAuth/API 키)은 사용자 본인의 터미널 설정이고 VibeLign이 고르지 않는다. 구독-tier 집행이 걱정되는 사용자가 자기 CLI를 API 키로 구성하는 건 자유지만, 그건 VibeLign이 제공하는 경로가 아니다.
- 사용자는 온보딩에서 쓰는 AI를 선택만 하고, VibeLign이 그 선택을 공식 CLI로 매핑해 설치/로그인 흐름을 대신 띄운다 (§4-1-2). 사용자는 CLI 이름이나 `claude login` 같은 명령을 몰라도 된다.
- VibeLign은 로그인 토큰이나 세션 파일을 읽지 않는다. "연결"은 공식 CLI의 로그인 흐름을 실행하는 것이지 자격증명을 가로채는 것이 아니다.
- VibeLign은 `claude`, `codex`, `antigravity` 같은 공식 CLI를 실행하고 stdout/stderr 결과만 수집한다.
- 각 LLM 응답은 VibeLign이 발급한 `session_id`, `turn_id`, `run_id`, `persona_id`, `cli_id`로 관리한다.
- LLM이 자체 ID를 응답에 포함하더라도, 신뢰 기준은 VibeLign metadata다.

저장 예시:

```text
.vibelign/planning/plan_20260531_abc123/session.json
.vibelign/planning/plan_20260531_abc123/turns/turn_001_claude.md
.vibelign/planning/plan_20260531_abc123/turns/turn_002_codex.md
.vibelign/planning/plan_20260531_abc123/turns/turn_003_antigravity.md
.vibelign/planning/plan_20260531_abc123/final.md
```

생성 결과:

```text
plans/cafe-reservation-plan.md
```

사용자에게 보여줄 말:

```text
기획안을 만들었어요.
이 파일을 AI에게 보여주면 더 정확하게 작업을 시작할 수 있어요.
```

이 기능은 초보자에게 Markdown을 가르치는 기능이 아니다. Markdown을 몰라도 AI가 이해할 수 있는 기획 문서를 자동으로 만들어주는 기능이다.

락인 효과:

- 사용자의 기획안이 `plans/*.md`에 계속 쌓인다.
- 클로이/지오/미나의 역할 분담이 사용자의 기본 기획 습관이 된다.
- Claude Code, Codex, Antigravity를 오가며 복사/붙여넣기 하던 루프가 VibeLign 안으로 들어온다.
- 기획안이 이후 GUI 안전장치와 AI 작업실로 이어진다.
- VibeLign은 특정 모델을 소유하지 않아도 "구독한 AI CLI들을 한 팀처럼 부르는 기획 작업실"이 된다.

핵심은 모델 락인이 아니라 워크플로우 락인이다.

### 9-0-1. 페르소나 채팅 UI는 제품 핵심이다

멀티 AI 기획 도우미의 성패는 모델 호출보다 채팅 UI에 달려 있다. 초보자는 "Claude, Codex, Antigravity를 순서대로 실행한다"를 이해하고 싶어 하지 않는다. 대신 "기획방에 클로이, 지오, 미나를 불러서 같이 검토한다"처럼 보여야 한다.

채팅 UI 원칙:

- 모델명보다 페르소나 이름을 먼저 보여준다.
- 한 화면에서 `나`, `설계자 클로이`, `검토자 지오`, `탐색자 미나`, `VibeLign 정리`가 대화하는 구조로 보여준다.
- 각 메시지에는 역할 배지와 연결 CLI 상태를 작게 표시한다.
- 사용자가 직접 부르는 명령은 `@클로이`, `@지오`, `@미나`, `@모두` 네 개로 제한한다.
- 최종 산출물은 채팅 아래에 "기획안 저장됨"으로 표시하고, 원문 Markdown 문법은 기본적으로 숨긴다.

기획방 시각 스타일:

- 레퍼런스 repo: <https://github.com/unclejobs-ai/satgat> (repo 외부 자산. 구현 시 디자인 토큰·채팅 화면 스크린샷을 `vibelign-gui` 안에 박제하거나 링크를 코드 주석에 남겨, 구현자가 출처를 잃지 않게 한다.)
- 채팅방은 `satgat`의 한국형 문서 톤을 참고한다. 백자지 계열 배경, 먹색 본문, 단청색 포인트, 명조/돋움 대비를 기본 감성으로 삼는다.
- 그대로 복제하지 않는다. VibeLign의 화면은 문서 샘플 갤러리가 아니라 초보자가 입력하고, 페르소나가 검토하고, 기획안이 저장되는 작업실이어야 한다.
- 메시지는 둥근 말풍선보다 조밀한 업무용 타임라인에 가깝게 둔다. 종이 위에 검토 기록이 쌓이는 느낌을 주되, 각 턴의 주체와 상태는 즉시 구분되어야 한다.
- 단청색은 전송, 중요한 경고, 최종 행동 버튼처럼 의미 있는 액션에만 쓴다. 배경과 메시지 전체를 한 색 계열로 덮지 않는다.
- 큰 여백, 한자 장식, 인쇄물 장식, 과한 카드 중첩은 피한다. 결과물인 기획안과 다음 행동이 먼저 읽혀야 한다.

권장 화면:

```text
기획방

나
카페 예약 앱을 만들고 싶어

설계자 클로이 · Claude Code
예약 앱의 목적과 사용자 흐름을 먼저 정리할게요.

검토자 지오 · Codex
예약 취소 정책과 관리자 권한이 아직 정해지지 않았어요.

탐색자 미나 · Antigravity
처음에는 예약 신청과 승인만 MVP로 두는 게 좋아요.

VibeLign 정리
기획안을 만들었어요.
[기획안 보기] [AI 작업 시작]
```

OpenDesign 활용 판단:

- OpenDesign는 Claude Design의 유료 대안으로 참고할 수 있는 오픈 디자인 자산으로 본다.
- 1차 구현에서 OpenDesign을 런타임 UI 라이브러리처럼 의존하지 않는다.
- 대신 `satgat` 스타일을 1차 채팅방 레퍼런스로 삼고, `DESIGN.md` 스타일 가이드, 디자인 토큰, 채팅 화면 레퍼런스, anti-pattern 체크리스트는 보조 기준으로 사용한다.
- 기존 GUI가 React/Vite/Tauri 기반이므로, 실제 컴포넌트는 VibeLign 내부 React 컴포넌트로 만든다.

### 9-0-2. 사용자가 자기 페르소나 캐릭터를 만들 수 있게 한다

기본 캐릭터를 repo에 넣는 것보다 한 단계 더 나아가, 사용자가 "내 AI 팀"의 캐릭터를 쉽게 꾸미게 하면 락인 효과가 더 강해진다. 사용자는 단순히 VibeLign 기능을 쓰는 것이 아니라, 자신의 기획 동료를 만든다고 느끼게 된다.

제품 판단:

- 매우 좋은 방향이다.
- 다만 첫 버전부터 매번 AI 이미지 생성을 실행하면 비용, 실패, 품질 흔들림이 커진다.
- 그래서 초보 모드는 "템플릿 커스터마이즈"부터 제공하고, 고급 모드에서 sprite-gen 기반 생성/큐레이션을 제공한다.

초보 모드:

```text
내 AI 팀 꾸미기

설계자 이름: 클로이
색상: 민트
분위기: 차분함
아이콘: 노트

[캐릭터 만들기]
```

초보 모드에서는 이미 준비된 base sprite set을 조합한다.

- 이름 변경
- 색상 변경
- 역할 배지 변경
- idle/thinking/writing/done 상태 선택
- 3~5개 스타일 preset 선택

고급 모드:

- sprite-gen으로 새 캐릭터 run 생성
- curation webview로 frame 검수
- 승인된 sprite atlas와 manifest만 VibeLign asset으로 등록
- 원본 생성 로그와 중간 산출물은 project cache에 보관

저장 위치:

```text
.vibelign/personas/{persona_id}/persona.json
.vibelign/personas/{persona_id}/sprite-sheet-alpha.png
.vibelign/personas/{persona_id}/manifest.json
```

repo에 들어갈 기본 캐릭터:

```text
vibelign-gui/src/assets/personas/default/chloe/
vibelign-gui/src/assets/personas/default/gio/
vibelign-gui/src/assets/personas/default/mina/
```

원칙:

- 앱 실행 때마다 캐릭터를 생성하지 않는다.
- 생성은 사용자가 명시적으로 "캐릭터 만들기"를 눌렀을 때만 한다.
- 실패해도 기획 기능은 계속 동작해야 한다.
- 기본 캐릭터가 항상 fallback으로 존재해야 한다.
- 캐릭터는 32~48px 상태 아바타로 쓰고, 기획 내용보다 튀지 않게 한다.

락인 효과:

- 사용자가 만든 페르소나 이름과 캐릭터가 쌓인다.
- 프로젝트마다 다른 AI 팀을 만들 수 있다.
- 기획 대화가 "내 팀과 만든 산출물"처럼 느껴진다.
- VibeLign을 떠나면 모델은 그대로 있어도 이 팀 구성과 기록은 사라진다.

피해야 할 것:

- 기능 카드 그리드로 페르소나를 보여주기
- 모델 선택 드롭다운을 첫 화면에 노출하기
- 모든 CLI 로그를 채팅에 그대로 흘리기
- Markdown 원문 편집기를 먼저 보여주기
- 유료 디자인 도구가 있어야만 구현 가능한 구조로 만들기
- 캐릭터 생성 실패 때문에 기획 생성을 막기
- 큰 애니메이션으로 채팅 내용을 가리기

### 9-1. AI 작업실

초보자는 Claude Code, Cursor, Codex를 어떻게 시작해야 하는지도 모른다. VibeLign 안에 "AI 작업실"이라는 단일 진입점을 둔다.

화면 구조:

```text
무엇을 만들고 싶나요?

[예: 로그인 화면을 만들고 싶어요]

[AI에게 안전하게 맡기기]
```

동작:

1. 사용자가 자연어로 원하는 작업을 적는다.
2. VibeLign이 작업 전 checkpoint를 자동 생성한다.
3. VibeLign이 프로젝트 구조 요약과 안전 규칙을 host AI에 전달한다.
4. host AI가 직접 MCP/project map/anchor 도구를 읽고 수정한다.
5. VibeLign이 guard와 변경 요약을 자동 실행한다.
6. 사용자는 "괜찮아요" 또는 "되돌리기"만 선택한다.

중요한 점:

- CodeSpeak를 만들거나 보여주지 않는다.
- patch 지시문을 복사하게 하지 않는다.
- VibeLign은 "수정안 생성기"가 아니라 "AI 작업 세션 관리자"가 된다.

### 9-2. 첫 성공 경험 가이드

코알못에게는 첫 성공이 중요하다. 첫 실행 후 바로 큰 기능을 만들게 하지 말고, 작고 안전한 첫 작업을 제안한다.

예시:

```text
처음에는 작은 변경부터 해볼까요?

1. 버튼 문구 바꾸기
2. 제목 색 바꾸기
3. 설명 문장 추가하기
```

동작:

- VibeLign이 안전한 작은 변경 후보를 추천한다.
- 사용자가 하나를 고르면 checkpoint를 만든다.
- host AI에 작은 작업으로 넘긴다.
- 변경 후 바로 guard와 되돌리기 버튼을 보여준다.

목표:

```text
"AI가 바꿔도 되돌릴 수 있구나"를 첫 5분 안에 경험하게 한다.
```

### 9-3. "망했어요" 버튼

코알못은 문제가 생겼을 때 어떤 명령을 실행해야 하는지 모른다. Home에 항상 "망했어요" 또는 더 부드러운 "이전으로 돌아가기" 버튼을 둔다.

권장 문구:

```text
문제가 생겼나요?
[이전 안전한 상태로 돌아가기]
```

동작:

- 최근 checkpoint 목록을 쉬운 말로 보여준다.
- 기본 추천은 "AI 작업 전 상태"다.
- 되돌리기 전에 바뀔 파일 수만 간단히 보여준다.
- 고급 정보에는 실제 diff를 제공한다.

### 9-4. 쉬운 변경 요약

AI가 코드를 바꾼 뒤 초보자에게 raw diff를 먼저 보여주면 안 된다.

초보 요약:

```text
AI가 한 일

- 로그인 화면 파일을 바꿨어요.
- 버튼 문구를 "시작하기"로 바꿨어요.
- 위험해 보이는 변경은 없어요.

[괜찮아요] [되돌리기] [자세히 보기]
```

원칙:

- 파일명과 줄 번호는 기본 숨김
- 바뀐 목적과 위험도 먼저 표시
- 자세히 보기에서만 diff 표시

### 9-5. 프로젝트 상태를 사람 말로 설명

Code Explorer나 project map은 고급 기능이다. 초보자에게는 "내 프로젝트가 어떤 상태인지"를 문장으로 보여준다.

예시:

```text
이 프로젝트는 React 웹앱으로 보여요.
화면 파일은 src/pages 쪽에 있고, 테스트 파일도 있어요.
지금은 되돌리기 지점이 준비되어 있어요.
```

활용:

- 첫 온보딩 완료 화면
- Home의 프로젝트 안전 상태 카드
- AI 작업실에서 작업 시작 전 요약

### 9-6. 초보자용 질문 템플릿

코알못은 AI에게 어떻게 말해야 하는지도 모른다. VibeLign이 안전한 요청 템플릿을 제공한다.

템플릿 예시:

```text
화면을 바꾸고 싶어요
버그를 고치고 싶어요
글자를 바꾸고 싶어요
새 페이지를 만들고 싶어요
```

각 템플릿은 자연어 질문을 돕는 용도다. CodeSpeak나 patch 지시문으로 변환하지 않는다.

예시:

```text
어느 화면을 바꾸고 싶나요?
무엇이 어떻게 보이면 좋겠나요?
바뀌면 안 되는 부분이 있나요?
```

### 9-7. 자동 안전 리포트

AI 작업이 끝나면 VibeLign이 자동으로 안전 리포트를 보여준다.

리포트 구조:

```text
작업 결과

상태: 안전해 보여요
되돌리기: 가능
바뀐 파일: 2개
다음 추천: 직접 실행해서 화면을 확인해 보세요
```

상태는 3단계만 사용한다.

| 상태 | 의미 |
|---|---|
| 안전해 보여요 | guard 통과, 위험 신호 없음 |
| 확인이 필요해요 | guard warning 또는 큰 파일 변경 |
| 되돌리는 걸 추천해요 | guard fail 또는 위험 변경 |

### 9-8. "고급 말 번역기"

VibeLign 내부나 AI 응답에서 기술 용어가 나오면 쉬운 말로 바꿔 보여준다.

예시:

| 기술 용어 | 쉬운 말 |
|---|---|
| pre-commit hook | 커밋 전에 자동으로 확인하는 장치 |
| MCP | AI가 프로젝트 정보를 읽는 연결 통로 |
| anchor | 코드 북마크 |
| diff | 바뀐 내용 |
| guard failed | 위험한 변경이 있어요 |

이 기능은 별도 카드가 아니라, tooltip/도움말/에러 메시지에서 공통으로 사용한다.

### 9-9. Claude Code 준비 상태를 제품 핵심 상태에 포함

코알못에게 Claude Code 설치는 부가 기능이 아니라 AI 코딩 시작 조건에 가깝다.

Home의 프로젝트 안전 상태에 포함한다.

```text
Claude Code: 준비됨
```

또는

```text
Claude Code: 아직 준비되지 않았어요
[자동으로 준비하기]
```

설치 실패 시:

```text
Claude Code 준비만 실패했어요.
프로젝트 안전장치는 켜져 있어요.
```

### 9-10. 사용자가 배울 필요 없는 자동 루프

최종 목표는 사용자가 다음 루프를 명령어 없이 경험하는 것이다.

```text
작업 시작
→ 자동 checkpoint
→ AI 수정
→ 자동 guard
→ 쉬운 변경 요약
→ 괜찮으면 저장
→ 아니면 되돌리기
```

이 루프가 VibeLign의 핵심 제품 경험이다.

## 10. 통합 로드맵

### Phase 1 — 온보딩 첫 화면 수정

목표: 사용자가 처음 보는 화면을 ChatGPT식 입력 바 중심으로 바꿔 제품 인상을 즉시 바꾼다.

작업:

- 온보딩 첫 화면을 단일 입력 바로 재설계
- 기존 기능 카드, 터미널 시뮬레이터, 도움말 카드, GitHub 카드, 첫걸음 아코디언 제거
- 폴더 선택과 자연어 입력을 같은 입력 바에서 처리
- Claude Code 자동 준비 체크박스는 입력 바 아래 보조 라인으로 유지
- 최근 프로젝트와 고급 설정은 입력 바보다 뒤로 강등

완료 기준:

- 첫 화면의 주인공은 입력 바 하나다.
- 사용자는 첫 화면에서 patch/CodeSpeak/plan-structure/anchor/watch/guard를 보지 않는다.
- 폴더 선택과 텍스트 입력이 이후 기획방 흐름으로 이어질 수 있는 상태가 된다.

### Phase 2 — plan 기능: LLM CLI 채팅 기획방

목표: 각 LLM을 공식 CLI로 연결해 채팅으로 기획안을 만들고 `plans/*.md`로 저장한다.

작업:

- 기획방 UI 골격과 `vib plan --template-only` 엔진을 먼저 만든다
- 쉬운 질문 3~5개로 요구사항을 수집한다
- Claude Code/Codex/Antigravity CLI 실행기와 페르소나 역할을 분리한다
- CLI 미설치/미로그인/timeout은 전체 실패가 아니라 fallback으로 처리한다
- 최종 Markdown 기획안을 `plans/{slug}.md`에 저장한다
- `plan-structure`를 기획방/초보 UI/help/manual의 추천 경로에서 숨긴다

완료 기준:

- 사용자가 Markdown을 몰라도 AI에게 전달 가능한 `.md` 기획안이 생성된다.
- 준비된 CLI가 하나만 있어도 동작하고, 준비된 CLI가 없어도 템플릿 기획안은 생성된다.
- CodeSpeak나 patch 지시문을 생성하지 않는다.

### Phase 3 — 기존 Home 화면 정리

목표: 기존 Home을 기능 카드 런처가 아니라 심플한 안전 상태 화면으로 바꾼다.

작업:

- 초보 모드 기본 ON
- Home을 `프로젝트 상태`, `다음 할 일`, `되돌리기`로 축소
- 고급 기능 보기 토글 추가
- 기존 Home 카드들은 고급 모드로 이동
- GUI 온보딩 문구를 "폴더 선택하면 끝"으로 변경

완료 기준:

- 첫 실행 사용자는 3개 이상의 주요 카드에 노출되지 않는다.
- 고급 모드 전환 시 기존 기능 접근 가능하다.

### Phase 4 — 기존 기능 제거 준비와 legacy 숨김

목표: `patch`, `CodeSpeak`, `plan-structure`를 초보 제품 표면에서 제거하고, 실제 삭제는 의존성 정리 후 단계적으로 진행한다.

작업:

- GUI 초보 모드와 주요 도움말에서 `patch`, `CodeSpeak`, `plan-structure` 제거
- README/온보딩/도움말에서 patch 첫 경로 제거
- CLI `vib patch`와 `plan-structure`에 deprecated 또는 legacy 안내 추가
- MCP 기반 host AI 수정 경로는 유지하되 초보 문구에서는 CodeSpeak를 노출하지 않는다
- 내부 의존성, 테스트, 문서 참조를 점검한 뒤 삭제/hidden command 여부를 결정한다

완료 기준:

- 초보자는 첫 화면, Home, 주요 문서에서 `patch`, `CodeSpeak`, `plan-structure`를 보지 않는다.
- 기존 고급 사용자는 고급/legacy 경로에서 필요한 기능을 찾을 수 있다.
- 실제 삭제 전에 깨지는 내부 의존성이 목록화되어 있다.

### Phase 5 — `vib start` 자동 안전장치화

목표: GUI에서 폴더를 선택하면 안전장치가 자동 준비된다.

작업:

- GUI 프로젝트 선택 핸들러에서 비대화형 start 실행
- `vib start --auto` 또는 동등한 내부 API 추가
- 첫 체크포인트 자동 생성
- project map 자동 생성
- Git 프로젝트면 pre-commit safety 자동 연결
- start 실패/부분 성공 상태를 GUI에서 표시

완료 기준:

- 사용자가 터미널을 열지 않아도 프로젝트가 VibeLign 준비 상태가 된다.
- 실패 시 "다시 켜기"가 가능하다.

### Phase 6 — 자동 watch / anchor / guard

목표: 안전장치를 사용자가 실행하지 않아도 계속 작동하게 한다.

작업:

- 프로젝트 열기 후 watch 자동 시작
- 파일 변경 시 project map 자동 갱신
- anchor stale 감지
- 안전한 경우 anchor 자동 정리
- 애매한 경우 "코드 북마크를 다시 정리할까요?" 안내
- guard를 commit 전 / AI 작업 후 / 문제 카드 진입 시 자동 실행

완료 기준:

- 초보자가 `watch`, `anchor`, `guard` 명령을 몰라도 안전 상태를 볼 수 있다.
- 자동 소스 변경은 항상 checkpoint 이후에만 발생한다.

### Phase 7 — 에러/문제 해결 가이드

목표: 오류를 사용자가 이해할 수 있는 행동으로 바꾼다.

작업:

- 상위 에러 5~7개 인간화
- "최근 문제" 또는 "다음 할 일" 카드 추가
- doctor/guard 결과에서 최우선 이슈 1개를 뽑아 표시
- "자세히 보기"에는 개발자용 원문/로그 제공

완료 기준:

- 초보 UI에 raw error code가 바로 노출되지 않는다.
- 모든 문제 카드에 최소 하나의 행동 버튼이 있다.

### Phase 8 — 점진적 공개

목표: 기능을 처음부터 모두 보여주지 않는다.

작업:

- 초보 / 중급 / 고급 모드 정의
- 설정에서 수동 전환
- 사용 횟수 기반 제안은 후순위로 둔다
- 중급 모드에 코드 북마크 보기, 저장 포인트 관리, 안전 검사 상세 추가
- 고급 모드에 CLI/MCP/rules/export/customization 추가

완료 기준:

- 초보 모드는 단순하고, 고급 사용자는 기존 기능을 잃지 않는다.

## 11. 구현 우선순위

가장 먼저 해야 할 일은 사용자가 처음 보는 화면을 ChatGPT 메인 입력 바 한 줄로 만드는 것이다. 그 입력 바가 곧 기획방의 입구이고, 사용자 표면 정리·엔진·LLM 연결은 그 다음에 한다. CLI `vib plan`은 사용자 표면이 아니라 엔진을 호출하는 부산물로 다룬다.

권장 순서:

1. **온보딩 첫 화면 = ChatGPT 입력 바** — §4-1 적용, 기존 카드(코드맵 생성·AI 폭주 방지·원클릭 복구·AI 이동 자유·도움말 질문하기·바이브라인 깃허브·바이브라인 첫걸음 아코디언)와 터미널 시뮬레이터 전부 제거
2. **plan 기능 = LLM CLI 채팅 기획방** — template-only → CLI 1개 → 3 페르소나 best-effort 순서로 붙이고 `plans/*.md` 저장
3. **기존 Home 화면 정리** — 프로젝트 안전 상태, 지금 할 일, 되돌리기 3개 중심으로 축소하고 기존 기능은 고급 뒤로 이동
4. **기존 기능 제거 준비** — `patch`, `CodeSpeak`, `plan-structure`를 초보 표면에서 제거하고 legacy/deprecated 경로로 격하
5. **자동 start/watch/anchor/guard 정밀화** — 자동 안전장치를 기본값으로 만들되 자동 소스 변경은 checkpoint 이후로 제한
6. **에러 인간화 + 점진적 공개 + 페르소나 커스터마이즈** — 제품 안정화 이후 락인 강화

이 순서가 안전한 이유:

- 사용자가 처음 보는 화면이 곧 제품의 정체성이다. 입력 바부터 정리하지 않으면 그 뒤의 모든 작업이 카드 그리드에 가려진다.
- 엔진을 LLM 없이 먼저 만들면(PR 3) 흐름 버그와 LLM 응답 변동성 디버깅이 분리된다.
- LLM은 한 번에 한 페르소나씩 붙인다(PR 4 → PR 5). 한 번에 셋을 연결하면 어느 CLI가 문제인지 격리 불가능.
- Home 정리는 plan 기능 이후가 맞다. 온보딩과 기획방이 먼저 정해져야 Home이 보여줄 다음 행동이 명확해진다.
- 기존 기능 제거는 실제 삭제보다 초보 표면 제거가 먼저다. 내부 의존성과 고급 사용자 경로를 확인하지 않고 삭제하면 회귀가 크다.
- 자동 watch/anchor/guard는 잘못 만들면 신뢰 손실이 크므로 checkpoint·되돌리기 UX가 안정된 다음 정밀화한다.

dogfood 시점:

- **PR 4 시점**부터 본인이 agy 1명으로 매일 사용 가능 → LLM 왕복 페인 1/3 해소.
- **PR 5 시점**에 3 페르소나 풀 워크플로우 → 페인 전체 해소.
- 이후 PR 6~8은 Home 정리, legacy 제거, 자동 안전장치 안정화.

§10과 §11의 진실 기준은 동일하다: 온보딩 화면 수정 → plan 기능 → Home 정리 → 기존 기능 제거 준비 순서다.

## 13. 검증 원칙

코알못 UX는 "기능이 있다"가 아니라 "안 봐도 되는 걸 안 봤다"로 검증한다.

### 자동 테스트

- 초보 모드에서 patch/anchor/MCP/rules 문구가 첫 화면에 없는지 검사
- `vib patch`가 deprecated 안내를 내는지 검사
- GUI 초보 모드가 3개 주요 카드만 렌더링하는지 검사
- start 자동 실행 실패/성공 상태가 구분되는지 검사
- watch/guard 상태가 UI에 표시되는지 검사
- `vib plan`이 `plans/{slug}.md`를 만들고 CodeSpeak/patch 문구를 넣지 않는지 검사
- CLI가 없거나 로그인되지 않았을 때 로컬 템플릿 기획안을 만드는지 검사
- AI 작업실이 CodeSpeak/patch 지시문을 생성하지 않는지 검사
- 쉬운 변경 요약이 raw diff보다 먼저 표시되는지 검사

### 수동 QA

시나리오 1: 완전 초보 첫 실행

```text
GUI 실행 → 폴더 선택 → 준비 완료 → Home 3카드 확인
```

통과 기준:

- 터미널 명령을 요구하지 않는다.
- patch/anchor/watch/guard 용어가 첫 화면에 나오지 않는다.
- 되돌리기 가능 상태가 보인다.

시나리오 2: 자동 안전장치 실패

```text
watch 실패 상황 재현 → "자동 안전장치 일부가 꺼져 있어요" 표시
```

통과 기준:

- raw error만 보여주지 않는다.
- 다시 시도 버튼이 있다.

시나리오 3: 고급 모드 전환

```text
초보 모드 → 고급 기능 보기 → 기존 카드 접근
```

통과 기준:

- 기존 기능이 사라지지 않는다.
- patch는 deprecated/legacy로 표시된다.

시나리오 4: 첫 AI 작업 세션

```text
AI 작업실 → "버튼 문구 바꾸기" 선택 → 작업 전 checkpoint → 작업 후 쉬운 요약
```

통과 기준:

- CodeSpeak가 화면에 나오지 않는다.
- patch 지시문 복사를 요구하지 않는다.
- 작업 후 "괜찮아요"와 "되돌리기"가 모두 보인다.

시나리오 5: 첫 기획 세션

```text
터미널에서 vib plan "예약 앱 만들고 싶어" 실행 → 쉬운 질문 답변 → plans/*.md 생성
```

통과 기준:

- 사용자가 Markdown 문법을 입력하지 않는다.
- 준비된 CLI에 연결된 페르소나가 기획안을 검토한다.
- 준비된 CLI가 없어도 템플릿 기획안은 생성된다.
- 생성 문서에 목표, 사용자, 핵심 기능, 제외 범위, 다음 단계가 있다.

## 14. 최종 제품 원칙

1. **코알못은 명령어를 배우지 않는다.**
2. **VibeLign은 수정 도구가 아니라 안전 환경이다.**
3. **자동 안전장치는 기본으로 켜진다.**
4. **자동화 실패는 반드시 UI에 보인다.**
5. **자동 소스 변경은 항상 되돌릴 수 있어야 한다.**
6. **고급 기능은 사라지는 게 아니라 늦게 드러난다.**
7. **patch는 핵심 기능에서 퇴출한다.**
8. **바이브코딩은 작업 세션 단위로 안전하게 감싼다.**
9. **초보자에게는 raw diff보다 쉬운 변경 요약을 먼저 보여준다.**
10. **기획안은 사용자가 쓰는 것이 아니라 VibeLign이 만들어준다.**

## 15. 보수적 구현 판단

현재 VibeLign에서 이 방향은 단계적으로 구현 가능하다. 다만 한 번에 전부 구현하면 위험하다.

바로 가능한 것:

- `vib plan --template-only`로 Markdown 기획안 생성
- `claude`, `codex`, `antigravity` 설치 여부 확인
- 준비된 CLI가 없어도 템플릿으로 fallback
- React/Vite/Tauri 기반 GUI에 기획방 컴포넌트 추가
- 기본 클로이/지오/미나 아바타 제공

가장 큰 문제:

```text
구독 CLI 자동화 안정성
```

이유:

- 각 CLI는 API처럼 안정된 JSON 계약을 제공하지 않을 수 있다.
- 로그인 만료, TTY 요구, 사용량 제한, stdout 형식 변경이 발생할 수 있다.
- Antigravity CLI의 non-interactive 호출 안정성은 별도 검증이 필요하다.
- 실패한 CLI가 전체 기획 생성을 막으면 초보 UX가 깨진다.

안정성과 별개로 **약관/계약 리스크가 벤더별로 다르다** (멀티AI 설계안 §13 약관 준수 분석 참조):

- OpenAI(Codex)는 가장 안전(구독+`codex exec` 비대화형 공식 지원).
- Anthropic(Claude)은 회색지대 — `claude -p`는 허용이나 제3자 자격증명 라우팅은 능동 단속. VibeLign은 "공식 바이너리 subprocess + 토큰 미열람"으로 방어 가능한 쪽에 선다.
- Google Antigravity(`agy`): 밴 사례는 **토큰 추출/프록시**(OpenClaw식)이지 공식 바이너리 relay가 아니다(§13-7). VibeLign은 인증 무관 relay(agy OAuth/토큰 미취급)라 그 우려는 N/A. 남는 건 ToS 6조 문구뿐(인증 방식 무관). 무료 OSS라 재판매 조항도 무관(§13-8).
- 어떤 벤더도 VibeLign의 정확한 패턴(BYO 로그인 로컬 래핑)을 명시 허용하지 않았다. "위반 아님"이 아니라 "정면으로 다뤄지지 않음"이다.

따라서 첫 구현은 다음 순서로 제한한다.

1. `vib plan --template-only`
2. CLI 탐지와 로그인 필요 상태 표시
3. 가장 안정적인 CLI 1개만 실제 호출
4. 3 CLI best-effort orchestration
5. GUI 기획방과 기본 아바타
6. 페르소나 preset 커스터마이즈
7. sprite-gen 고급 생성/큐레이션

첫 MVP에서 제외:

- 앱 안에서 sprite-gen 실시간 생성
- 캐릭터 curation webview 내장
- 모든 CLI 동시 안정화
- 자동 코드 수정 연결

핵심 판단:

```text
sprite-gen은 락인 효과를 강화하지만, 첫 번째 기술 리스크는 아니다.
먼저 CLI adapter contract와 fallback을 안정화해야 한다.
```
