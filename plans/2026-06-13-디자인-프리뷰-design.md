# 디자인 프리뷰 (Design Preview) — 설계 (design)

> 작성 2026-06-13 · 사용자와 단계별 합의 + 강한 리뷰어 점검 반영
> 문제 제기(사용자): "기획안으로 어떤 디자인·레이아웃이 나올지 몰라서 코딩 후에 UI를 자주 수정하게 된다. 코딩 전에 기획안이 어떤 형태·디자인으로 나올지 먼저 프리뷰해주면 어떨까?"
> 선행: 기획방(PlanningRoom, 페르소나 LLM) → 작업방 Tier1(2026-06-12, BYO CLI 헤드리스) → RunPanel(실행 후 실앱 미리보기). 본 기능은 **기획안과 코딩 사이**에 "디자인 미리보기" 단계를 삽입한다.

## 0. 한 줄 정의

**확정된 기획안을, 초보도 고를 수 있는 이름 있는 디자인 스타일로 고충실 컬러 HTML 목업으로 그려 보여주고 — 피드백으로 다듬어 확정한 목업을 작업방 코딩의 "시작 코드 스캐폴드"로 넘겨, 코딩 후 UI 재작업을 코딩 전에 흡수한다.**

## 1. 확정 결정 (세션 합의 이력)

| 결정 | 선택 | 기각 대안 · 이유 |
|---|---|---|
| 프리뷰 충실도 | **고충실 컬러 목업** | 저충실 와이어프레임 — "이 느낌"이 안 보여 재작업 방지 효과 약함. 클릭 프로토타입 — 생성·검증 비용 과대, 첫 슬라이스엔 과함 |
| 목업의 역할 | **구속 입력(1) + 수정 가능 목업(3) 결합** | 참고용(2)만 — "본 것과 다르게 나옴"이 남아 핵심 문제 미해결 |
| 스타일 표현 | **C. 하이브리드 — 토큰 코어 + 프롬프트 레시피** | A(토큰 SSOT) — 스타일별 토큰 전세트 손수 정의, 초기 비용 큼. B(프롬프트팩) — 목업↔코드 일치가 LLM 눈대중 의존, 신뢰 갭 잔존 |
| 스타일 출처 | **하이브리드(자동제안+저장) + 큐레이팅된 디자인 언어 라이브러리** | 초보는 디자인 언어를 모름 → 이름 있는 스타일(네오브루탈리즘·레트로다이너·프루티거에어로·스칸디나비아·멤피스·팝아트·리소그래프·하프톤·사이키델릭·그런지…)을 갤러리에서 고르게 |
| **목업↔코드 결합 방식** | **확정 목업 HTML/CSS = 코딩의 시작 스캐폴드** (AI가 이어서 확장) | "스크린샷 참고용"으로 넘기면 레이아웃이 코딩에서 재생성되어 발산 — 신뢰 갭의 진짜 원인이 안 풀림(리뷰어 지적) |
| 타깃 | **웹 UI 한정** (HTML/CSS/Tauri webview) | C의 메커니즘(토큰→CSS 변수 + 목업=스캐폴드)은 웹일 때만 성립. 비웹(Flutter/SwiftUI/CLI)은 다른 결합이 필요 → 본 기능 범위 밖 |
| 웹 게이트(첫 슬라이스) | **최소 게이트 — 웹 UI 확인 불가 시 경고 후 계속** | 첫 슬라이스라도 무게이트는 위험(리뷰 반영). package.json/웹 마커 휴리스틱으로 비웹 의심 시 경고 배너, 차단까진 안 함. 강한 차단 게이트는 후속 |
| 생성 입력 안전 | **렌더러 호출 커맨드 = 입력 경계 강제** | project_dir canonicalize, plan_path는 프로젝트 내부 상대경로만, style_id는 `[a-z0-9-]+` allowlist, 저장은 `.vibelign/design_preview/` 내부로 canonicalize 검증 (경로 탈출·임의 파일 읽기 차단) |
| 생성 HTML 안전 | **저장 전 validator** | sandbox iframe은 프리뷰만 보호. 확정 HTML은 파일·스캐폴드가 되므로 `<!doctype html>` 확인·`<script>`/inline 핸들러/`javascript:`/외부 URL 차단·크기 상한 |
| 큰 프롬프트 전달 | **stdin 경유** (argv ❌) | 기획안+레시피+피드백이 Windows `.cmd` 셔임의 명령행/파싱 한계(~8191자)를 넘을 수 있음 → work_room과 동형으로 stdin 전달 |
| 피드백 연속성 | **이전 목업 HTML을 기준으로 수정** | "마지막 피드백만" 넣으면 직전 의도 소실. 직전 HTML을 프롬프트에 포함해 누적 의도 보존 |
| 첫 스펙 범위 | **생성→보기→피드백 루프→구속 핸드오프 / 웹 전용 / 손수 다듬은 5개 스타일** | 전체 6서브시스템 일괄 — 한 스펙에 과함. 척추부터 깔고 신뢰 갭이 실제로 닫히는지 증명 후 확장 |
| 첫 슬라이스 스타일 5종 | **미니멀 SaaS · 네오브루탈리즘 · 프루티거에어로 · 레트로다이너 · 리소그래프** | 시각적 대비가 뚜렷한 5종 → "목업↔실빌드 일치" 검증에 유리. 갤러리 전체 확장(10+)은 후속 |

## 2. 사용자 여정 변화 (기획안 → 코딩 사이 삽입)

| 단계 | 현재 | 디자인 프리뷰 도입 후 |
|---|---|---|
| 기획안 확정 | → 곧장 작업방 코딩 | → **디자인 프리뷰** 단계 진입 |
| 디자인 결정 | (없음, 코딩이 디자인을 즉흥 결정) | 이름 있는 스타일 갤러리에서 "이 느낌" 선택 |
| 디자인 확인 | 코딩 끝나고 RunPanel에서야 처음 봄 | **코딩 전** 고충실 컬러 목업으로 확인 |
| 다듬기 | 코딩 결과를 보고 재작업 지시(비쌈) | 목업 단계에서 "여긴 빨강, 버튼 크게" → 재생성(쌈) |
| 코딩 | 기획안만 입력 → 디자인 즉흥 | 기획안 + **확정 목업 스캐폴드 + 토큰** 입력 → 목업대로 빌드 |
| 실앱 확인 | "내가 본 거랑 다른데" 빈발 | 목업과 일치(토큰 고정 + 스캐폴드 계승) |

핵심 가치: **재작업 비용을 코딩 후(비쌈)에서 목업 단계(쌈)로 이동**. 목업 1회 생성 비용 < UI 재코딩 1회 비용.

## 3. 아키텍처 — C 하이브리드

### 3.1 스타일 표현 = 토큰 코어 + 프롬프트 레시피

각 디자인 스타일 = 두 부분:

```
스타일 정의 {
  id: "neo-brutalism",
  name: "네오브루탈리즘",
  설명: "두꺼운 검정 테두리·날것 색·하드 그림자",
  썸네일: <예시 이미지/마크업>,        // 갤러리 표시용

  // ── 토큰 코어 (결정적, 작음 — 목업·코드가 그대로 공유) ──
  tokens: {
    palette: { bg, surface, text, primary, accent, border },
    type:    { fontFamily, scale, weight },
    radius:  "0px",
    shadow:  "6px 6px 0 #000",
    border:  "3px solid #000",
    spacing: <스케일>,
  },

  // ── 프롬프트 레시피 (LLM 재량 — 컴포넌트 분위기·예시) ──
  recipe: "버튼은 굵은 테두리에 hover 시 그림자 이동. 카드는 겹침 허용. ..."
}
```

- **토큰 코어**: 사람이 "목업이랑 다르네"라고 한눈에 느끼는 4축(색·폰트·모서리·그림자)을 결정적으로 못박음. CSS 변수로 직렬화 가능 → 목업과 실코드가 **같은 변수**를 import.
- **프롬프트 레시피**: 컴포넌트 분위기·레이아웃 관용구. LLM이 기획안에 맞춰 유연하게 채움.
- 저장 위치: GUI의 정적 데이터 모듈(TS) 또는 동봉 JSON. 첫 슬라이스는 손수 다듬은 5개(미니멀 SaaS·네오브루탈리즘·프루티거에어로·레트로다이너·리소그래프).

### 3.2 파이프라인

```
[기획안 확정]
  → 스타일 선택 (첫 슬라이스: 갤러리에서 수동 선택. 자동추천은 후속)
  → 목업 생성: LLM( 기획안 + 선택 스타일 tokens + recipe ) → 자기완결 HTML/CSS
      · CSS 변수 = tokens (결정적 주입), 마크업·레이아웃 = recipe 따라 LLM
  → webview 표시 (RunPanel webview 인프라 재사용)
  → 피드백 루프: "여긴 빨강, 버튼 크게" → 재생성 (반복)
      · 캐시 키 = (기획안 해시 + 스타일 id + 피드백 해시)
  → [이 디자인으로 만들기] 확정
  → 핸드오프: 확정 HTML/CSS(스캐폴드) + tokens(CSS 변수) + recipe + 스타일명
            → 작업방 work handoff(workHandoff.ts)에 동봉
  → 작업방 AI: 목업 HTML을 시작 코드로 이어서 확장 (재생성 ❌)
```

### 3.3 재사용 자산

| 능력 | 소스 | 재사용 방식 |
|---|---|---|
| LLM 호출 | 기획방 페르소나 인프라(`planning_persona.rs`, BYO CLI/내부 프로바이더) | 목업 생성 LLM 호출에 동형 활용 |
| 목업 표시 | RunPanel webview(포트감지+webview, run_preview) | 로컬 HTML 파일/문자열을 webview로 렌더 |
| 코딩 핸드오프 | `workHandoff.ts` (작업방 입력 구성) | 디자인 산출물(스캐폴드+토큰+레시피)을 핸드오프 페이로드에 추가 |
| 해시 캐시 | 기존 AI 보강 해시 캐시 패턴(메모리: vibelign_ai_enhancement_optin) | (기획안+스타일+피드백) 해시로 재생성 방지 |
| 스트리밍 진행 | `vib_bridge.rs run_vib_with_progress` 패턴 | 목업 생성 진행 표시(스트리밍 모델일 때) |

**신규**: 스타일 라이브러리 데이터 모듈 + 목업 생성 커맨드(기획안+스타일→HTML) + 디자인 프리뷰 UI(갤러리·목업 뷰·피드백 입력·확정 버튼).

## 4. 신뢰 갭 — 어떻게 진짜로 닫는가

옵션 1(구속 입력)의 약점 = "목업과 실결과가 다르면 신뢰가 깨진다". 두 겹으로 방어:

1. **토큰 고정** — 색·폰트·모서리·그림자는 목업·코드가 *같은 CSS 변수*를 쓰므로 결정적으로 일치.
2. **스캐폴드 계승** — 확정 목업의 HTML/CSS가 코딩의 *시작 코드*. 레이아웃·컴포넌트 구조가 LLM 선의가 아니라 **구조적으로** 보존됨. (토큰만 공유하면 레이아웃은 코딩에서 재생성되어 "내비가 왼쪽이었는데 위로 갔네"가 남음 — 리뷰어 지적의 핵심 대응)

→ 한눈에 보이는 축(토큰)은 결정적 일치, 레이아웃은 스캐폴드 계승. 둘을 합쳐 신뢰 갭을 닫는다.

## 5. UI 골격 (첫 슬라이스)

- **위치(IA)**: 기획안 확정 직후 동선. 기획방→작업방 사이 또는 기획안 뷰(`PlanDocView`)에 "디자인 미리보기" 진입.
- **화면 구성** (위→아래):
  - 스타일 갤러리 — 이름 있는 스타일 카드(썸네일+이름+한줄설명), 5개(첫 슬라이스: 미니멀 SaaS·네오브루탈리즘·프루티거에어로·레트로다이너·리소그래프). 골라 "이 스타일로 그려보기"
  - 목업 뷰 — webview에 고충실 컬러 목업. 로딩 중 진행 표시
  - 피드백 입력 — "여긴 빨강, 버튼 크게" 자연어 → [다시 그리기]
  - 확정 — [이 디자인으로 만들기] → 작업방으로 스캐폴드 동봉 핸드오프
- **상태**: 생성 중 취소 가능. 재생성 횟수/캐시 적중 표시(비용 투명성).

## 6. 모델·비용

- **모델은 judge=opus 고정을 적용하지 않는다.** 그 결정(메모리: judge_model_quality_over_speed)은 기획 *판정*용. 목업 생성은 *생성* 작업이라 더 싸고 빠른 모델이 피드백 루프 비용을 낮춤. 모델은 구현 시 의도적으로 선택(품질 충분한 최저 비용).
- **캐시**: (기획안 해시 + 스타일 id + 피드백 해시). 같은 입력 재생성 0회.
- **비용 투명성**: BYO CLI/계정 사용 시 "본인 계정으로 실행" 명시(작업방 정책과 동형).

## 7. 리스크

| 리스크 | 대응 |
|---|---|
| **이름 있는 스타일이 곧 제품 — 잘 안 렌더되면 실패** | 배관(LLM+webview)은 쉬움. 프루티거에어로·리소그래프 같은 스타일을 토큰+레시피로 *잘* 렌더되게 만드는 게 디자인 공예 → 첫 슬라이스는 5개를 **제대로** 다듬고, 10+ 나열은 후속 |
| 비웹 타깃 유입 | "웹 UI 한정" 명시 + UI 게이트를 "웹 UI인지"까지 좁힘. 비웹이면 본 기능 비활성 + 안내 |
| 목업↔코드 발산 | §4 두 겹 방어(토큰 고정 + 스캐폴드 계승). 첫 슬라이스의 **검증 목표 = 신뢰 갭이 실제로 닫히는가** |
| 생성 비용/지연 | 캐시 + 저비용 모델 + 취소. 고충실은 무겁지만 캐시 적중이 흡수 |
| 스캐폴드가 작업방 CLI에 너무 큰 입력 | 작업방 Tier1의 stdin 전달·크기 한계(P0) 동형 점검. 목업 HTML 크기 상한·요약 전략 |
| 경로 탈출·임의 파일 읽기(렌더러 호출 커맨드) | project_dir canonicalize + plan_path 내부 상대경로만 + style_id allowlist + 저장 경로 canonicalize 검증(§1 안전 행) |
| LLM이 JS·외부 리소스 주입 | 저장 전 validator(§1). 프롬프트의 "JS 금지"만 믿지 않음 |
| Windows 명령행 길이 초과 | 프롬프트 stdin 전달(§1). CLI별 stdin 지원 확인(claude `-p`·codex `exec` = work_room 검증済), 미지원 시 temp 파일 폴백 |
| **에이전트 CLI가 HTML 대신 파일 저장 시도** | A12 검증(2026-06-13): `claude -p`는 목업 프롬프트에 Write 도구로 파일 저장 시도 → stdout 무HTML. 대응: 프롬프트에 "도구 금지·본문 출력" 명시 + claude `--disallowedTools`. claude·codex 둘 다 실호출 16/14 검증 통과 |
| **codex는 git/신뢰 디렉터리 밖에서 거부** | codex `exec`는 비-git 프로젝트에서 "Not inside a trusted directory" 로 실패. 비-git 프로젝트 지원하려면 codex 생성에 `--skip-git-repo-check` 추가 필요(기존 페르소나 codex 호출과 동일 제약 — 보류). 알려진 제약 |
| 생성 취소 미지원(첫 슬라이스) | UI 버튼만 꺼지고 backend는 timeout까지 진행 — **알려진 한계**. 진짜 취소는 watch식 자식 등록 필요(후속). 첫 슬라이스는 생성 timeout 단축으로 완화 |
| 캐시 무한 누적 | 생성 후 `.vibelign/design_preview/` 최근 N개만 유지(오래된 것 prune) |
| 확정 바인딩 휘발(앱 재시작) | designBinding은 메모리 — 목업 파일은 디스크에 영속. 바인딩 레코드(planPath→mockupPath→tokens)는 manifest 기록을 후속에 |

## 8. 마일스톤 (첫 슬라이스) — **1스타일 E2E 먼저, 4스타일은 별도 튜닝 PR** (리뷰 반영)

배관(UI·커맨드·바인딩)과 스타일 품질 튜닝을 한 PR에 섞지 않는다. 먼저 1개 스타일로 끝까지 닫고, 나머지는 분리.

**PR A — E2E 척추 (네오브루탈리즘 1종)**
1. **목업 생성 커맨드** — (기획안 + 스타일 + 직전 HTML) → 자기완결 HTML/CSS, CSS 변수=토큰 결정적 주입, **입력 검증 + HTML validator + stdin 전달 + 캐시(+prune)**
2. **디자인 프리뷰 UI** — 갤러리(1종) + sandbox iframe 뷰 + 피드백 재생성 루프 + 웹 게이트 경고 + 확정 버튼
3. **구속 핸드오프** — 확정 HTML(스캐폴드)+토큰을 `buildPlanningWorkInstruction`에 동봉, 작업방 AI가 이어서 확장
4. **진입/확정 배선** — PlanDocView의 **선택된 기획안**으로 진입(활성 세션 아님), 확정→작업방
5. **E2E 검증** — 신뢰 갭 닫힘(목업 ↔ 실빌드 일치), 웹 게이트, Win/mac 패키징

**PR B — 스타일 품질 (나머지 4종)**
6. 미니멀 SaaS·프루티거에어로·레트로다이너·리소그래프를 토큰+레시피로 추가, 각각 생성→육안→토큰 조정 루프로 다듬기 (제품의 핵심, 시간 투자처)
