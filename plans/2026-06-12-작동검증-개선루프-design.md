# 작동 검증 + 개선 루프 — 설계 (design)

> 작성 2026-06-12 · 브레인스토밍 세션에서 사용자와 단계별 승인
> 선행: 실행해보기(Run & Preview) M1~M3b 완료(plans/2026-06-12-실행해보기-run-preview-design.md) · 가이드레이어(2026-06-10)
> 근거: 실행해보기는 "앱이 켜지나"까지 닫았지만, **"진짜 잘 되나"는 사용자가 써봐야 안다**. 그리고 써보고 부족함을 느꼈을 때 다음 제스쳐(개선)가 없으면 루프가 다시 끊긴다. 가이드 5️⃣가 천명한 **"검증의 두 축(안전 guard + 작동 run)"** 중 작동 축이 아직 앱 상태로 인지되지 않는다.

## 0. 한 줄 정의

**작동 검증을 "서버가 떴다"(자동)가 아니라 "사용자가 써보고 확인했다"(명시)로 만들고, 미리보기에서 부족함을 느꼈을 때 작업방으로 개선 요청을 보내는 제스쳐를 연다 — guard(안전)와 run(작동) 두 축을 VibeLign이 함께 인지하게 한다.**

## 1. 확정 결정 (브레인스토밍 승인 이력)

| 결정 | 선택 | 기각 대안 · 이유 |
|---|---|---|
| 작동 검증 정의 | **사용자 명시 확인("써봤다 ✓")** | 자동("서버가 떴다") — 켜진 것과 잘 되는 것은 다르다. 써봐야 안다 |
| 개선 요청 라우팅 | **하이브리드 — 기본 작업방 직행 + 보조 기획방** | 무조건 기획방(무겁다) / 무조건 작업방(방향 전환엔 부적합) |
| 가이드 엮기 | **추가 축(인지만, 강제 안 함)** | 필수 조건 — 라이브러리·CLI는 실행 감지가 안 돼 영원히 완주 못 함. inferStep 게이팅은 보존 |
| 핸드오프 구조 | **M3b runErrorFix 채널을 workHandoff{kind}로 일반화** | 별도 채널 신설 — consume-once 기계 중복 |

## 2. 작동 검증 신호 `runVerified`

- App 에 `runVerified: boolean` 신호 추가. `guardStatus` 와 **짝**이되 `inferStep` 게이팅은 건드리지 않는다(별도 신호).
- **persist/reset = `guardStatus` 와 동일 규칙**: 마지막 변경 지문(`changedFingerprint`)이 달라지면 자동 리셋. 코드를 또 고치면 이전 "작동함"은 stale → 무효화. 이게 "정확한 상태 인지"의 핵심 — guard 의 stale-pass 방지와 같은 메커니즘(spec §3.1/H2 동형).
- 설정 경로: RunPanel 이 사용자 "✓ 잘 돼요" 클릭 시 `onRunVerified()` 콜백으로 App 에 보고(guard 의 `onGuardResult` 와 동형 채널).

## 3. RunPanel 제스쳐 (running 중 노출)

dev 서버가 떠 있을 때(`status === "running"`; 웹=미리보기 가능, electron=창 열림) 두 제스쳐를 노출한다. **미리보기 열기를 강제하지 않는다** — 서버가 뜨면 바로 노출(사용자가 이미 봤을 수도, 따로 열 수도).

- **✓ 잘 돼요 — 작동 확인 완료** → `onRunVerified()` → `runVerified=true`. (guard 도 ok 면) 🎉 완주 축하.
- **✏ 뭔가 부족해요 — 고치고 싶어요** → 개선 요청 입력창(인라인 textarea) → 입력 후 핸드오프(§4).

종료(stopped/done/failed) 후에는 제스쳐를 숨긴다(running 중에만 의미).

## 4. 개선 요청 핸드오프 (하이브리드, M3b 일반화)

M3b 의 `runErrorFix: string | null` 채널을 일반화한다:

```
workHandoff: { kind: "error" | "improve"; text: string } | null
```

- consume-once 기계(WorkRoom 마운트 시 `== null` 가드 캡처 → `onWorkHandoffConsumed`)는 그대로 재사용.
- **기본: 작업방 직행** — 개선 요청 텍스트 → `buildImproveInstruction`(범위 안에서 다듬기, "다시 실행/미리보기 하지 말라") → 체크포인트→실행→guard 안전 시퀀스 그대로. guard 가 범위 이탈을 잡으므로 plan-first 안전성 유지.
- **보조 링크: "큰 방향을 바꿀래요 → 기획방"** — 새 의도는 기획 갱신부터(onNavigate("planning")).

### WorkRoom `errorFix` 모드 → `handoff` 모드 일반화
- 로컬 상태 `errorFix: string | null` → `handoff: { kind; text } | null`.
- `effectiveInstruction = handoff != null ? (handoff.kind === "error" ? buildRunErrorFixInstruction(...) : buildImproveInstruction(...)) : instruction`.
- 배너·버튼 라벨 kind 별: error="실행 에러를 고칠게요", improve="요청하신 개선을 작업할게요".
- `buildImproveInstruction({ requestText, planPath })`(순수, 테스트): "이미 만든 것 위에 이런 개선을 해줘 + 범위 밖은 확인 + 다시 실행 금지". planPath 있으면 기획안 범위 참조.

## 5. 가이드 / 완주

- `inferStep` **무변경**(검증된 상태기계 보존). `runVerified` 는 별도 App 신호로 UI 표현만.
- guard✓ + runVerified 동시 → 5️⃣ 카피/홈에 "안전·작동 둘 다 확인했어요 🎉" 완주 축하.
- 한 축만 통과 → 나머지 축 nudge:
  - guard ok·작동 미확인 → "작동도 확인해볼까요? → 실행해보기"
  - 작동✓·guard 미확인 → "안전도 확인해볼까요? → 상태 확인"

## 6. 컴포넌트 경계 / 순수 함수

- `buildImproveInstruction`(신규, lib/run-preview/, 순수+vitest) — 개선 요청 프레이밍.
- `buildRunErrorFixInstruction`(기존) — error 프레이밍. 둘은 `handoff.kind` 로 분기.
- App: `runVerified` 상태 + `changedFingerprint` 리셋 effect(guardStatus 리셋과 동형으로 합류).
- RunPanel: 제스쳐 2종 + `onRunVerified`/개선 입력 + `onRequestWorkHandoff({kind, text})` props.
- WorkRoom: `runErrorFix`/`onErrorFixConsumed` → `workHandoff`/`onWorkHandoffConsumed` 일반화.

## 7. 리스크 / 비범위

- **리스크**: M3b 코드(runErrorFix/errorFix 모드) 일반화 — 최근 자작 코드라 낮음. `inferStep` 미변경으로 검증된 상태기계 보존. no-handoff 경로 바이트 동일 유지(M3b 리뷰 기준 재적용).
- **비범위(후속)**: 자동 작동 감지(서버 뜸=잠정 작동) 미채택(사용자 명시 확인이 본 설계). 개선 요청의 다중 턴/이력. 작동 검증의 세분(부분 작동).

## 8. 마일스톤

1. **작동 검증 신호** (0.5일): App `runVerified` + 리셋 합류 + RunPanel "✓ 잘 돼요" + `onRunVerified`. 완주 🎉 카피.
2. **개선 핸드오프 일반화** (1일): `buildImproveInstruction`(순수+테스트) + App/WorkRoom 채널 `runErrorFix`→`workHandoff{kind}` 일반화 + RunPanel "✏ 부족해요" 입력창 + 기획방 보조 링크. no-handoff 무변화 검증.
3. **가이드 nudge** (0.5일): guard/run 두 축 상호 nudge 카피 + 완주 합류.
4. **검증**: cargo(영향 없음)·tsc·vitest·eslint + code-reviewer(WorkRoom 회귀 집중).
