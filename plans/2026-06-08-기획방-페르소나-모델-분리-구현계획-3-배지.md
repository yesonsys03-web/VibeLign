# 기획방 페르소나↔모델 분리 — 구현계획 3 (폴백 "대체됨" 배지)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 기획방 대화에서 어떤 페르소나가 1순위 모델이 아니라 폴백 모델로 답했을 때, 그 메시지에 "OO로 대체됨" 배지를 보여준다. 사용자가 "왜 다른 AI가 답했지?"를 바로 알 수 있게 한다.

**Architecture:** Rust `run_persona_response` 가 실제 응답한 provider 가 1순위와 다를 때만 `provider_used` 를 채워 `PersonaRun`→`PlanningChatMessage` 에 싣는다(폴백이 아니면 None). 프런트 `PlanningChatMessage` 타입에 `providerUsed?` 를 추가하고, `PlanningMessages` 버블이 값이 있을 때 배지를 렌더한다. 폴백 판정은 런타임에 backend 가 하므로(설정 변경과 무관하게 정확) 프런트는 단순 렌더만.

**Tech Stack:** Rust(serde, cargo test), React/TS(Vitest). 구현계획 1·2 완료 전제.

**관련:** 스펙 `...-design.md` §9.2, 구현계획 1(백엔드 폴백), 구현계획 2(Settings UI).

## 범위 / 비범위

- **포함:** 대화 경로(Rust chat) 폴백 시 메시지 배지 한 종류("{provider}로 대체됨").
- **제외:** 기획방 인라인 빠른 편집(칩에서 즉시 provider 교체) → 별도 구현계획 4. Settings(구현계획 2)가 이미 재배정/ON·OFF 를 제공하므로 우선순위 낮음.
- **제외:** 계획서 생성 경로(Python)의 배지 — 그 경로는 마크다운 문서를 만들고 페르소나 칩 상태는 `agent_statuses` 로 이미 표시. 별도 요구 없으면 손대지 않음.

## 폴백 판정 규칙

`run_persona_response` 의 시도 순서 `resolve_provider_order(persona_id, spec)` 의 **첫 원소가 1순위(preferred)**. 실제로 `ok` 를 낸 provider 가 preferred 와 **다르면** 폴백 → `provider_used = Some(그 provider)`. 같으면 `None`. 실패(전부 소진)면 `None`(배지 없음, 기존 실패 상태 라벨로 충분).
