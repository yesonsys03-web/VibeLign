# 모션 차원 (Motion Dimension) — 설계 (design)

> 작성 2026-06-13 · 사용자와 단계별 합의
> 출처: animations.dev/vocabulary(모션 어휘 글로서리) — [[vibelign_design_skill_directions]] 3트랙 중 1번
> 선행: 디자인 프리뷰 PR A(plans/2026-06-13-디자인-프리뷰-design.md, 구현계획.md). 본 트랙은 그 위에 **모션 차원**을 얹는 후속(PR C).

## 0. 한 줄 정의

**각 디자인 스타일에 고유한 모션 성격을 베이크 — 목업 프리뷰엔 CSS 모션으로 실제 움직이게 보이고, 코딩 핸드오프엔 풀 모션 어휘(animations.dev: 스프링·스크롤·reduced-motion 등)로 전달한다.**

## 1. 확정 결정 (세션 합의 이력)

| 결정 | 선택 | 기각 대안 · 이유 |
|---|---|---|
| 모션 범위 | **둘 다 — CSS는 프리뷰, 풀어휘는 핸드오프** | 프리뷰 우선만/핸드오프 우선만 — 한쪽 가치를 버림. `sandbox=""` 제약상 JS 모션은 프리뷰 불가라 자연히 CSS=프리뷰·풀어휘=핸드오프로 갈림 |
| 데이터 모델 | **스타일에 베이크 (StyleSpec.motion)** | 별도 모션 축 — UI 선택 컨트롤 추가, YAGNI. 모션 성격은 스타일 정체성의 일부(브루탈리즘=즉각·딱딱, 프루티거에어로=부드럽게 플로팅) |
| motion 필드 구조 | **A. 단일 recipe + 작은 토큰** `{ tokens:{duration,easing}, recipe }` | B(이중 recipe) — 큐레이팅 2배. C(토큰만) — 스타일별 모션 성격 소실. CSS-vs-풀어휘 분리는 *데이터*가 아니라 *프롬프트 빌더*가 처리 |

## 2. 데이터 모델 (StyleSpec 확장)

TypeScript (`vibelign-gui/src/lib/design-preview/styles.ts`):
```ts
export interface MotionTokens {
  readonly duration: string;  // 예: "80ms"
  readonly easing: string;    // 예: "cubic-bezier(.2,0,0,1)"
}
export interface MotionSpec {
  readonly tokens: MotionTokens;
  readonly recipe: string;    // prose: CSS 모션 + 풀 어휘 함께 기술
}
export interface StyleSpec {
  /* 기존 필드 ... */
  readonly motion?: MotionSpec;  // optional → 기존 스타일 하위호환
}
```

Rust 미러 (`vibelign-gui/src-tauri/src/commands/design_preview.rs`):
```rust
#[derive(Debug, Clone, Deserialize, Serialize)]
pub(crate) struct MotionTokens { pub duration: String, pub easing: String }
#[derive(Debug, Clone, Deserialize, Serialize)]
pub(crate) struct MotionSpec { pub tokens: MotionTokens, pub recipe: String }
// StyleSpec 에 추가:
#[serde(default)] pub motion: Option<MotionSpec>,
```
`#[serde(default)]` 로 motion 없는 입력도 역직렬화 통과(하위호환).

## 3. 메커니즘

### 3.1 토큰 결정적 주입 (색 토큰과 동형)
`motion.tokens` → CSS 변수 `--dur`·`--ease`. 기존 `tokens_to_css_vars` 의 `:root{}` 블록에 합류(목업·실코드 공유, 결정적). motion 없으면 추가 안 함.

### 3.2 목업 프롬프트 주입 (`build_mockup_prompt`, CSS-only 제약)
motion 있을 때 프롬프트에 모션 섹션 추가:
```
[모션 — 이 스타일의 움직임 성격]
<motion.recipe>
- 목업에선 CSS 전환(transition)·@keyframes·:hover 로만 표현. 자바스크립트 금지.
- 지속·이징은 var(--dur)·var(--ease) 변수만 사용.
- @media (prefers-reduced-motion: reduce) 에서 모션을 비활성/축소.
```
→ `sandbox=""` iframe 은 스크립트는 막아도 **CSS 애니메이션·전환은 실제로 동작**하므로 프리뷰에서 살아 움직인다. 등장 애니메이션은 로드 시 1회 재생(라이브 iframe이라 캡처 타이밍 문제 없음).

### 3.3 핸드오프 주입 (`buildPlanningWorkInstruction` design 섹션 확장)
`design` 바인딩에 motion 추가:
```ts
design?: { mockupPath; tokens; motion?: { tokens:{duration,easing}, recipe } }
```
지시문에 "[모션 가이드]" 추가: `motion.recipe` 전문(풀 어휘 — 스프링·스크롤·등장/퇴장 등) + `--dur`/`--ease` 토큰 + "prefers-reduced-motion 존중, 공간 일관성 유지". 실코드 단계는 JS/모션 라이브러리 사용 가능.

DesignPreview 의 `DesignBinding` 에 `motion?` 추가 → 확정 시 선택 스타일의 motion 을 바인딩에 실어 작업방으로.

## 4. 검증 안전성

현 `validate_mockup_html` 은 `<script>`·`<iframe>`·`javascript:`·외부 리소스 로드(src/href/url/@import)·인라인 `on*=` 핸들러만 차단. CSS `@keyframes`/`transition`/`animation`/`:hover`/`@media` 는 스크립트가 아니므로 **통과**. 모션 추가가 검증기를 건드리지 않는다 ✓ (검증 테스트로 못박음).

## 5. UI 영향 (최소)
모션은 데이터 + 프롬프트 중심이라 DesignPreview UI 변경 거의 없음. iframe 이 이미 라이브라 CSS 모션이 그대로 재생됨. (갤러리 카드에 "✨ 모션" 뱃지 정도는 optional — 첫 슬라이스 범위 밖.)

## 6. 첫 슬라이스 범위

1. **데이터**: `MotionTokens`/`MotionSpec`(TS+Rust) + 네오브루탈리즘 1종에 motion 베이크
   - 예: `tokens:{duration:"80ms", easing:"cubic-bezier(.2,0,0,1)"}`, recipe: "호버 시 그림자 방향으로 2px 즉각 이동, 전환은 거의 없이 딱딱(--dur 짧게). 카드 등장은 계단식 pop(스케일 0.96→1, 빠르게). 부드러운 ease 금지 — 각지고 즉각적. reduced-motion 에서 전부 정지."
2. **목업 주입**: `tokens_to_css_vars` 에 `--dur`/`--ease` 추가 + `build_mockup_prompt` 모션 섹션(CSS-only 제약)
3. **핸드오프 주입**: `buildPlanningWorkInstruction` design.motion 섹션 + `DesignBinding.motion`
4. **검증**: 순수함수 테스트 + 실호출(목업에 CSS 전환/키프레임/prefers-reduced-motion 들어가고 validator 통과)

## 7. 후속 (범위 밖)
- 나머지 4스타일 motion 베이크(PR B 스타일 튜닝과 함께): 미니멀SaaS=부드러운 ease·은은, 프루티거에어로=플로팅·스프링감, 레트로다이너=바운스, 리소그래프=스냅·약간 거칢
- 모션 강도 오버라이드 축(none/subtle/rich) — 별도 모션 축
- 갤러리 모션 뱃지

## 8. 리스크

| 리스크 | 대응 |
|---|---|
| 목업이 모션을 JS로 시도 | 프롬프트 "CSS만·JS 금지" 제약(§3.2) + validator 가 `<script>` 차단(이중) |
| validator 가 CSS 모션 오차단 | §4 — 현 validator 는 CSS 안 막음. 검증 테스트로 회귀 방지 |
| 등장 애니메이션 캡처 타이밍 | iframe 은 라이브 렌더라 무관(스크린샷 아님) |
| reduced-motion 미존중 | 프롬프트·핸드오프 양쪽에 명시(§3.2·3.3) |
| 하위호환(motion 없는 스타일·바인딩) | optional 필드 + Rust `#[serde(default)]` + 프롬프트/핸드오프 분기 가드 |
