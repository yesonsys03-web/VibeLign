# 작업방 "실행해보기" CTA 강조 — 설계 스펙

- 날짜: 2026-06-14
- 대상: `vibelign-gui/src/pages/WorkRoom.tsx` (+ 필요 시 `src/styles/brutalism.css`)
- 상태: 설계 확정 대기(사용자 리뷰)

## 1. 배경 / 문제

AI 코딩 작업이 끝나면 다음 자연스러운 수순은 **결과를 실행해 테스트**하는 것이다. 하지만 현재는 사용자가 상단 네비의 **"실행해보기" 탭을 직접 눌러야** 그 단계로 넘어간다.

작업방 완료 화면(`WorkRoom.tsx`)을 위→아래로 보면:

```
.page-content (스크롤 컨테이너, overflow-y:auto, height:100%)
└ 콘텐츠 그리드 (maxWidth:860)
   ├ page-header "작업방"
   ├ 기획안 카드
   ├ 실행 카드  ── phase 별 블록
   │   └ phase==="finished" 완료 패널 (605–700)  ← "상태창"
   │        · OUTCOME_LABEL: "AI 작업이 끝났어요"
   │        · guard 판정 박스(pass/prepare/stop)
   │        · 버튼 행: [앵커 자동추가][체크포인트 저장][▶ 실행해보기 →][홈][다시실행]
   └ 스트리밍 로그 카드 (708–745)
        · ...AI 긴 출력...
        · ✅ 작업 완료   ← 사용자가 "제일 하단부"에서 실제로 보는 줄(에이전트 로그)
```

**진단**: `▶ 실행해보기 →` 버튼은 **이미 존재**한다(`WorkRoom.tsx:688`, `guardSafe`일 때). 그러나
1. **작은 보조 스타일**(`btn btn-sm`)이라 5개 버튼 틈에 묻히고,
2. **로그 위쪽**에 있어, 사용자가 긴 로그를 스크롤해 바닥의 "✅ 작업 완료"를 볼 때는 화면 밖으로 밀려나 있다.

그래서 "다음 버튼이 없네 → 탭을 직접 누르자"가 된다. **새 기능이 아니라, 완료 신호와 실행 액션을 시각적으로 연결하는 배치 문제**다.

## 2. 목표 / 비목표

**목표**
- 작업 완료 후 "실행해보기"로 가는 액션을 **눈에 확 띄게**, 사용자가 보는 위치 어디서든 한 번에 닿게 한다.
- 자동 이동/자동 실행은 하지 않는다(원클릭, 사용자 주도).

**비목표**
- 완료 시 run 탭 자동 전환 또는 앱 자동 기동(명시적으로 제외 — 사용자 결정).
- run 동시성/슬롯 모델 변경, RunPanel 로직 변경.
- 탭 재진입 후 "지난 실행 기록"(`showRestored`) 상태에 CTA 추가(이번 범위 밖, §6 참조).

## 3. 설계 — 두 지점 배치

### 3-1. 상태창(완료 패널) — 기존 버튼 승격

`WorkRoom.tsx:688` 의 기존 버튼을 **1차 CTA**로 승격한다.

- 스타일: `btn btn-sm`(작은 보조) → **큰 버튼**(`btn`, primary 강조색 + 굵은 보더/그림자, 브루탈리즘 톤).
- 위치: 완료 버튼 행에서 **맨 앞**으로 이동(체크포인트/홈/다시실행보다 시각 우선).
- 라벨/아이콘: `▶ 실행해보기 →` 유지.
- 게이팅: 기존과 동일하게 `guardSafe`(verdict `pass` | `prepare`)일 때만 노출.

### 3-2. 패널 하단부 — sticky 고정 CTA 바 (신규)

콘텐츠 그리드(`maxWidth:860` div)의 **마지막 자식**으로 sticky 바를 추가한다.

- 조건부 렌더: `phase === "finished" && guardSafe` 일 때만.
- 배치: `position: sticky; bottom: 0` — 스크롤 컨테이너(`.page-content`) 기준으로 **뷰포트 하단에 고정**되어, 로그를 끝까지 스크롤해도 항상 보인다. 자연 위치(콘텐츠 맨 끝)에 닿으면 그대로 안착.
- 내용: 풀폭(콘텐츠 컬럼 폭) 큰 버튼 — `▶ 이대로 실행해보기 →`, `onClick={() => onNavigate("run")}`.
- 시각(확정): 바 배경은 **불투명 라이트 톤**(완료 패널 카드와 같은 `#F5F1E3`) + `border-top: 2px solid #1A1A1A` + 위쪽 box-shadow로 아래 로그가 비치지 않게. 바 **안의 버튼**은 **다크 1차 버튼**(`background:#1A1A1A; color:#fff; border:2px solid #1A1A1A` — 기존 선택형 버튼과 동일 톤). `z-index`는 로그 카드 위.
- `.page-content` 의 `padding: 20px` 때문에 `bottom: 0` 이 바닥에서 20px 떠 보일 수 있다 → 바 컨테이너에 `margin: 0 -20px -20px` 또는 음수 보정으로 컨테이너 가장자리에 밀착시킨다(구현 시 시각 확인).

두 버튼 모두 **동일하게 `onNavigate("run")`** 를 호출한다. 의도된 중복(상단 패널 + 하단 고정)으로 도달성을 확보한다.

## 4. 동작 / 게이팅

| guard 판정 | 상태창 버튼 | 하단 sticky 바 | 비고 |
|---|---|---|---|
| `pass` / `prepare` (`guardSafe`) | 표시(승격) | 표시 | 정상 실행 경로 |
| `stop` | 숨김 | 숨김 | 안전 우선 — 기존 "백업에서 되돌리기" 안내 유지 |
| guard 미실행/에러 | 숨김 | 숨김 | 실행 적합성 미확정 |

- **동시성**: run 은 작업방과 슬롯 1개를 공유하나, `finished` 시점엔 작업방이 돌지 않으므로 충돌 없음(기존 모델 그대로).
- **모션**: sticky 바는 등장 애니메이션 불필요. 추가 시 전역 `prefers-reduced-motion` 가드 준수.

## 5. 영향 범위 / 파일

- `src/pages/WorkRoom.tsx`
  - `WorkRoom.tsx:688` 버튼 스타일·순서 변경(3-1).
  - 콘텐츠 그리드 마지막에 sticky CTA 바 JSX 추가(3-2).
- (선택) `src/styles/brutalism.css` — sticky 바/승격 버튼용 클래스가 인라인보다 깔끔하면 추가.
- 신규 파일·구조 변경 없음. RunPanel/run 백엔드 무변경. `onNavigate("run")` 재사용.

## 6. 엣지 케이스 / 후속

- **탭 재진입(showRestored)**: 완료 후 탭을 떠났다 돌아오면 `phase` 가 `idle` 로 복원되고 "지난 실행 기록"만 표시되어 두 CTA 모두 사라진다. 기존 `실행해보기` 버튼과 동일한 동작이라 일관성 유지 차원에서 이번 범위에서 그대로 둔다. (후속: 복원 상태에서도 마지막 실행이 safe 였으면 sticky 바 유지 검토.)
- **짧은 로그**: 콘텐츠가 뷰포트보다 짧으면 sticky 바는 자연 위치(맨 아래)에 그냥 놓인다 — 문제 없음.
- **handoff 모드**: error/improve 핸드오프 진입 시엔 `phase` 가 `finished` 가 아니므로 CTA 미표시 — 의도된 동작.

## 7. 테스트

- WorkRoom 렌더 테스트(존재 시 동일 패턴)로:
  - `phase==="finished" && guardVerdict==="pass"` → 상태창 버튼 + sticky 바 둘 다 렌더, 클릭 시 `onNavigate("run")` 호출.
  - `guardVerdict==="stop"` → 두 CTA 모두 미렌더.
- 회귀: 기존 완료 패널 버튼(체크포인트/홈/다시실행) 동작·표시 유지.
- 수동: 긴 로그에서 끝까지 스크롤해도 sticky 바가 뷰포트 하단에 계속 보이는지, 로그가 비쳐 보이지 않는지 시각 확인.
